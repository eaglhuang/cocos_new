# 多 Agent 協作不互撞：task lock 與 handoff contract 的實戰設計

> 這篇要回答的不是「怎麼讓 AI 一起工作」，而是「怎麼讓多個 AI 和人類一起工作時不互相踩腳」。

## TL;DR

當 AI 從單點助手變成多 Agent 協作，最大的問題通常不是能力，而是邊界。

只要沒有 ownership，大家就很容易一起讀、一起改、一起猜，最後變成互相覆寫。這種情況下，工具再強也沒用，因為真正壞掉的是協作規則。

所以要把多 Agent 協作做好，核心只有兩件事：

- `task lock`，先把寫入權說清楚
- `handoff contract`，再把交接格式固定下來

## 1. 多 Agent 最常撞車的地方

多 Agent 協作最容易出問題的，不是做錯功能，而是做了別人的功能。

常見症狀像這些：

- 兩個 agent 同時改同一份檔案
- 一個 agent 重新寫掉另一個 agent 剛整理好的摘要
- 交接時只剩一段聊天紀錄，沒有可執行的狀態
- 下一位接手的人不得不從頭重看所有內容

這代表問題不在模型，而在流程沒有定義清楚。

## 2. `task lock` 的本質是 ownership

`task lock` 不是在管制創作，而是在定義誰對這個任務負責。

它的價值是讓每個人都知道：

- 這個任務現在屬於誰
- 哪些檔案在這個任務範圍內
- 哪些範圍暫時不要碰
- 什麼時候可以交棒

這很像團隊在做版控，但更精準。不是只有「有沒有改檔案」，而是「這些改動是不是同一個責任邊界內的事情」。

如果兩個 agent 的 scope 重疊，最好的做法不是硬讓它們同步，而是拆小範圍、重新定界，必要時直接停下來讓人類裁決。

## 3. `handoff contract` 不能只是一段摘要

很多人以為交接就是貼一段 summary，但真正有用的 handoff 要包含可以接手的狀態。

一份好的 handoff 至少要說清楚：

- 現在做到哪裡
- 已經確認了哪些決策
- 還有哪些阻塞
- 下一個人應該先做什麼

如果只給摘要，下一位 agent 仍然得自己猜。

## 4. 一份可執行的 handoff 應該長這樣

可以把 handoff 想成一個小型協作合約。最基本的欄位像這樣：

```json
{
  "agentId": "agent-01",
  "taskId": "UI-2-0032",
  "phase": "post-flight",
  "changedFiles": [
    "assets/resources/ui-spec/layouts/detail.json",
    "assets/resources/ui-spec/screens/detail.json"
  ],
  "decisions": [
    "Tab 區塊改為 lazy load",
    "header 區域沿用既有 skin family"
  ],
  "blockers": [
    "右側資訊欄內容長度仍需確認"
  ],
  "nextAction": "補齊 content contract 後再跑 validate-ui-specs"
}
```

這裡最重要的不是格式漂不漂亮，而是資訊要能直接驅動下一步。

## 5. `changedFiles`、`decisions`、`blockers`、`nextAction` 為什麼最重要

這四個欄位幾乎就是交接的骨架。

### `changedFiles`

告訴下一位接手的人，實際變動在哪裡。

### `decisions`

告訴下一位接手的人，哪些問題已經拍板，不用重問。

### `blockers`

告訴下一位接手的人，現在卡住的是技術問題、資訊問題，還是產品決策問題。

### `nextAction`

告訴下一位接手的人，第一步應該做什麼，而不是叫他自己重新分析整個世界。

## 6. 交接的重點，是不要重做前一位已經做過的事

這件事很重要。

handoff 的目的不是把所有歷史都保存下來，而是讓下一位接手的人可以直接從「已確認狀態」開始工作。

所以交接時應該避免：

- 原封不動貼整段聊天
- 重新解釋所有背景
- 把已經結論的事情又打開一次

交接時應該保留：

- 現況
- 決策
- 風險
- 下一步

## 7. 發生衝突時，先停，不要硬合

多 Agent 協作最怕「看起來都有做事，但最後合不起來」。

如果你發現：

- 兩個任務碰到同一個檔案
- 兩個 agent 對同一個規格有不同理解
- 一個人已經改了結構，另一個人還在寫舊結構的內容

那就不要硬 merge。

比較好的處理方式是：

1. 先停下來
2. 對齊 scope
3. 重新鎖定 ownership
4. 再繼續

這不是保守，而是避免後面花十倍時間救火。

## 8. 人類在這裡仍然很重要

多 Agent 協作不是把人移除，而是把人移到最值得決策的地方。

人類最重要的角色是：

- 在歧義太大時拍板
- 在範圍重疊時裁決
- 在驗收時確認品質
- 在流程失控前決定要不要收斂

也就是說，AI 很適合處理大量機械性協作，但最終的邊界管理，還是要有人在場。

## 9. 寫給想學的人：你可以先從這三件事開始

如果你也想把多 Agent 協作做穩，我建議先做三件事：

1. 任何任務都先指定 ownership
2. 交接一定要有 `changedFiles`、`decisions`、`blockers`、`nextAction`
3. 只要 scope 重疊，就先停下來重新切分

這三條做到了，很多協作問題會直接少一半。

## 結語

多 Agent 協作真正難的，不是讓大家一起做事，而是讓大家知道自己該做什麼、不該做什麼、做完之後要怎麼交棒。

當你把 task lock 和 handoff contract 做好，AI 才不會變成一群很努力但一直互撞的助手。
