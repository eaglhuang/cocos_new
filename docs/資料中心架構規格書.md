### M1-C：Skills 體系設計

### M1-D：稀有度系統設計

- 兩層稀有度模型：核心 `rarityTier`（平衡）+ 商業 `commercial band`（展示 / 包裝）
- 核心七級稀有度：N / R / SR / SSR / UR / LR / GOD，作為戰力、適性、EP 與資料稽核的唯一真相來源
- 商業七色分層：白 / 綠 / 藍 / 紫 / 金 / 紅 / 彩，作為卡面、招牌角色、活動 / 限定 / 旗艦包裝的衍生顯示層
- 角色分類欄位：`civilian` / `general` / `famed` / `mythical` / `titled`
- 自動判定演算法：雙軸制（`maxStat + avg5`），取較高 tier
- 商業分層不得反寫核心平衡；女將商業補正只允許作用於商業層，不直接改原始屬性
- 少數招牌角色可用白名單包成金卡；活動卡 / 限定卡 / 旗艦卡可與純戰力分開
- 門檻值改為 JSON 可配置，不寫死在程式碼
- `GeneralDetailRarityTier` 已擴展加入 `mythic`（UR）
- `characterCategory` 納入 `GeneralConfig` 與資料管線

### M1-E：戰法與奧義自動產生器 (Tactic & Ultimate Generator)

- **目標**：根據武將的稀有度（Rarity Tier）、歷史資料與兵種適性，自動分配初始的「天賦戰法（來自血統繼承/初始攜帶）」與「個人奧義」。
- **分配規則 (依據 Rarity tier)**：
  - `UR` (mythic)：3 個天賦戰法 + 1 個武將奧義（必定專屬）。
  - `SSR` (legendary)：2 個天賦戰法 + 1 個武將奧義。
  - `SR` (epic)：1 個天賦戰法，無武將奧義。
  - `R` (rare) / `N` (common)：隨機給予 1 個基礎天賦戰法 或 0 個，無武將奧義。
- **產生依據**：
  - **戰法選取**：依據武將的最高適性（例如：弓兵 S 適性必定骰出遠程系戰法）與屬性（如高智力給計略）從對應庫中挑選。
  - **奧義生成**：名稱與描述可由 AI (`general-tactic-generator` Skill) 配合武將的「勢力、生平趣聞」自動生成，建立專屬感（例如趙雲對應【七進七出】或【龍膽激昂】）。
- **實作方式**：
  - 新增 Node.js 腳本 `tools_node/generate-general-tactics.js`。
  - 這支腳本將讀取 `generals-base.json`，進行規則判定，然後將結果寫回 JSON 供後續 Pipeline 使用。

> **實作狀態（2026-04-15）: ✅ 完成**（errors=0, warnings=0） — 腳本已落地並執行，231/235 武將有戰法，13/235 有奧義（全部 legendary 武將已覆蓋）；戰場 runtime 已正式接入 `tacticSlots / ultimateSlots`，GeneralDetailPanel 與 BattleScene HUD 皆直接消費正式資料。
> **新增補齊名單**：liu-bei / sun-quan / zhuge-liang / zhou-yu / sima-yi / guo-jia / cao-zhi / diao-chan。
> **戰場技能覆蓋（2026-04-15）**：`skills.json` 目前含 13 條 signature battle skills；13 位 legendary 武將皆具備 `battlePrimarySkillId`，其 65 條 `ultimate-definitions` 亦全部具備 `battleSkillId` 對應。

### M1-F：虛擬血統祖先產生器 (14-Person Bloodline Generator)

- **目標**：為了讓史實武將在培育系統（子嗣 5 歲開獎）時具備完整的可遺傳基因池，系統需要為每位基礎史實武將自動「向上回溯」，生成 3 代共 14 位（父母 2 + 祖父母 4 + 曾祖父母 8）的虛擬祖先血統矩陣。
- **產生規則與實作細節**：
  1. **基因逆向推導**：先讀取該名史實武將的核心數據（如高武力、S級適性、Rarity），由演算法將這些優勢基因（六色因子如：紅色屬性因子、綠色戰法種子）隨機「向上分配」給其 14 位虛擬祖先，確保其子嗣在開獎時有合理機率遺傳到名將本系的強勢特徵。
  2. **姓氏與血脈繼承邏輯**：
     - **父系祖先**（如父親、祖父、曾祖父）強制綁定與該史實武將相同的姓氏，且自動繼承或發放具備 `[家]` 標籤的特有家傳戰法因子。
     - **母系祖先**則依據歷史資料（若有）或《三國》百家姓庫隨機抽取。
  3. **資料羽量化**：自動產生的 14 位祖先全部帶有 `is_virtual: true` 標籤，不生成 3D 模型、圖標與過多歷史背景。僅記錄 `uid`、`name`、`gender` 與核心的 `gene_refs`（身上的遺傳因子 ID 陣列）。
  4. **血緣共用與去重**：配合 M3 的 `PersonRegistry`，腳本需具備關係感知。例如先產生了「曹操」的 14 位祖先後，在處理「曹丕」時，腳本會偵測到其身分，並直接將曹丕的父系鏈接 Point 到曹操及其既有祖先樹上，絕不重複產生。
- **實作方式**：
  - 新增 Node.js 腳本 `tools_node/generate-bloodline-ancestors.js`。
  - 在 `generate-general-tactics.js` (M1-E) 執行後緊接執行，直接將 14 人的關聯圖譜與 `gene_refs` 分配結果寫入到本地資料庫（或 L3 血統專屬 JSON）之中。

> **實作狀態（2026-04-15）: ✅ 完成 + P1 修復完成** — 腳本已落地並執行，3279 虛擬祖先已分配 gene_refs（含 35 位女武將補完），11 組父系親子鏈接已去重。
> **驗收結果**：`errors=0, warnings=0`（全 235 位武將全數通過 ✅）

### M1-E / M1-F Data Schema 彙整（生產版）

> 以下欄位為 `generals-base.json` 每筆武將紀錄新增的生產欄位（2026-04-15 M1-E/M1-F 執行後正式落地）。

#### `tacticSlots[]`（每個武將最多 3 個）

```json
{
  "slotNumber": "number   // 1~3",
  "tacticId":   "string   // 對應 tactic-library.json[].id",
  "isLocked":   "boolean  // false = 初始可用，true = 條件鎖定",
  "source":     "string   // bloodline | talisman | education | event"
}
```

| 稀有度 | 最多槽數 | 備註 |
|---|---|---|
| mythic (UR) | 3 | 評分最高的 3 個戰法 |
| legendary (SSR) | 2 | 評分最高的 2 個 |
| epic (SR) | 1 | 1 個 |
| rare (R) | 1 | 需評分 ≥ 5 才給 |
| common (N) | 0 | 無天賦戰法 |

#### `ultimateSlots[]`（有奧義定義的武將才有 5 個）

```json
{
  "slotNumber":   "number  // 1~5（對應轉生次數）",
  "ultimateId":   "string  // 對應 ultimate-definitions.json[].id",
  "isLocked":     "boolean // 初始全鎖，轉生後依序解鎖",
  "source":       "string  // template（名將固有）| bloodline（家傳覺醒）"
}
```

> **templateId → ultimateSlots 對應**：腳本透過 `deriveTemplateId(general)` 計算 `GEN_{FACTION}_{ID}` 後，在 `ultimate-definitions.json` 中過濾匹配的條目並依 slot 欄位排序寫入。
> **目前覆蓋率（2026-04-15）**：65 條 `ultimate-definitions` / 13 位 legendary 武將 / 每位 5 槽完整。

#### `gene_refs[]`（PersonRegistry PersonRecord 欄位，虛擬祖先使用）

["gene_strategy", "gene_politics", ...]
// 每個元素為 gene-dictionary.json 中的 gene id
// 虛擬祖先帶 3~5 個 gene_refs，由 assignGenesForSlot() 依 bloodline-templates 權重分配

### 2.2 本地分層儲存架構

**三層分檔策略（解決 JSON 過大問題）：**

| 層級 | 檔案 | 載入時機 | 大小估算（中期） |
|------|------|---------|----------------|
| L0 索引層 | `generals-index.json` | App 啟動 | ~50KB（350 武將 × 150B） |
| L1 活躍層 | `generals-active.json` | 進入 Lobby | ~200KB（當前陣容 30 位 × 5KB + 經濟/時間） |
| L2 倉庫層 | `generals-storage-{page}.json` | 開倉庫/圖鑑 | 每頁 ~100KB（20 位 × 5KB） |
| L3 故事層 | `generals-lore-{id}.json` | 點開人物詳情 | 每位 ~2-5KB |
| L4 歷史層 | `battle-records-{season}.json` | 查看戰績 | 每季 ~50KB |
| L5 血統層 | `bloodline-tree-{uid}.json` | 查看血統樹（由 `PersonRegistry` + `BloodlineGraph` 衍生） | 每位 ~1-3KB |

Unity 對照：L0=Addressables catalog、L1=首屏 Preload Group、L2=Lazy Load Group、L3-L5=Remote Bundle

**關鍵技術：**
- 使用 IndexedDB (Web) / SQLite (Native) 取代純 JSON 檔案
- `DataStorageAdapter` 抽象層：統一 API，底層自動選擇 IndexedDB 或 SQLite
- 分頁查詢：倉庫採 page cursor，不一次載入全部

### 2.3 血統資料正規化

**問題描述：**  
如果張飛的孫子和趙雲的孫女結緣，他們子嗣的 `Ancestors_JSON` 會重複儲存張飛/趙雲的因子資料，造成大量冗餘。

**正規化方案：**

```
┌─────────────────────┐     ┌──────────────────────┐
│  Person Registry    │     │  Bloodline Link      │
│  (主檔)             │     │  (關係圖)             │
├─────────────────────┤     ├──────────────────────┤
│ uid: string (PK)    │←────│ child_uid: string    │
│ template_id: string │     │ parent_uid: string   │
│ name: string        │     │ relation: F|M        │
│ gene_refs: string[] │     │ generation: number   │
│ ep_base: number     │     └──────────────────────┘
│ faction: string     │
│ is_virtual: boolean │
└─────────────────────┘
```

- 每個人（含虛擬祖先）只有一筆 `Person Registry` 記錄
- `Ancestors_JSON` 改為 `ancestor_chain: string[]`（14 個 uid），查表取因子
- **節省量估算**：同一祖先被 N 個子嗣引用時，從 N × 200B 降為 N × 16B (uid) + 1 × 200B (主檔)
- 中期 350 位武將 × 14 祖先中平均 30% 重複 → 省 ~30% 血統資料量

### 2.4 壓縮策略

| 層級 | 技術 | 壓縮率 | 適用場景 |
|------|------|--------|---------|
| JSON 欄位縮寫 | 因子用 `["B","STR_05",3]` 取代完整物件 | ~50% | 全部 JSON |
| MessagePack | 二進制序列化替代 JSON.stringify | ~40% | 存檔快照 + 同步傳輸 |
| pako (zlib) | gzip 壓縮 MessagePack 結果 | ~60-70% | 同步上傳/下載 |
| 增量 Delta | 只傳差異欄位 | ~80-95% | 每次同步 |

**中期資料量估算：**
- 全量 JSON：~2MB
- MessagePack：~1.2MB
- + gzip：~400-600KB
- Delta sync（日常）：~5-20KB / 次

### 2.5 增量同步協議（Delta Sync）

**現有設計限制**：全量 `Data` + `Action_Records` → Server，中期 2MB 每次全量上傳不合理

**三階段演進方案：**

```
Phase 1（MVP）: 全量同步
  Client → { Data, Action_Records } → Server
  Server → Event Replay → 覆蓋存檔

Phase 2（正式版）: Delta 同步
  Client → { delta_patch, Action_Records, data_hash } → Server
  Server → 驗證 hash → apply delta → confirm

Phase 3（多裝置）: CRDT-lite
  Server 維護 vector clock
  多裝置各自離線 → 上傳時 merge
  衝突解決：Action_Records 的 timestamp 排序 + Server 仲裁
```

**Delta Patch 格式（JSON Patch RFC 6902）：**

```json
{
  "version": "1.3.0",
  "base_hash": "abc123...",
  "patches": [
    { "op": "replace", "path": "/Generals/G001/Stats/STR", "value": 95 },
    { "op": "add", "path": "/Generals/G350", "value": { "..." : "..." } },
    { "op": "remove", "path": "/Battle_Records/S120" }
  ]
}
```

### 2.6 資料增長控管策略

| 資料類型 | 控管策略 | TTL |
|---------|---------|-----|
| 戰鬥日誌 | 最近 100 場完整 → 之前按季摘要 → 2 年前刪除 | 100 場 / 2 年 |
| 培育會話 | 完成後僅保留摘要（最終數值 + 關鍵事件） | 完成即壓縮 |
| 史實武將（已故） | 轉為 Spirit_Card 快照，保留基本檔 / 血脈索引 / 復活資格；完整主體移冷歸檔層 | 永久保留 |
| 動態後裔（存活） | 熱/冷分層；非活躍資料季結封存，只保留目前玩法必要欄位常駐 | 每季滾動整理 |
| 動態後裔（已故） | 轉為 Spirit_Card + 血脈摘要；完整主體移待刪區，達條件後批次刪除 | 待刪 1–4 季 |
| 已滅絕支系 | 壓縮為 FamilyBranchSummary，只保留族譜節點 / 基因摘要 / 代表事件 | 支系滅絕即收斂 |
| 故事條 | LRU 快取，最多 20 位武將的完整故事常駐記憶體 | LRU 20 |

**長局單機資料生命周期策略（2026-04-08 追加）**

- **A 類：史實武將** — 視為世界觀常駐資產。即使死亡，也只做「熱資料釋放」，不做硬刪除；保留 `SpiritCard`、基本身分檔、血脈索引與可復活資格。
- **B 類：動態後裔（玩家 / AI 自行繁衍）** — 視為可回收資料。存活時只保留目前玩法需要的 hot fields；故事、歷程、戰績、完整祖先快照全部走 cold archive / season rollup。
- **C 類：動態後裔（已故）** — 死亡後立刻產生 `SpiritCard`，只保留最小必要集：`uid`、父母鏈、`gene_refs`、出生/死亡年、勢力、`rarityTier`/`characterCategory`、`ep_snapshot`、英靈卡索引。完整主體進入 pending-delete quarantine。
- **D 類：不可再復活的後裔** — 若同時滿足「非史實」、「無復活池資格」、「無存活後代依賴」、「無未結算事件 / 任務 / 合約引用」，即可由批次 GC 從待刪區清除完整主體，只留下血脈摘要或支系摘要。
- **AI 陣營額外限制** — AI 後裔必須有生育配額、世代上限或人口預算；超額時優先走支系壓縮與 seed 重建策略，而不是持續常駐全量武將物件。
- **建議批次節奏** — 每季做一次 rollup（摘要化），每年做一次 branch compaction（支系壓縮），每 4 季做一次 pending-delete 清掃；玩家可在設定中保留「最近 X 季家族史」供回看。
- **補充收斂方案**：
  - 非關鍵 AI 後裔可只存 lineage seed + delta，必要時再重建，而不是永久存完整物件。
  - 已滅絕支系只保留 `FamilyBranchSummary`，不再保留每一代完整行為日誌。
  - 戰績、社交、事件日志一律 seasonal rollup，不允許綁死在後裔主體上無限累積。
  - 設定全域存檔配額警戒線；超標時先降級為摘要模式，再執行延後清掃，而不是讓存檔無限制膨脹。

## 三、稀有度與角色分類系統

### 3.1 五級稀有度

| 等級 | 代號 | 說明 |
|------|------|------|
| N | common | 一般武將；核心平衡最低層 |
| R | rare | 稀有武將；基礎可培養層 |
| SR | epic | 精銳武將；中堅戰力層 |
| SSR | legendary | 傳說武將；高端戰力層 |
| UR | mythic | 神話武將；僅神話 / 手動指定 |

- `rarityTier` 是**核心平衡 tier**，只負責戰力、EP、適性標準、資料盤點與系統解鎖條件。
- 核心 tier 不等於卡面框色，也不直接等於轉蛋包裝等級。
- UI 顏色可引用 tier 色作為輔助視覺，但不得把商業包裝結果反向回寫 `rarityTier`。

**商業化 7 色分層（Commercial Band）**

| Band | 說明 | 主要用途 |
|------|------|---------|
| white | 白 / 基礎 | 基礎卡面與低商業包裝 |
| green | 綠 / 進階 | 初階稀有展示 |
| blue | 藍 / 精良 | 主流量產卡面 |
| purple | 紫 / 菁英 | 主力商業高稀有層 |
| gold | 金 / 招牌 | 招牌角色與主視覺卡池 |
| red | 紅 / 限定級 | 活動、聯動、期間限定 |
| rainbow | 彩 / 旗艦級 | 旗艦包裝、最高商業展示 |

- `commercial band` 是**衍生展示層**，由 `rarityTier`、`characterCategory` 與商業規則共同計算，不作為戰力判定真相。
- 商業 band 允許和核心 tier 解耦，用來支援女將商業呈現、招牌金卡、活動卡、限定卡、旗艦卡等包裝需求。
- 設計原則：**戰力平衡不要被商業包裝污染；商業包裝也不要回寫原始五維。**

### 3.2 角色分類欄位

- `civilian`：平民
- `general`：一般將領
- `famed`：名將
- `mythical`：神話人物
- `titled`：帝王/封號（未來擴充）

### 3.3 稀有度自動判定演算法（雙軸制）

- **雙軸**：maxStat（最高單項屬性）+ avg5（六維去 luk 後平均值）
- 取兩軸各自判定後的**較高 tier**
- `mythic` 等級須明確設定 `characterCategory: 'mythical'` 或手動指定 `rarityTier`

**門檻值（可 JSON 設定，不寫死）：**

| Tier | maxStat 門檻 | avg5 門檻 |
|------|------------|---------|
| legendary | ≥ 95 | ≥ 80 |
| epic | ≥ 80 | ≥ 65 |
| rare | ≥ 65 | ≥ 50 |
| common | < 65 | < 50 |

**計算公式：**

```text
Axis A（偏科天才）
  maxStat = max(str, int, lea, pol, cha)

Axis B（綜合實力）
  avg5 = average(str, int, lea, pol, cha)

coreTier = max(resolveByMaxStat(maxStat), resolveByAvg5(avg5))
```

**商業分層計算（獨立於核心 tier）：**

```text
commercialScore =
  baseScoreByTier(coreTier)
  + categoryBonus(characterCategory)
  + genderScoreBonus(gender)
  + thresholdBonus(maxStat, avg5, commercialCha)
  + aptitudeBonus(A/S count)

commercialBand = scoreBands(commercialScore)
commercialBand = applyFeaturedBandWhitelist(commercialBand, id)
commercialBand = applyPremiumFallback(commercialBand)
commercialBand = applyGenderCap(commercialBand, gender)
commercialBand = applyBandAvg5Floor(commercialBand, gender, avg5)
```

**現行商業規則結論：**

- 女將魅力補正只存在商業層：`commercialCha = round(sqrt(cha) * 10)`；不改寫原始 `cha`。
- 招牌金卡採 `featuredBandIds` 白名單；不是所有 `legendary` 都自動變金卡。
- `red` / `rainbow` 屬於活動、限定、旗艦等商業包裝層，不能拿來定義核心戰力。
- 男將 `purple` 需額外滿足 `avg5` 下限，避免商業高稀有層數量膨脹。
- 原則上只維護一套主資料：`rarityTier` / `characterCategory` / 原始五維；商業 band 一律由規則衍生。

**公式層平衡設計原則（2026-04-08）：**

- 遊戲平衡不應直接截斷基本屬性（str/int/lea/pol/cha）的數值上限，應透過公式層實現差距收斂。
- **戰鬥公式**：以加權組合（`STR×0.7 + LEA×0.3`）或指數壓縮（`stat^0.7`）降低頂端差距，不修改原始屬性。
- **生產 / 政務公式**（已實作基礎版）：`adminOutput = floor(sqrt(pol) × multiplier × aptitudeBonus)`、`diplomacyOutput = floor(sqrt(cha) × multiplier × aptitudeBonus)`；以開根號曲線收斂高端差距。
- **適性加乘**（已實作 helper，待串接到完整戰場 / 內政流程）：`FormulaSystem.aptitudeMultiplier(grade)`，目前規則為 S=1.2x、A=1.1x、B=1.0x、C=0.9x、D=0.8x。
- 此原則與商業層分離原則相同：基本屬性是「歷史詮釋的語意」；公式層是「遊戲體驗的平衡」，兩者職責分離，互不污染。
- 實作位置：`assets/scripts/core/systems/FormulaSystem.ts`（現有 `calculateGeneralAttack`、`calculateAdminOutput`、`calculateDiplomacyOutput`、`aptitudeMultiplier`）；`assets/scripts/core/models/GeneralUnit.ts` 已補 `pol` / `cha` 正式欄位。

**代碼位置**：`assets/scripts/ui/components/GeneralDetailOverviewMapper.ts` → `resolveRarityTier()`

### 3.4 整合位置

- `assets/scripts/core/models/GeneralUnit.ts`：`GeneralDetailRarityTier`、`characterCategory` 欄位
- `assets/resources/data/rarity-thresholds.json`：核心 tier 門檻與 `commercialRarity` 規則來源
- `tools_node/audit-generals-commercial-balance.js`：核心 tier / 商業 band / category / aptitude 一致性稽核
- `general-balance-tuner` skill：自動分類入庫

## 七、關鍵技術決策

| 決策 | 選擇 | 理由 |
|------|------|------|
| 本地儲存引擎 | IndexedDB (Web) / SQLite (Native) | JSON 檔超過 1MB 後 parse 耗時劇增；DB 支持分頁查詢 |
| 血統正規化 | Person Registry + 關係圖 | 跨世代引用去重，減少 30% 儲存 |
| 序列化格式 | MessagePack + gzip | 遊戲業界標準，壓縮率 70%+ |
| 同步協議 | JSON Patch (RFC 6902) | 標準化、可驗證、有成熟函式庫 |
| 史料收集 | Wiki 爬取 + AI 生成 | 公開史料自動化，AI 補充故事性內容 |
| 因子儲存 | gene_id 引用 + 共用字典 | 已在用，正確做法 |
| Skills 執行環境 | 本機 Copilot Skills | 零 API 成本，無外部依賴 |
| 稀有度計算 | 雙軸制（maxStat + avg5） | 避免單一指標偏差，取較高 tier 較保守 |

## 八、風險與緩解

| 風險 | 影響 | 緩解 |
|------|------|------|
| IndexedDB 在某些瀏覽器達到限額 | 超過 50MB 限制 | 壓縮 + 歸檔策略確保 < 10MB |
| AI 生成故事品質不穩 | 文案風格不一致 | 人工審核 + 風格模板 + 品質閥門 |
| MessagePack 與 Cocos Creator 相容性 | 需要適配 | 用純 JS 版本（不依賴 WASM） |
| 多裝置同步衝突 | 資料遺失 | Phase 3 CRDT-lite，MVP 先單裝置 |
| 動態後裔長局繁衍造成存檔膨脹 | 單機長局存檔失控、載入變慢 | 史實/後裔分區、SpiritCard 快照、pending-delete、支系摘要壓縮、AI 生育配額 |
| Wiki 爬取被封鎖 | 史料收集中斷 | 手動匯入備援；使用 User-Agent 規避限制 |
| 遷移腳本破壞現有資料 | 血統資料損毀 | 遷移前完整備份；空跑 dry-run 確認 |
