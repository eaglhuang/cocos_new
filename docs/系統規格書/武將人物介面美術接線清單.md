## 1. 接線原則

1. 節點名稱優先固定，icon 與貼圖可換，但節點 ID 不可隨意改動。
2. 圖層順序固定為：遮罩層 → 主面板層 → 頭部資訊層 → 摘要層 → 頁籤層 → 內容層 → 底部操作層。
3. 未公開欄位以 `🔒`、`?` 或灰階/折疊方式處理，不新增額外提示文字遮蓋主要資訊。
4. icon 資產優先使用獨立 SpriteFrame，若正式美術尚未完成，可暫用 placeholder 圖示。
5. 頁籤按鈕與摘要卡片需維持 44×44 以上觸控區，避免手機端點擊誤差。

### 骨架材質規則

1. `ContentFrame` 優先使用 AI Asset Pipeline 導入的大型框體，若已有審核通過的通用框原稿，應優先使用通用框派生 variant，而非沿用程式化暫存框。
2. `HeaderPanel`、摘要卡、頁內 section card、Footer 採同一族九宮格框體，維持一致的金邊材質感；目前可先共用 `sprites/common/popup_card_bg`，後續再統一切換到正式通用框家族。
3. 高密度頁籤（血統 / 技能 / 適性）不得再以裸 label 直接堆疊；至少要包成 section card。
4. 頁籤按鈕需保留 normal / pressed / disabled 狀態圖或等效視覺層次，不能只靠文字變色。
5. `detail.bg.fullscreen` / `general-list` 共用的 `bg_ink_main` 必須走不對稱九宮格：左 border 保留立繪留白，右 border 保留山景與角飾，禁止再以 `simple` 方式直接整張縮放。

### 2.2 分層落地規則

1. 相容期允許維持單一 `*.bg` slot 對應既有 panel node，但新增美術時必須依照上表先交出分層原稿。
2. 正式升級期應把單一 panel 拆成 2 到 4 個子節點，順序固定為 `fill -> bleed -> frame -> content`。
3. `bleed` 不可用實心色塊代替；若需要色偏，應以低對比煙墨貼圖或局部暗角處理。
4. `accent` 只允許用在 active tab、摘要高亮、特殊警示或稀有度，不可整個畫面全面開啟。
5. 若同一區塊需要 win / lose / locked 等狀態，優先替換 `accent` 或 `fill`，不要複製整套 `frame` 造成資產爆量。

## 8. 獨立全幅立繪面板 (GeneralPortraitPanel)

此面板專精於高視覺衝擊的全幅立繪展示與核心摘要，不同於 `GeneralDetailPanel` 的頁籤式詳細資料，主要應用於：抽卡獲得武將、圖鑑立繪鑑賞、大廳名將展示等。

### 8.1 骨架節點接線表

| 節點名稱 | 類型 | 建議資產 / 樣式 | 綁定資料 | 備註 |
|---|---|---|---|---|
| `GeneralPortraitRoot` | container | 無 | 無 | 面板根節點，負責進出場動畫 |
| `Overlay` | panel | `common.overlay` | 無 | 點擊關閉全螢幕背景黑底 |
| `PortraitImage` | image | 動態載入 SpriteFrame | 由邏輯載入 | **獨立立繪圖**，為求不變形，強制配置 `SizeMode.RAW` |
| `InfoPanel` | panel | `general.info.bg` | 無 | 半透明資訊背板，靠右直向排版 |
| `VitalityBox` | panel | `general.box.bg` | 無 | 單行屬性面板（如：精力） |
| `StatsBox` | panel | `general.box.bg` | 無 | 多行屬性面板（如：六色屬性） |
| `TagsBox` | panel | `general.box.bg` | 無 | 多行標籤面板（如：核心指令與標籤）|
| `BtnClose` | button | `common.button.primary` | `ui.close` | 關閉按鈕 |

### 8.2 視覺設計建議與防呆規則

1. **立繪絕對比例**：為對應不同尺寸的外部匯入立繪（如：376x670 或 1024x1024），底層 `UIPreviewBuilder` 禁止對 `PortraitImage` 進行強制延展，組件層將鎖定 `SizeMode = RAW`，排版錨點需設定明確的 left/bottom 建置（如 left: 300, bottom: 0）使其居於左半側。
2. **文字溢位縮放保護**：部分屬性數值（如：武力 98, 精力 100/100）若未給定寬度，將被預設的 `Overflow.SHRINK` 壓縮至 0 導致消失。因此所有會變動的 Label 必須明確宣告足夠的寬度 (如 `width: 250` 或 `width: 550`) 以維持正常的安全展示空間。
3. **動態載入無縫 Fallback 機制**：Cocos 編輯器容易在反覆修改立繪檔案時重置 `.meta` 的 Texture type，導致單張 SpriteFrame 讀不到。開發期間的立繪展示，程式將會現場藉由 `Texture2D` 反解出 SpriteFrame，確保立繪隨插即用！
