# 武將人物介面規格書

> **文件目的**：定義武將人物介面的正式資訊架構、顯示規則、鎖定規則與資料對映，作為 `GeneralDetailPanel`、大廳武將卡、詳情彈窗、血統/戰法/適性延伸畫面的母規格。
> **適用範圍**：武將角色頁、詳情頁、彈窗頁、人物卡、收藏頁、未來培育頁的角色摘要模組。
> **核心原則**：預設顯示已知資訊；未解鎖、未發現或不想揭露的欄位，一律以鎖圖示 `🔒`、`?` 佔位、折疊區塊或灰階卡表現。
> **Unity 對照**：此介面相當於 Unity 的角色詳情面板 + 樣板卡片 + 多頁籤資訊視圖，由 ScriptableObject 資料驅動，View 只負責呈現。

---

## 1. 文件定位與設計目標

### 1.1 定位

武將人物介面是玩家查看「一名武將完整面貌」的主要入口。它不是單純的數值頁，而是結合以下資訊的正式角色介面：

1. 基本身份與生命週期
2. 六色屬性與能力評語
3. 血統、因子、爆發力與覺醒狀態
4. 戰法、教官啟發與習得狀態
5. 兵種、地形、天氣適性
6. 隱藏欄位、未解鎖欄位與折疊欄位

### 1.2 設計目標

1. 讓玩家一眼辨識這名武將「是誰、強在哪、能做什麼、還缺什麼」
2. 兼容正式版的長線養成資料，避免未來擴欄位時大改 UI
3. 保留手機可讀性，避免單頁資訊過載
4. 對接現有 `GeneralUnit` 與 `GeneralDetailPanel`，減少實作轉換成本
5. 讓血統、戰法、適性這三大子系統在同一頁形成完整敘事

### 1.3 非目標

1. 不在這份文件中定義戰鬥數值公式本體
2. 不在這份文件中定義培育流程本體
3. 不在這份文件中定義轉蛋或抽卡 UI
4. 不把所有隱藏資訊直接揭露給玩家

## 3. 介面總覽

### 3.1 建議容器形式

武將人物介面建議採用「頁籤式詳情彈窗」或「抽屜式人物面板」。

#### 方案 A：頁籤式詳情彈窗
- 適合完整展示多系統資訊
- 適合中後期正式版
- 建議放在 PopupLayer

#### 方案 B：人物卡 + 詳情抽屜
- 適合大廳、列表、收藏頁
- 點擊人物卡後右側或底部滑出詳情
- 適合手機橫豎切換

### 3.2 建議頁面結構

```text
GeneralDetailPanel
├─ Header（頭像、名稱、稱號、陣營、職能、狀態）
├─ Summary Strip（HP、SP、年齡、精力、EP 評級）
├─ Tab Bar
│  ├─ 基本資料
│  ├─ 六色屬性
│  ├─ 血統與因子
│  ├─ 戰法與技能
│  ├─ 戰場適性
│  └─ 延伸資訊
└─ Footer Actions（返回、收藏、鎖定、比較、分享）
```

### 3.3 顯示優先級

1. 必顯：名稱、稱號、陣營、職能、狀態、HP、SP、年齡、精力
2. 高優先：六色屬性、EP 評級、血統簡覽、已習得戰法、兵種/地形適性
3. 中優先：因子槽、血脈警告、可習得戰法、戰法等級、天氣適性
4. 低優先：UID、Bloodline_ID、Template_ID、來源標籤、進階劇本標籤

### 3.3.1 血脈內容優先級補充

- 武將立繪不再只是展示美術，而是進入 `表象面 / 血脈面` 的切換入口
- 角色內容層正式分成兩種：
  - `歷史趣聞`：對應史實逸事、人格魅力與大眾熟悉記憶
  - `血脈傳聞`：對應命紋靈獸、祖紋命篆與其勇猛、謀略、魅力來源的合理化敘事
- 兩種故事內容在視覺上應共用同一個抽屜層，但標籤、色調與標題要能一眼區分

### 3.4 框體分層視覺規格

武將人物介面若要達到正式水墨渲染質感，不應再把每個 panel 視為一張單獨的九宮格框，而應拆成可疊合的框體家族。

Unity 對照：這相當於把一個 Panel Prefab 拆成 `Frame`、`Inner Glow/Bleed`、`Fill` 三個子 Image，再由外層容器統一排版；而不是只放一張 sliced sprite 撐完整個視覺。

#### 分層原則

1. `frame`：只承擔外輪廓、金屬紋飾、角花與主要結構線，避免把大面積灰霧直接烘在最外框。
2. `bleed`：負責邊框往內容區滲入的墨暈、霧邊與暗角，是讓框與背景自然過渡的主因。
3. `fill`：負責內容區的大面積紙感、煙墨、山霧或深色底紋，不應以單色實心底板取代。
4. `accent`：只用於 active tab、稀有度高亮、警示狀態或頁面焦點，不應變成每個 panel 都有的常駐發光層。

#### 套用範圍

1. `PopupBody / RightContentArea` 使用大型框體家族，必須具備 `frame + bleed + fill` 三層。
2. `HeaderPanel`、`SummaryStrip`、`FooterPanel` 使用中型框體家族，至少具備 `frame + fill`，建議補 `bleed`。
3. 頁內卡片如血統摘要、技能卡、適性卡使用 section 家族，不得再直接與 header 共用完全相同的視覺層級。
4. `RightTabBar` 與 tab button 應拆成「tab rail」與「tab plate」兩種家族，避免整條頁籤列和單顆按鈕共享同一張小卡框。

#### 質感目標

1. 讓邊框與背景的過渡發生在 `bleed` 層，而不是靠透明邊緣硬撐。
2. 讓角色立繪後方的大底圖、右側內容框、頁內卡片維持同色系不同密度，而不是各自獨立成一塊塊貼紙。
3. 讓 active 與 idle 的差異主要來自 `accent` 與亮度階層，不是單純換色塊。

### 3.5 血脈世界觀微調原則

- 本頁面不做整體 UI 翻新，只在既有骨架上做世界觀微調
- 允許新增的元素：
  - `祖紋命篆` 紋樣
  - `命紋靈獸` 小型 badge / 虛影 / 角標
  - 暗金、玉白、月銀、墨青、暗赤的微調色階
  - 血脈面點亮時的墨痕、玉刻光痕、祖紋連線
- 不允許新增的方向：
  - 科技感 HUD 線條
  - 恐怖污染或怪物化裂臉
  - 重奇幻神魔粒子鋪滿畫面
- 角色立繪的改動原則是 `符號加法`，不是推翻原本商品化畫風

### 3.6 日常人物頁 v2 母型

依據目前概念 proof，日常人物頁正式收斂為 `人物主導 + 血脈輔助 + 故事條固定化` 的骨架。

#### 桌面版首選結構

1. 左側 `38%~42%`
   - 近全身角色主立繪
   - 人物永遠是第一主角
2. 中右主資訊區 `42%~48%`
   - 名稱、稱號、稀有度、五屬、戰鬥特性
   - 使用單一主卡或同語系分層卡片，不做破碎小模組拼貼
3. 右側窄欄 `8%~12%`（可選）
   - 頁籤 rail、切換器或血統標籤
4. 下方固定故事條 `16%~22%` 高
   - 5 格或 6 格連續故事圖
   - 作為頁面的一部分，而不是小型附贈漫畫

#### 血脈區定位

- 日常人物頁中的血脈區不是第二主角，也不是雙面世界
- 它應像 `深層人格區`：
  - 血脈摘要
  - 覺醒傾向
  - 血脈性格註記
  - 命紋靈獸徽記
- 不應做成另一張大卡把人物重複一遍

#### 故事條定位

- 桌面版 premium layout 優先使用固定故事條，而不是預設抽屜
- 故事條建議 5~6 格，由左往右讀
- 節奏建議：
- 前段：歷史人生
- 中段：命運轉折
- 後段：血脈暗示 / 祖靈牽引

### 3.7 日常人物頁 v3 與多分頁整合原則

- `GeneralDetailPanel` 仍是本規格的正式功能容器，負責完整的多分頁人物詳情。
- `GeneralDetailBloodlineV3` 的定位不是另一套平行正式人物頁，而是 `GeneralDetail` 的預設首頁 / 總覽態視覺骨架。
- 玩家從大廳武將列表點入人物頁時，第一眼應優先看到 `日常人物頁 v3` 的總覽畫面，再由頁籤進入 `基本資料 / 六維屬性 / 血統與因子 / 戰法與技能 / 戰場適性 / 延伸資訊`。
- `日常人物頁 v3` 承接的是 `人物立繪 + 核心摘要 + 血脈摘要 + 5~6 格故事條`，屬於高可讀的首頁殼層；它不是用來取代後續深度分頁。
- 實作上允許把 `GeneralDetailBloodlineV3` 做成第一個頁籤、預設展開態，或 `Overview` 狀態，但不得把既有詳情分頁功能直接刪除或降格成次要規格。
- 欄位分工上，`日常人物頁 v3` 應優先承接 `TabBasics` 的核心身分資訊與 `TabBloodline` 的摘要資訊；完整血統矩陣、因子細節、戰法細項與適性列表仍保留在對應分頁。

## 5. UI 資訊架構

### 5.1 Header 區

Header 是人物介面的第一視覺焦點，必須穩定呈現以下內容：

1. 頭像 / 立繪 / 角色剪影
2. 名稱
3. 稱號
4. 陣營
5. 職能（Combat / Mentor）
6. 狀態（Active / Training / Retired / Dead）
7. 稀有度 / 版本標籤（若有）
8. 命紋靈獸 badge 或祖紋命篆縮記號
9. 立繪點擊提示（可進入血脈面 / 故事層）

### 5.2 Summary Strip 區

Summary Strip 放在 Header 下方，快速顯示核心狀態：

1. HP
2. SP
3. 年齡
4. 精力
5. EP 分級
6. 主技能是否可用

### 5.3 Tab 1：基本資料

基本資料頁顯示不需要展開即可理解的核心資訊。

#### 需顯示欄位
- UID
- Template_ID
- Name
- Title
- Faction
- Gender
- Age
- Role
- Status
- Lifespan
- Vitality_Current / Vitality_Max

#### 建議視覺
- 左側放頭像與狀態章
- 右側放文字欄位與短標籤
- `Lifespan` 建議預設隱藏為「天命值：🔒」或條件顯示

### 5.4 Tab 2：六色屬性

六色屬性頁用於顯示能力骨架與戰鬥定位。

#### 需顯示欄位
- STR
- INT
- LEA
- POL
- CHA
- LUK

#### 視覺規則
- 採列表 + 雷達圖雙呈現
- 主力屬性高亮
- 未主修屬性灰化但不隱藏
- 可附帶一句定位評語，例如「武力型」、「謀略型」、「統率型」

### 5.5 Tab 3：血統與因子

這是本規格最重要的區塊之一，需兼顧輕度與硬核玩家。

#### 預設內容
1. EP 卡片
2. EP_Rating 文案
3. 父母簡覽
4. 因子槽簡圖
5. 近親警告或歸宗標籤
6. 命紋靈獸名稱 / 縮圖
7. 血脈面狀態提示
8. `歷史趣聞 / 血脈傳聞` 入口

#### 展開內容
1. 14 人血統矩陣
2. 虛擬祖先標記
3. Bloodline_ID
4. 覺醒名號
5. 因子來源軌跡
6. 祖紋命篆說明
7. 命紋靈獸圖騰與對應符號 slot
8. 父母系譜與因子流向
9. 子嗣預測摘要與覺醒預兆

### 5.6 Tab 4：戰法與技能

#### 需顯示欄位
- 主戰法
- 已習得戰法列表
- 可習得戰法列表
- 未啟發戰法列表
- 戰法等級
- TP 消耗
- 戰法類型
- 技能簡述

#### 三態規則
1. 已習得：正常彩色顯示，可點擊看詳情
2. 可習得但未啟發：半透明、鎖定、顯示教官需求
3. 無法習得：灰階、鎖定、顯示因子不符或條件未滿

### 5.7 Tab 5：戰場適性

#### 需顯示欄位
- 兵種適性：騎 / 步 / 弓 / 機械
- 地形適性：平原 / 山地 / 水域 / 林地 / 沙漠
- 天氣適性：晴 / 雨 / 霧 / 風 / 夜 / 雷

#### 視覺規則
- 採 S / A / B / C / D 星級或級別卡
- S 級顯示金色高亮
- D 級顯示警示色與排斥符號
- 可展開文字說明對應加成

### 5.8 Tab 6：延伸資訊

#### 需顯示欄位
- UID
- Template_ID
- Bloodline_ID
- 來源標籤
- 取得方式
- 成長摘要
- 戰績摘要
- 收藏狀態
- 英靈虎符摘要
- 血脈卡等級 / 血脈濃度
- 特種軍隊連動標籤
- 分享 / 比較 / 鎖定按鈕

## 6. 鎖定與遮蔽規則

### 6.1 鎖圖示規則

以下欄位預設使用鎖圖示 `🔒` 或 `?` 佔位：

1. 未解鎖因子
2. 未啟發戰法
3. 隱藏戰法
4. 未展開的 14 人血統矩陣
5. 天命值（若設定為隱藏）
6. 覺醒名號（尚未達成時）
7. 虛擬祖先內容（若玩家尚未解鎖細節）
8. 未公開的稱號來源

### 6.2 遮蔽層級

| 層級 | 意義 | 顯示方式 |
|---|---|---|
| 0 | 已公開 | 完整顯示 |
| 1 | 條件公開 | 灰階或半透明 |
| 2 | 可點擊解鎖 | `🔒 + ? + 提示` |
| 3 | 完全隱藏 | 僅顯示佔位框 |

## 8. 互動與展開規則

### 8.1 血統展開

1. 預設顯示父母兩張卡
2. 點擊「查看完整血統」後展開 14 人矩陣
3. 再次點擊可收合
4. 若資料不足，仍保留骨架位與鎖定標籤

### 8.2 戰法點擊

1. 點擊已習得戰法卡可進詳情
2. 若為可習得戰法，點擊顯示啟發來源
3. 若為不可習得戰法，點擊僅顯示鎖定原因

### 8.3 適性點擊

1. 點擊某項適性可看加成說明
2. 長按可顯示地形或天氣對應效果
3. 移動端預設使用 tooltip 底部抽屜，不要依賴 hover

### 8.4 立繪點擊：血脈面與故事抽屜

1. 點擊角色立繪後，先在人物上方短暫浮現 `命紋靈獸虛影`
2. 接著角色身上的祖紋命篆與符號 slot 依序點亮
3. 日常 premium layout 可直接顯示下方固定故事條；若版面較緊或手機版，再從下方滑出故事抽屜
4. 故事內容預設顯示兩個頁籤：
   - `歷史趣聞`
   - `血脈傳聞`
5. `歷史趣聞` 用來補史實逸事與人物記憶點；`血脈傳聞` 用來合理化其勇猛、執念、魅力或天命來源
6. 故事內容建議採 `5~6 格靜態連環漫畫 + 短文本`，不依賴角色動畫
7. 抽屜或固定故事條都不得遮住角色臉部、胸甲命紋與主要血脈符號 slot

## 9. 版面規格建議

### 9.1 桌面版

- 左：頭像 / 人物關鍵資訊
- 中：Tab 內容
- 右：摘要卡 / 快速操作 / 戰法快捷欄
- 建議比例：左側角色主視覺區約 44%~48%，右側內容卡約 36%~40%，最右頁籤欄維持 96~112 px 固定寬度，避免頁籤文字在 Web 與手機橫向模式下被壓扁。
- 摘要卡（HP / SP / 精力 / EP 等）優先靠右上排列，避免壓在立繪底部，維持角色主視覺完整度。
- 若啟用血脈故事抽屜，左側角色主視覺區底部需保留故事浮層空間，但不可讓人物主體寬度低於畫面寬度的 68% 可視佔比

### 9.2 手機版

- 上：頭像與摘要
- 中：Tab 切換
- 下：內容卡垂直排列
- 血統矩陣與戰法列表一律支援折疊
- 手機橫向模式優先沿用「左視覺、右資訊、右側 tab rail」配置，只縮窄左右留白與內容欄寬；直向模式才退化成上下堆疊。
- 頁籤欄文字不可依賴 hover 或超小字硬塞，應允許兩行標籤並保持按鈕高度不少於 84 px。

### 9.3 尺寸建議

1. 觸控目標不小於 44 × 44 px
2. 關鍵卡片最小高度建議 72 px
3. 血統矩陣卡片最小寬度建議 84 px
4. 戰法卡片需保留圖示、名稱、類型、等級四層資訊
5. 角色頁標題建議使用 42~48 px；頁籤文字 14~16 px；內文資訊 18~20 px；次要說明 16~18 px
6. 主要資訊卡內距建議 18~24 px；Tab 內容區左右留白建議 24~32 px；段落間距建議 12~18 px

### 11.2.3 分頁架構對 `日常人物頁 v3` 母型的影響評估

1. 分頁機制本身不衝突；衝突點在於若把 `GeneralDetailBloodlineV3` 硬塞進舊 `RightContentArea/TabBasics`，會直接壓縮 v3 需要的 `左人物 + 右大資訊卡 + 下方故事條` 比例。
2. 舊版 `general-detail-main.json` 的 `RightContentArea` 只有右側窄欄內容區，而 v3 是整體首頁殼層；因此 v3 應升級為 `GeneralDetail` 的外層首頁 layout，而不是 `TabBasics` 內的一張子卡。
3. `RightTabBar` 仍可保留為正式導航，但應視為外層 chrome；若保持 96~112 px 常駐寬度，需在 desktop 與 mobile 驗證是否擠壓 v3 資訊卡與故事條。
4. 對目前已收斂的 `日常人物頁` UI 結構來說，真正需要保護的是三件事：人物仍是第一主角、血脈只是深層人格區、故事條是首頁固定層；只要這三件事不被分頁 chrome 吃掉，分頁架構就可以共存。

### 11.2.4 其他 tab 的設計策略與文件維護原則

1. 其他 tab 預設不另開一張全新正式人物頁，而是沿用同一個 `GeneralDetail` 母型，在同一套外框下切換內容模組。
2. `Basics / Stats / Bloodline` 屬於與 overview 高度連動的內容，應優先做成同 family 的模組化延展，不必先重設一套新美術母板。
3. `Skills / Aptitude / Extended` 若只是資訊密度較高，可先以同母型下的次層卡片、抽屜或捲動模組處理；只有互動模型明顯改變時，才考慮補一張專屬子規格圖。
4. 後續若要排程開發，建議至少拆成三類任務卡：`overview 首頁整合`、`bloodline 深度內容整合`、`skills/aptitude/extended 模組化整理`。
5. 文件維護上，正式欄位與互動若有新增，優先回寫本規格書；若涉及整體視覺母型與 family 規則，再同步回寫 `docs/UI 規格書.md`，不可再只停在補遺。
6. 工程過渡期允許先以 `overview shell host + tab content host` 的雙層結構實作；其中 overview shell 可先獨立掛載、驗證 slot 與資料映射，再併回 `GeneralDetailPanel` 外層，不視為第二正式人物系統。
7. 過渡整合期允許先讓 `Basics` 分頁路由到 overview shell，而 `Stats / Bloodline / Skills / Aptitude / Extended` 繼續沿用既有 tab content host；待外層 host 收斂完成後，再決定 `Basics` 是否保留為次層詳情頁。

### 11.3 重構原則

1. 不直接把所有資料灌進單一長頁
2. 以分頁或折疊群組降低認知負擔
3. 讓資料載入與 UI 呈現分離
4. 鎖定欄位只做視覺表現，不在 View 中推導遊戲邏輯

### 11.2.5 `GeneralDetailBloodlineV3` 第一批 slot family / skin 收斂原則

1. `GeneralDetailBloodlineV3` 第一批收斂優先順序為：`slot family / skin contract -> skeleton node contract -> story / bloodline data contract`，不得把這三件事混成同一輪半做。
2. 視覺收斂第一步優先重用既有 `general_detail` family 資產，包含 `detail_content_bg`、`detail_summary_card_bg`、`detail_section_bg`、`detail_footer_bg` 等共用 frame / fill / rail，而不是繼續擴張 `color-rect` placeholder。
3. `日常人物頁 v3` overview 殼層至少必須先固定下列 slot family：
   - `portrait`：人物載體、未持有提示
   - `info.card`：右側外層資訊殼
   - `summary.card`：核心能力 / 角色定位 / 性格摘要
   - `bloodline`：血脈摘要主卡、命紋徽記承載區
   - `story.strip`：底部故事帶 rail 與 cell family
4. 在 `UI-2-0058` 完成前，不應直接大改 `GeneralDetailBloodlineV3` 節點結構；`UI-2-0059` 應承接已固定的 slot family，再做 node contract / runtime 收斂。
5. 在 `UI-2-0059` 完成前，不應把暫時文案視為正式內容來源；`UI-2-0061` 應承接正式 node / slot 結構後，再收斂故事帶、血脈摘要與趣聞欄位契約。
6. `overview shell` 的卡片文案、故事帶文字與血脈摘要節點，應直接宣告在 `layout JSON` 中；runtime 層只負責填值與切換狀態，不應再以臨時建立 `Label` 節點的方式長出最終畫面。

### 11.2.6 `UI-2-0059` skeleton 命名與進度條契約

1. `GeneralDetailBloodlineV3` 的 overview shell 命名應採用可重用的中性語意；`PortraitModeRibbon / PortraitModeHint / OverviewModeBadge` 屬於首頁狀態 chrome，不應再沿用暫時性的 `Unowned*` 命名。
2. 覺醒條應固定為 `AwakeningBarTrack + AwakeningBarFill` 的 node contract，並由 skin 提供 `track / fill` slot；不得再依賴未正式納管的 `progress` node type。
3. overview mode badge 與 portrait mode hint 的文字節點必須直接宣告在 layout JSON 中，後續若有持有 / 未持有 / 限定狀態，只能沿用同一組節點切換文案與外觀，不再新增平行節點。
4. `StoryStrip` 的 6 格節點命名應採用語意欄位，而非流水號；現階段固定為 `StoryCellOrigin / StoryCellFaction / StoryCellRole / StoryCellAwakening / StoryCellBloodline / StoryCellFuture`，供後續 `UI-2-0061` 直接對應資料欄位。
5. `general-detail-bloodline-v3-screen.json` 一旦視為正式 skeleton，即應收緊為 `allowMissingSkin: false`，避免 screen contract 長期停留在可缺皮膚的過渡狀態。
6. overview shell 的狀態層與進度條應以可複用群組存在；現階段固定為 `OverviewStateChrome` 與 `AwakeningProgressGroup`，作為後續 template / fragment 化的第一批骨架模組。
7. summary 區與 bloodline 區也應具備 template-ready 的群組邊界；現階段固定為 `OverviewSummaryModules` 與 `BloodlineOverviewModules`，避免後續 fragment 化時仍需從散落節點回推模組邊界。
8. `GeneralDetailOverviewShell.ts` 應以欄位表或等價資料結構批次填值 `OverviewSummaryModules` / `BloodlineOverviewModules`，避免各卡片 path 長期散落在方法本體。
9. `general-detail-bloodline-v3-main.json` 因上述 skeleton 群組化已升為 `version: 3`，`general-detail-bloodline-v3-default.json` 與 `general-detail-bloodline-v3-screen.json` 目前為 `version: 2`；後續若再改 node contract，須沿用版本遞增規則。

### 11.2.7 `UI-2-0061` story / bloodline / crest 內容契約

1. `GeneralDetailBloodlineV3` 首頁內容不得再長期依賴 `title / source / bloodlineId` 的 generic fallback 拼字；正式內容來源應收斂為 `historicalAnecdote`、`bloodlineRumor`、`storyStripCells`、`crestHint`、`crestState`。
2. `storyStripCells` 固定使用六格語意槽位：`origin / faction / role / awakening / bloodline / future`。後續 family 可重排視覺，但不得改動槽位語意。
3. `crestState` 固定為 `placeholder / rumored / revealed / awakened`。這是內容狀態，不是視覺 slot 名稱；視覺表現仍由 skin / crest family 決定。
4. `historicalAnecdote` 與 `bloodlineRumor` 屬於內容契約，優先由資料源提供；若缺值，runtime 才能暫以既有欄位 fallback，但 fallback 不可視為正式文案來源。
5. `crestHint` 應描述目前命紋 / 祖紋的解鎖程度與預留方向，用於人物頁、命鏡、虎符與特種軍隊之間的 crest family 串接。
6. 量產流程中，structure 應盡量固定在 template / skeleton；真正會隨角色大量變動的部分，應優先落在本內容契約，而不是重新修改 layout JSON。這是 `日常人物頁 v3` 往量產化前進的正式原則。

### 11.2.8 `GeneralDetailBloodlineV3` contentRequirements 與 preview capture

1. `assets/resources/ui-spec/screens/general-detail-bloodline-v3-screen.json` 現在必須正式宣告 `contentRequirements`，並對應 `assets/resources/ui-spec/contracts/general-detail-overview-content.schema.json`。

## 14. 總結

武將人物介面不是單一角色卡，而是玩家理解一名武將完整生命資料的入口。它應該同時承載：

1. 身份
2. 數值
3. 血統
4. 戰法
5. 適性
6. 隱藏與解鎖狀態

本規格的核心精神是：
- 已知就清楚顯示
- 未知就鎖起來
- 深度資訊可展開
- 正式版欄位先預留完整骨架
### 11.2.9 `GeneralDetailBloodlineV3` 第一輪品質收斂與 proof 規則
1. `PortraitImage` 這類 runtime 後載立繪節點，不得再因 `image slot` 沒有 spriteFrame 就回退成深色 placeholder。`UIPreviewNodeFactory.buildImage()` 需先正確處理 `color-rect / transparent` 類型，避免把正式立繪壓成灰黑。這條規則屬於通用 UI tooling 修正，不只適用人物頁。
2. `GeneralDetailBloodlineV3` 的主框體、背景與 stories rail 第一輪已轉向 `common parchment` family，目標是先清除舊黑金 chrome 混入造成的質感落差；後續若要再往正確參考圖逼近，應優先補 `jade-parchment header / crest glyph family / portrait stage art`，而不是反覆重切 layout。
3. `StoryStrip` 的正式方向確認為 `一張 master art 長條圖 + 六個 overlay slot`。資料契約仍保留 `origin / faction / role / awakening / bloodline / future` 六格語意，但視覺資產應一次生成長條敘事圖，不再拆成六張各自獨立的小圖。
4. `StoryStrip` 在 `GeneralDetailBloodlineV3` 的預設呈現不應再顯示每格標題字或 caption badge。overlay slot 的存在僅供語意定位、互動熱區或後續 metadata 使用；正式商業視覺應以純連續敘事 strip 為主。
5. canonical reference 可裁出 `story strip proof` 來驗證 family 結構與比例；此類 proof 資產僅作 layout / compare 用途，不可直接當正式商業素材結案。正式量產時，應以同構的 AI master art 或美術輸出替換 proof 資產。
6. 若角色故事帶尚未建立角色一致性錨定流程，`StoryStrip` 應優先採用弱臉化構圖：背影、側臉、剪影、遠景、手勢、武器、旗幟、命紋碑等元素優先，避免正臉特寫直接暴露與專案內正式角色立繪不一致的風險。
7. `StoryStrip` 不應再只靠 prompt 疊加文字來驅動 AI。正式量產流程應增加 `ArtRecipe + asset direction + compare input` 三層：先定義 family 載體與安全區，再定義 AI 生圖 / 裁切 recipe，最後固定 compare board 輸入，避免每輪 review 重新口頭解釋需求。
8. 2026-04-05 已完成第一份 `story-strip-horizontal-scroll-r1.art-recipe.json` 與 `story-strip-horizontal-scroll-pilot-v1.md`。這證明故事帶量產應視為獨立 family，而不是附屬於單一畫面的臨時素材。
### 11.2.10 `GeneralDetailBloodlineV3` jade-parchment 與 crest medallion 過渡規則
1. `GeneralDetail` 進入 `overview shell` 時，右側 `RightTabBar` 不應再沿用過重的舊黑金 rail。過渡期應至少切到較輕的 parchment/jade chrome，避免 `overview shell` 與 `tab chrome` 視覺世代不一致。
2. `GeneralDetailBloodlineV3` 的 header 正式定義為 `bleed + band + frame` 三層結構，不可只剩單一色帶。色彩語言以 `jade-parchment` 為準，並允許用 rarity accent 去驅動 header band / badge / progress / crest glow。
3. `crest` 在正式美術 glyph 到位前，應先維持 `circular medallion skeleton`，包含 `glow / fill / crest / frame / glyph label / state label`。proof 階段可用圓形 family sprite 替代，但不得誤用與商業語意不相干的 badge 圖示充當最終命紋。
