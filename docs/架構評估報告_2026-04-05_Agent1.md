## 詳細評估

### A1 + A2 — UIManager / UIConfig 的結構性斷裂（P0）

**問題核心：**

`UIConfig.ts` 的 `UIID` enum 只有 6 個入口：

```typescript
export enum UIID {
    BattleHUD, DeployPanel, BattleLogPanel,
    ResultPopup, Toast, SystemAlert
}
```

然而 `assets/resources/ui-spec/screens/` 已有完整 JSON 契約的畫面超過 22 個（LobbyMain、ShopMain、Gacha、GeneralDetail、GeneralList、GeneralPortrait、GeneralQuickView、DuelChallenge、UnitInfoPanel、ActionCommandPanel 等）。

這代表：
- 這 16+ 個畫面**完全繞過** UIManager 的 6-Layer 系統（Game/UI/PopUp/Dialog/System/Notify）
- 沒有層級管理 → 沒有遮罩控制、沒有堆疊行為、沒有統一的 open/close/cache 語義
- 每個畫面各自決定怎麼顯示、怎麼關閉，缺乏一致的 lifecycle

**根本缺口：**

`UIManager.open()` 完全依賴 `register()` 預先掛載場景節點，沒有動態 Prefab 載入路徑：

```typescript
// UIManager.ts 現況 — 沒有動態載入
open(uiId: UIID): void {
    const entry = this._registry.get(uiId);
    // entry 必須已由 register() 預先綁定場景節點
    // 沒有 ResourceManager.loadPrefab() 呼叫
}
```

對照 Unity 的等效架構：UI Manager 對應 Unity 的 `UIManager` + `Addressables.LoadAssetAsync<GameObject>`，而目前的實作只有前半，完全缺少後半。

**影響：**
- Demo 階段可運作（所有 UI 都在 BattleScene 預先擺好），但一旦需要跨場景 UI 或按需載入 Popup，整個系統就必須重建
- UIConfig 的 `prefab?` 和 `bundle?` 欄位已預留（有 M-2 TODO 標記），代表設計者已知，但實作零進展

**建議方向（UI-2-0074）：**
1. 擴充 `UIID` enum，補全 16+ 個 Lobby/非戰鬥畫面
2. 在 `UIManager.open()` 加入 `ResourceManager.loadPrefab(entry.prefab, entry.bundle)` 路徑
3. 先做「有 prefab 路徑則動態載入，否則沿用 register() 舊路徑」的向後相容設計

### A3 — UISpecLoader 無共享實例（P1）

**問題核心：**

grep 結果顯示 **15 個**元件各自建立獨立的 `new UISpecLoader()`：

```
GeneralDetailPanel.ts, BattleLogPanel.ts, DuelChallengePanel.ts,
ActionCommandPanel.ts, BattleHUD.ts, ResultPopup.ts, UnitInfoPanel.ts,
GeneralQuickViewPanel.ts, GeneralListPanel.ts, GeneralPortraitPanel.ts,
TigerTallyPanel.ts, ToastMessage.ts, StyleCheckPanel.ts,
UltimateSelectPopup.ts, UIScreenPreviewHost.ts
```

每個實例都有自己的 `_cache: Map<string, T>`，代表同一份 JSON 可能被讀入記憶體 15 次。UISpecLoader 目前沒有跨實例共享快取，也無法透過 ServiceLoader 取得共享單例。

對照 Unity：等同於每個 MonoBehaviour 都生一個獨立的 `Resources.Load()` 快取，而不是走 Addressables 的共享 cache。

**建議方向（UI-2-0075）：**
1. 在 `ServiceLoader` 中加入 `specLoader: UISpecLoader` 共享入口
2. `UISpecLoader` 改為可被注入（建構子接受 `ResourceManager`），或直接從 `services()` 取得
3. 各元件改為 `services().specLoader` 而非 `new UISpecLoader()`

### A4 — Binder 遷移不完全（P1）

**問題核心：**

`UITemplateBinder` 已有完整的 `bind(rootNode, spec)` → `getLabel/getButton/getNode/setTexts()` API（`DuelChallengePanel.ts` 是正確使用範例），但 11 個元件仍使用脆弱的舊式操作：

| 元件 | 問題模式 |
|------|---------|
| `BattleHUD.ts` | 15+ 個私有節點引用（`_turnLabel`, `_foodLabel`...），在 `onBuildComplete` 手動填充 |
| `GeneralDetailPanel.ts` | `_bindClick('RightTabBar/BtnClose', ...)` 路徑字串導航、手動 Color 操控 |
| `GeneralListPanel.ts` | `populateList()` 直接 `.string` 賦值，`getChildByPath` |
| `GeneralPortraitPanel.ts` | 手動節點查找 |
| `GeneralQuickViewPanel.ts` | 手動節點查找 |
| `ActionCommandPanel.ts` | 手動節點查找 |
| `StyleCheckPanel.ts` | 手動節點查找 |
| `ToastMessage.ts` | `getChildByName` |
| `BattleLogPanel.ts` | 手動 Label 設置 |
| `ResultPopup.ts` | 部份手動 |
| `UltimateSelectPopup.ts` | 使用 Binder 但 `onReady()` pattern 未完整 |

手動路徑字串（如 `'RightTabBar/BtnClose'`）在 JSON 結構調整後不會報編譯錯誤，只會在 runtime 靜默失敗。這是目前最大的維護性脆弱點之一。

**建議方向（UI-2-0076）：**
- 批量遷移，以 `DuelChallengePanel.ts` 為模板
- 優先處理高互動元件：`BattleHUD`、`GeneralDetailPanel`

---

### A5 — i18n 語系 hardcoding（P1）

**問題核心：**

14 個元件都以字串常數直接呼叫：

```typescript
const i18n = await this._specLoader.loadI18n('zh-TW');
```

`I18nSystem` 服務已存在於 `ServiceLoader` 中，但它的 `currentLocale` 沒有被任何元件查詢。若要在未來支援繁中/簡中/英文切換，需要逐一改 14 個檔案。

**建議方向（UI-2-0077）：**
1. `I18nSystem` 加入 `get currentLocale(): string` 屬性
2. 各元件改為 `this._specLoader.loadI18n(services().i18n.currentLocale)`
3. 加入 locale 變更廣播訊號，讓已開啟的 UI 自動重新載入文字

---

### A6 — MemoryManager 是空殼（P2）

**問題核心：**

`MemoryManager.ts` 的程式碼有明確的自我說明：

```typescript
// 追蹤層空殼，預留完整擴充點
// onThresholdExceeded = null;
// onAssetFullyReleased = null;
// releaseByScope() — 預留接口
```

目前所有資產都靠 `ResourceManager` 的 `Map<string, T>` 快取，**沒有** LRU 淘汰、**沒有** `releaseByScope()`、**沒有**記憶體壓力偵測。在 Demo 規模下可接受，但一旦 22 個畫面都被開過，記憶體會持續增長。

對照 Unity：等同於 `Resources.Load()` 沒有任何 `Resources.UnloadAsset()` 呼叫。

**建議方向（UI-2-0078）：**
1. 實作基礎 LRU（容量 N 個 SpriteFrame / JSON，預設值可先設 50）
2. 實作 `releaseByScope(sceneId)` 讓場景切換能釋放資產
3. 接通 `ResourceManager.loadPrefab/loadJson` 的 reference count

---

### A7 — Constants.ts `UILayerName` deprecated 殘留（P3）

**問題核心：**

`Constants.ts` 仍有 `@deprecated M-1 重構後改用 LayerType` 的 `UILayerName` enum，與 `UIConfig.ts` 的 `LayerType` 並存，對新加入 Agent 造成語義混淆。

**建議方向（UI-2-0079）：**
- 確認無任何呼叫端使用 `UILayerName`
- 直接移除該 enum，只留 `LayerType`

## 結論

目前專案的 UI Template + Widget 架構本身相當完整（UITemplateResolver、UITemplateBinder、UISpecTypes、4 個 Template、11 個 Widget Fragment 全部到位），Phase P0/A/B/C 的設計質量良好。

**最根本的架構問題是 UIManager 與 22 個畫面之間的斷層**——整個 Lobby/非戰鬥 UI 系統目前沒有被任何層級管理框架覆蓋，這是後續版本必須優先解決的結構性缺陷。

其次是 15 個獨立 UISpecLoader 實例、11 個尚未遷移的 Binder 元件、14 個硬編碼語系呼叫，這三點共同構成「維護性脆弱帶」，在 Demo 規模內可接受，但在量產展開後會成為持續的技術債。
