# 美術素材規劃與使用說明

> **文件定位**：本文件是本專案所有美術資源目錄、命名規則、治理流程與 legacy 例外的**唯一正式規範**。
> **適用範圍**：`assets/resources/`、`assets/bundles/`、`assets/units/`、`assets/scenes/` 與所有引用這些資產的文件與程式。
> **優先順序**：若其他文件對美術資源路徑或命名的描述與本文件衝突，**以本文件為準**。
> **最後更新**：2026-03-31

---

## 1. 文件目的

本文件要解決三件事：

1. 讓美術、程式、企劃看到資料夾就知道用途，不再靠口耳相傳。
2. 統一全專案命名規則，避免不同系統各自發明前綴與資料夾結構。
3. 把目前仍存在的 legacy 路徑明確標記出來，避免新資源繼續往錯的地方堆。

Unity 對照：這份文件的角色，等同於把 Unity 專案裡的 `Assets/Art`、`Addressables Group`、`UI Atlas 命名規範`、`VFX 資源庫 SOP` 全部收斂成一份可執行的總章程。

## 3. 目錄定義與用途

### 3.1 `assets/resources/` 正式 runtime 資產

| 路徑 | 目前用途 | 允許檔案 | 載入方式 | 命名規則 | 備註 |
|---|---|---|---|---|---|
| `assets/resources/icons/` | 遊戲邏輯直接用的兵種或小型圖示 | `.png` | `resources.load('icons/...')` | `{gameplay_id}.png` | 目前 `DeployPanel.ts` 以兵種 key 直接載入，不能隨意加前綴 |
| `assets/resources/textures/` | 背景、棋盤底圖、matcap、白模貼圖 | `.png` | `resources.load('textures/...')` | 背景用 `bg_{theme}_{variant}.png`；一般貼圖用 `tex_{domain}_{desc}.png`；工具白模可保留 `white.png` | `grid-bg.png`、`white.png` 屬既有例外，不作為新命名模板 |
| `assets/resources/sprites/` | 2D 角色立繪、頭像、序列幀、SpriteFrame 來源 | `.png` | `resources.load('sprites/.../spriteFrame')` 或目錄掃描 | 單張圖用 `{entity_id}_{usage}.png`；序列幀用 `{entity_id}_{action}_{NN}.png` | 目前有 `sprites/generals/zhang_fei_portrait.png`；另有 `sprites/ui_families/`（見下方） |

| `assets/resources/effects/` | 自訂 Shader Effect 正式路徑 | `.effect` | `resources.load('effects/...')` | **新檔一律** `{domain}_{feature}.effect` | 既有 `board-jelly.effect`、`vfx-buff-quad.effect` 為 legacy 檔名，保留不再擴散 |

| `assets/resources/ui-spec/skins/` | UI 視覺 skin JSON | `.json` | `UISpecLoader` | `{screen_id}-default.json` | 樣式層 |
| `assets/resources/ui-spec/screens/` | UI 組裝契約 JSON | `.json` | `UISpecLoader` | `{screen_id}-screen.json` | screen 層 |
| `assets/resources/ui-spec/ui-design-tokens.json` | 全域 UI design tokens | `.json` | `UISpecLoader` | 固定檔名 | 不可另建第二份 tokens |
| `assets/resources/units/troops/<troop-id>/` | 小兵執行期模型與 Prefab | `.glb`、`.prefab` | `UnitAssetCatalog` / `ResourceManager` | 固定檔名 `unit.glb`、`unit.prefab` | `troop-id` 必須與 `troops.json` key 一致 |
| `assets/resources/units/heroes/<hero-id>/` | **正式標準**的英雄執行期模型與 Prefab | `.glb`、`.prefab` | `UnitAssetCatalog` / `ResourceManager` | 固定檔名 `hero.glb`、`hero.prefab` | 目前專案尚未完全遷到此結構 |
| `assets/resources/fonts/` | 新版正式字型治理目錄 | `.ttf`、`.otf`、`.fnt`、`.png` | `I18nSystem` / `FloatTextSystem` | 見 `assets/resources/fonts/_GOVERNANCE.md` | 新字型一律優先放這裡 |
| `assets/resources/dmgFont/bmfont/` | 舊版 BMFont 直接載入路徑 | `.fnt`、`.png` | `BattleScene.ts` | 舊名保留 | **legacy 凍結**，不可再新增新體系字型 |
| `assets/resources/effect/` | 舊版材質/測試資產 | `.mtl` 等 | 零星引用 | 舊名保留 | **legacy 凍結**，不可當正式材質目錄 |

## 4. 全專案統一命名規則

### 4.1 全域規則

1. **新檔名一律使用 ASCII 小寫**，單字以 `_` 分隔。
2. **資料夾一律使用小寫 snake_case**，禁止新建含空白、連字號混用或 CamelCase 的資料夾。
3. **名稱必須可推回用途**，禁止 `new`, `final`, `test2`, `copy`, `版本2` 這類不可維護命名。
4. **檔名若需要和程式 key 直接對應，優先跟 key 同名**，不要額外加前綴。
5. **副檔名前的主名只表達資產語意，不重複目錄資訊**。例如放在 `textures/glow/` 內仍寫 `tex_glow_soft.png`，是為了跨目錄搜尋時仍可辨識。

### 4.2 各資產族群命名表

| 資產類型 | 正式規則 | 範例 | 備註 |
|---|---|---|---|
| 共用貼圖 Texture | `tex_{domain}_{desc}.png` | `tex_glow_soft.png` | 適用 VFX、matcap、共用視覺貼圖 |
| 背景圖 | `bg_{theme}_{variant}.png` | `bg_normal_day.png` | 背景層專用 |
| 遊戲圖標 | `{gameplay_id}.png` | `archer.png` | 適用 `resources/icons/` 這種 config key 直載目錄 |
| 角色立繪 / 頭像 | `{entity_id}_{usage}.png` | `zhang_fei_portrait.png` | `usage` 例如 `portrait`, `card`, `half_body` |
| 序列幀 | `{entity_id}_{action}_{NN}.png` | `zhang_fei_idle_01.png` | `NN` 固定兩位數 |
| VFX Prefab | `fx_{system}_{variant}.prefab` 或 `{effect_key}.prefab` | `fx_buff_gain_3d.prefab` | Bundle 模組可用 effect key 同名 |
| 材質 Material | `mat_{domain}_{desc}.mtl` | `mat_unit_default.mtl` | 新材質標準；legacy `.mtl` 保留但不沿用 |
| Mesh | `mesh_{desc}.fbx` | `mesh_halfcircle.fbx` | 新檔用小寫；現有大寫副檔名可保留 |
| 動畫 | `anim_{desc}.anim` | `anim_revival.anim` | |
| Shader Effect | `{domain}_{feature}.effect` | `unit_base.effect` | 新檔一律 snake_case |
| UI layout spec | `{screen_id}-main.json` | `general-detail-main.json` | UI-spec 三層契約沿用既有後綴規則 |
| UI skin spec | `{screen_id}-default.json` | `battle-hud-default.json` | |
| UI screen spec | `{screen_id}-screen.json` | `general-list-screen.json` | |
| 小兵單位資產 | `unit.glb` / `unit.prefab` | `assets/resources/units/troops/archer/unit.glb` | 由資料夾名承接單位 id |
| 英雄單位資產 | `hero.glb` / `hero.prefab` | `assets/resources/units/heroes/guanyu/hero.glb` | 正式標準 |
| BMFont 目錄 | `{font_role}/` | `dmg_crit/` | 新字型跟 `fonts/_GOVERNANCE.md` 對齊 |
| BMFont 檔名 | `{font_role}.fnt` + `{font_role}_0.png` | `dmg_crit.fnt` | `.fnt` 與 atlas PNG 必須同名配對 |
| Locale 字型 | `{role}.ttf` | `body.ttf` | `role` 僅允許 `body`、`title` |

### 4.3 UI 資產命名補充

1. **UI 版位契約**：畫面 ID 採 kebab-case，例如 `battle-hud`、`general-detail`。
2. **UI 圖塊 path**：若走 UI-spec / skin manifest，SpriteFrame 路徑應使用 `sprites/<system>/<asset_key>/spriteFrame` 的語意結構。
3. **UI icon 實體檔名**：若是直接資源檔，建議用 `icon_{system}_{desc}.png`；若當前程式以 gameplay key 直載，則維持 key 同名。
4. **節點命名**：節點名稱遵守各畫面接線文件，例如武將詳情畫面以 `武將人物介面美術接線清單.md` 為準。
5. **UI 母框來源**：若存在已核可的通用框原稿，可先放在 `assets/resources/ui-spec/placeholders/` 作為母框來源，再派生輸出到 `assets/resources/sprites/common/`、`assets/resources/sprites/battle/` 等正式 runtime 路徑；母框本身不是最終綁定路徑，衍生 variant 才是正式畫面資產。
6. **目前落地範例**：`assets/resources/ui-spec/placeholders/bg_ink_detail.png` 已作為 panel family 母框，衍生覆蓋到 `sprites/common/popup_card_bg` 與 `sprites/battle/unitinfo_*` 等框體資產。
7. **分層框體命名（已落地）**：`sprites/ui_families/` 對應展間的分層 UI 工作流。同一家族的資產後綴統一為 `_frame`（主框）、`_bleed`（漏光/暖暈）、`_fill`（底色填充）、`_accent`（點綴直去、金屬尊個）、`_bg`（完整單層安全層）。例如 `sprites/ui_families/general_detail/detail_header_frame.png`、`detail_header_bleed.png`。目前已部署：`general_detail/`（武將詳情，指定分層框體 60+ 張）、`common/equipment/`、`common/commerce/`（共用按鈕三態圖）。
8. **family 命名原則**：family 名稱應反映功能而非單一畫面，例如 `section_card`、`tab_plate`、`drawer_root`、`action_plate`，避免把同類框體永遠綁死在某一個畫面專案名下。

### 4.4 參考圖驅動的按鈕 family 工作流

1. **參考圖中的按鈕不能只看顏色，要看「材質 + 層級 + 語意」**：成熟專案通常至少同時存在 `commerce CTA`、`equipment / progression`、`nav.ink`、`paper.utility`、`warning.destructive` 這幾類 recipe；它們的差異不是美術隨興，而是拿來區分購買、養成、導覽、工具、危險操作等不同互動層次。
2. **本專案已正式落地的 common button family 只有兩種**：`common/commerce/` 與 `common/equipment/`。其他 family 若要新增，必須先走參考圖分析、preview 驗證與 keep 核可，不得先開目錄再補規則。
3. **建議保留的預備命名**：
	- `assets/resources/sprites/ui_families/common/nav_ink/btn_nav_{state}.png`
	- `assets/resources/sprites/ui_families/common/paper_utility/btn_utility_{state}.png`
	- `assets/resources/sprites/ui_families/common/warning/btn_warning_{state}.png`
	以上目前屬於保留名，不代表已正式落地。
4. **family 提升條件**：只有在同一套按鈕 recipe 能跨 3 個以上畫面、且可共享 `normal / pressed / disabled / selected` 九宮格與材質語言時，才應升級為 `common/*` family。若只在單一畫面出現，應保留為 screen 專用資產或 screen 專用 skin slot。
5. **參考圖分析的標準步驟**：
	1. 建立整張畫面的按鈕盤點，不可只裁最漂亮的主按鈕。
	2. 為每顆按鈕標註 `語意 / 層級 / 材質 / 狀態 / 是否重複出現`。
	3. 先做分群，再決定哪些群值得抽成 family。
	4. 以 `artifacts/ui-layered-frames/` 或 `artifacts/ui-source/{screen_id}/` 產出 preview board，驗證同 family 在不同畫面是否仍成立。
	5. 通過後才建立 `sprites/ui_families/common/{family}/` 正式 runtime 目錄、更新 keep 與對應 skin manifest。
6. **此工作流要學的不是「全部按鈕做成一樣」，而是有規則地保留材質差**：全域導覽可偏墨黑，內容承載可偏宣紙，CTA 可偏亮金，養成可偏暖金墨染，危險操作可偏紅棕。這種刻意的材質差，才會形成較自然的手繪感與歷史感。

## 5. Legacy 目錄與例外清單

以下路徑**目前存在且可能仍被程式引用**，但已不再是新資產的正式落點。

| 路徑 / 類型 | 現況 | 規則 |
|---|---|---|
| `assets/resources/dmgFont/bmfont/` | `BattleScene.ts` 仍直接載入 `bao`、`jia`、`miss`、`pu` | 維持相容，不新增新字型；新字型一律走 `assets/resources/fonts/` |
| `assets/resources/effect/` | 只有 `colorBar/`、`star/` 等舊材質資產 | 凍結，不新增，不作為正式材質目錄 |
| `assets/resources/units/troops/heroes/` | 目前英雄 GLB / Prefab 被放在小兵路徑底下 | 視為 legacy；**新英雄資產一律放 `assets/resources/units/heroes/`** |
| `assets/units/` | 原始來源或中繼輸出 | 不可由 runtime 直接依賴 |
| `assets/bundles/vfx_core/prefabs/` 中含空白 / CamelCase / 匯入原名的資料夾 | 例如 `Star aura/`、`Love aura/`、`Buff/` | 保留現狀；新 Prefab 一律用 snake_case 命名與資料夾 |
| `board-jelly.effect`、`vfx-buff-quad.effect` 等 hyphen 檔名 | 已有程式路徑依賴 | 保留；新 `.effect` 不再沿用 hyphen 命名 |

### 5.1 遷移判準

1. **有程式直接載入路徑者，先保相容，再遷移。**
2. **legacy 路徑只允許修復，不允許擴張。**
3. **正式新規則只對新檔 / 新資料夾生效；若要清理舊路徑，需連同程式與文件一起改。**

## 7. 實作與提交流程

### 7.1 新增美術資產前必做

1. 先判斷它屬於 `resources`、`bundles`、還是 `assets/units` 原始來源層。
2. 確認是否已有對應分類，不可為了單一資產再發明新的頂層資料夾。
3. 依本文件命名，不可直接沿用外部專案原名。
4. 若屬於 VFX / UI / 字型，先確認是否還有 manifest 或治理文件要同步。

### 7.2 新增後必做

1. 確認 Cocos 匯入正常，`.meta` 已建立。
2. 確認程式引用路徑與檔名一致。
3. 若有新增新的目錄、前綴或命名族群，必須回寫本文件與 `keep.md`。
4. 若變更正式路徑，必須同步更新交叉索引與所有相關規格書，不得只改一處。

## 8. 當前專案的關鍵現況摘要

1. **VFX Core 的貼圖命名已相對成熟**，核心規則是 `tex_{category}_{desc}.png`。
2. **UI-spec 三層 JSON 已成形**，畫面命名以 `{screen_id}-main/default/screen.json` 為主。
3. **字型治理已建立新版 `assets/resources/fonts/` 規範**，但戰場飄字仍有 `dmgFont/bmfont` legacy 路徑尚未完全遷移。
4. **單位模型正式路徑尚未完全收斂**，目前存在 `assets/resources/units/...` 與 `assets/units/...` 兩層，且英雄資產還有 `troops/heroes/` 的 legacy 位置。
5. **VFX Prefab 區存在不少外部匯入原名**，新檔禁止再複製這種命名風格。
6. **`sprites/ui_families/` 分層框體已正式落地**：`general_detail/` 子目錄已含 60+ 張資產（header/summary/tabbar-rail/tab/content/section/footer/tree-node/action/field 五層全覆蓋）；`common/equipment/` 與 `common/commerce/` 已含 shared 按鈕三態圖。**其他畫面（general-list、battle UI 等）目前未具備分層框體資產，視覺一致性對齊為待追事項**。
7. **按鈕 family 工作流已升級為參考圖驅動模式**：未來新增 common button family 前，必須先完成截圖盤點、分群、preview board、keep 核可，再建立正式 runtime 目錄；不再接受「先做一張按鈕，再事後幫它找語意」的流程。

## 9. 結論

從這一版開始：

1. **本文件就是全專案唯一的美術資源總規範。**
2. **新的資料夾與檔名必須先對齊這份文件，再進專案。**
3. **遇到 legacy 路徑時，先標註、再遷移，不准繼續擴張。**

命名一旦不統一，後面不只是找資產困難，連 `ResourceManager`、`UnitAssetCatalog`、`UI skin manifest`、`VFX registry` 都會一起失控。這份規範的目的就是把這件事在制度層先鎖住。
