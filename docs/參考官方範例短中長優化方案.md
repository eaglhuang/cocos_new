## 一、2D vs 3D 粒子使用判斷標準

> 本節分析適合本專案（策略棋盤式戰鬥 + Cocos Creator 3D 引擎）的粒子系統選型原則。

### 1-1 核心判斷心法

```
特效在哪個空間？
├── UI Canvas 上（Screen Space）→ 2D 粒子
└── 遊戲世界中（World Space）→ 3D 粒子
    ├── 是否貼地/有地面感？      → 3D 粒子（必然）
    ├── 是否需要透視縮放？        → 3D 粒子（必然）
    ├── 是否跟隨單位 3D 位置？    → 3D 粒子（建議）
    └── 視覺效果只需面向攝影機？  → 3D Billboard 粒子（也是 3D）
```

> **與 Unity 對照**：Unity 中也有相同原則——UI Particle System 掛在 Canvas 下，世界特效掛在 World Space。Cocos 2D 粒子 ≈ Unity 的 Legacy Particle System；3D 粒子 ≈ Unity 的 Shuriken Particle System。

### 1-2 判斷標準表

| 特效類型 | 範例 | 使用 | 理由 |
|---------|------|------|------|
| **血量/傷害飄字** | `-156 傷害`、`暈眩！` | ❌ 不用粒子 | 由 `FloatTextSystem` + BMFont 處理，效能更佳 |
| **UI 裝飾光效** | 技能按鈕光暈、卡牌閃光 | **2D** | 純 Screen Space，不需深度 |
| **結算/勝利動畫** | 勝利彩帶、金幣散落 | **2D** | 覆蓋在 UI 層最頂端，Canvas 空間 |
| **選單/Loading 光粒** | 標題背景飄雪/星光 | **2D** | UI Layer，效能低消耗 |
| **單位受擊閃光** | 被攻擊時白光閃爍 | **3D** | 需貼合單位 3D 座標與材質 |
| **攻擊/技能拖尾** | 刀光、箭矢軌跡 | **3D** | 需要 TrailModule，需透視感 |
| **爆炸/衝擊波** | 單位死亡爆炸、技能衝擊 | **3D** | 需要立體擴散與地面反射感 |
| **地形煙塵** | 踩踏揚塵、落地煙霧 | **3D** | 必須貼地，需要 Y 軸方向性 |
| **環境氣氛** | 戰場火焰、硝煙 | **3D** | 需在世界空間與地形互動 |
| **Buff 狀態光環** | 護盾光圈、中毒綠霧 | **3D Billboard** | 懸浮於單位頭頂，需隨世界位置移動 |
| **武將技能演出** | 大範圍施法、元素爆發 | **3D** | 全場演出需要 3D 景深感 |
| **召喚/消散溶解** | 單位召喚出現 / 死亡消散 | **3D** | 配合 MaterialSystem 的 Dissolve Shader |

### 1-3 GPU vs CPU 渲染選擇

| 情境 | 使用 | 注意 |
|------|------|------|
| 需要 Trail（刀光、拖尾） | **CPU 渲染** | GPU 不支援 TrailModule |
| 需要 LimitVelocity（自然減速）| **CPU 渲染** | GPU 不支援此模組 |
| 大量粒子（50+ 個同時）| **GPU 渲染** | 效能優勢顯著，Material 改 `builtin-particle-gpu` |
| 戰場環境粒子（背景氛圍）| **GPU 渲染** | 量大、無需 Trail，GPU 最合適 |
| 技能演出（精準控制）| **CPU 渲染** | 需要程式動態修改參數 |

### 1-4 本專案推薦配置

```
戰鬥特效（World Space）→ 全部 3D 粒子
├── 技能演出：CPU 渲染（需 Trail / 程式控制）
├── 環境氛圍：GPU 渲染（大量粒子，靜態設定）
└── Buff 光環：3D Billboard CPU 渲染（需動態開關）

UI 特效（Screen Space）→ 全部 2D 粒子
├── 勝利/結算彩帶：2D 粒子（Canvas 最上層）
└── 按鈕光效：2D 粒子（UI 元件子節點）

文字演出 → 不用粒子，FloatTextSystem（BMFont）
```

## 一-B、Cyberpunk 優化項目全局可行性評估

> 根據可優化說明書 2 與美術素材規劃文件，對照本專案「2.5D 策略棋盤」的定位，逐一評估每個 Cyberpunk 優化項目的適用性。

### 架構差異前提

| 面向 | Cyberpunk | 本專案（三國 Demo） | 影響 |
|------|-----------|---------------------|------|
| **遊戲類型** | 3D 第三人稱射擊 | 2.5D 回合制棋盤策略 | 移動、AI、IK 需求完全不同 |
| **單位移動** | 自由移動 + 物理碰撞 | 格子到格子 tween | Steering Behaviors 過度設計 |
| **攻擊方式** | 即時射擊 + 瞄準 | 回合制自動戰鬥 | IK 瞄準系統不適用 |
| **場景規模** | 大型連續場景 | 5×5 棋盤 | A* 尋路需求極低 |
| **視角** | 自由攝影機 | 固定俯視 | Ghost 殘影效果可見度有限 |

### 項目評估結果

| 原始優先級 | 項目 | 評估結果 | 調整後 | 理由 |
|-----------|------|---------|--------|------|
| 🔴 P0 | Steering Behaviors | **降級** | 🟢 L-tier（備用） | 棋盤遊戲單位沿格子移動，tween + easing 已足夠。Steering 適合自由移動場景，非格子策略。但 `arrive()` 減速曲線概念可在中期用於強化 tween easing。 |
| 🟡 P1 | 動態音軌混合 | **納入** | 🟡 M-tier | AudioSystem 目前僅單軌 BGM，加入 crossfade/multi-track 對戰鬥沉浸感提升顯著。實作成本低（增加第二 AudioSource + lerp）。 |
| 🟡 P1 | Data-Driven Action | **納入** | 🟡 M-tier | 高度契合「資料驅動」架構原則。技能由 JSON 配置符合 vibe coding 理念。ServiceLoader 擴充零阻力。 |
| 🟢 P2 | Ghost 殘影 | **降級** | 🟢 L-tier | Cyberpunk 的 `FxGhostVertex` 針對 3D SkinnedMesh 設計，本專案 2.5D Sprite 需另寫適配 Shader。且固定俯視角殘影視覺效果有限。 |
| 🟢 P2 | A* 尋路 | **降級** | 🟢 L-tier（備用） | 5×5 棋盤 + 直線攻擊，A* 的價值在有障礙物/複雜地形時才浮現。目前遊戲規模不需要。 |
| 🟢 P2 | 事件匯流排強化 | **提升** | 🔴 S-tier | 極小改動（`onBind` + `once` 方法），防止常見 `this` 丟失 bug。EventSystem 目前僅 3 個方法，擴充成本極低。 |
| 🟡 P1 | VFX 表 JSON 化 | **納入** | 🟡 M-tier | `vfx-usage-table.ts` 目前是 TS 常量，改為 JSON 載入後設計師可免編譯配置特效組合。 |
| 🟡 P1 | 特效-音效-通知三位一體 | **納入** | 🟡 M-tier | 防止視/聽/邏輯不一致，與 M-3 EffectSystem 整合一起做。 |
| 🟢 P2 | Core/Logic 分層純化 | **納入** | 🟢 L-tier | 架構衛生，長期可移植性。需檢視並清理 `scripts/core/` 中的遊戲特化邏輯。 |
| — | Cyberpunk 素材遷移 | **納入** | 🔴 S-tier | 18 個風格中性素材已完成規劃（見美術素材規劃 2），依 VFX Core SOP 搬入。 |

## 二、短期優化（1~2 週）

> **目標**：補強現有系統的缺口，成本低、效益即時可見。

### S-1 VFX Block Registry 加入 GPU 相容標記

**問題**：目前 `vfx-block-registry.ts` 中的 40 個 Block 沒有標記渲染模式，未來切換 GPU 渲染可能導致 Trail 或 LimitVelocity 效果靜默失效。

**參考來源**：官方文件 — Renderer Module 的 CPU vs GPU 差異

**解法**：
```typescript
// vfx-block-registry.ts 擴充欄位
interface VfxBlockEntry {
    id: string;
    category: string;
    prefab: string;
    // === 新增 ===
    renderMode: 'cpu' | 'gpu' | 'auto';  // auto = 無特殊模組需求時自動選
    gpuIncompatibleModules?: string[];    // 例：['trail', 'limitVelocity']
    // ===========
    audioEvent?: string;
}
```

**驗收條件**：所有使用 TrailModule 或 LimitVelocityOvertimeModule 的 Block 都標記 `renderMode: 'cpu'`。

---

### S-2 EffectSystem 加入 FxGroup 批次粒子管理

**問題**：目前 `EffectSystem.playEffect()` 只控制單一節點，無法統一控制一個特效 Prefab 內的多個 `ParticleSystem`（例如技能特效由 5 個子粒子組成）。

**參考來源**：cocos-example-cyberpunk 的 `fx-group.ts` 設計

> **Unity 對照**：類似 `GetComponentsInChildren<ParticleSystem>()` 批次 Play/Stop，Cyberpunk 將此封裝成 FxGroup class。

**解法**：
```typescript
// EffectSystem 新增方法
playGroup(node: Node): void {
    const systems = node.getComponentsInChildren(ParticleSystem);
    systems.forEach(ps => ps.play());
}
stopGroup(node: Node, clear = false): void {
    const systems = node.getComponentsInChildren(ParticleSystem);
    systems.forEach(ps => clear ? ps.clear() : ps.stop());
}
setGroupLoop(node: Node, loop: boolean): void {
    node.getComponentsInChildren(ParticleSystem).forEach(ps => ps.loop = loop);
}
```

**驗收條件**：武將技能演出使用 `playGroup` 播放，所有子系統同步啟動。

---

### S-3 粒子動態覆寫工具函式

**問題**：VfxComposerTool 目前只能靜態預覽 Block，無法在預覽時動態修改粒子參數（顏色、速度、大小）。

**參考來源**：官方 3D 粒子 API 文件

**解法**：在 `assets/scripts/core/utils/` 下新增 `ParticleUtils.ts`：
```typescript
export interface ParticleOverride {
    startColor?: Color;
    startSpeed?: number;
    startLifetime?: number;
    rateOverTime?: number;
    startSize?: number;
    gravityModifier?: number;
    loop?: boolean;
}

/** 統一套用粒子覆寫參數 */
export function applyParticleOverride(ps: ParticleSystem, override: ParticleOverride): void {
    if (override.startColor)     ps.startColor.color = override.startColor;
    if (override.startSpeed !== undefined) ps.startSpeed.constant = override.startSpeed;
    if (override.startLifetime !== undefined) ps.startLifetime.constant = override.startLifetime;
    if (override.rateOverTime !== undefined) ps.rateOverTime.constant = override.rateOverTime;
    if (override.startSize !== undefined) ps.startSize.constant = override.startSize;
    if (override.gravityModifier !== undefined) ps.gravityModifier.constant = override.gravityModifier;
    if (override.loop !== undefined) ps.loop = override.loop;
}
```

**Unit Test 評估**：✅ 應建立 — `applyParticleOverride` 是純邏輯函式，可 mock `ParticleSystem` 物件測試每個覆寫欄位是否正確設置。

**驗收條件**：VfxComposerTool 的「顏色/速度/大小」調整滑桿呼叫 `applyParticleOverride`。

### S-4 EffectSystem 加入延遲自動銷毀

**問題**：目前特效節點需手動回收，一次性特效（爆炸、受擊）容易造成節點洩漏。

**參考來源**：cocos-example-cyberpunk 的 `FxBase.destroyTime` 機制

**解法**：
```typescript
// EffectSystem
playOnce(prefabPath: string, position: Vec3, delay: number = 2.0): void {
    const node = this.playEffect(prefabPath, position);
    if (node) {
        this.scheduleOnce(() => {
            this.recycleEffect(node);
        }, delay);
    }
}
```

**驗收條件**：`UnitDied` 事件觸發的死亡粒子使用 `playOnce`，不再需要手動回收。

---

### S-5 EventSystem 事件匯流排強化

**問題**：目前 `EventSystem` 只有 `on` / `off` / `emit` 三個方法，缺少 `once`（單次監聽）、`onBind`（自動綁定 `this`）功能，容易在 Component 銷毀後殘留監聽或因 `this` 丟失產生 bug。

**參考來源**：Cyberpunk 可優化說明書 2 §7（Event 自動綁定裝飾器）

> **Unity 對照**：Unity 沒有這個問題，因為 UnityEvent 是序列化引用。但 TypeScript 事件匯流排是弱型別註冊，需要手動管理解除監聽——`onBind` 就像 UniRx 的 `AddTo(this)` 自動清理。

**解法**：
```typescript
// EventSystem 新增方法
once<T>(eventName: string, callback: (data: T) => void): void {
    const wrapper = (data: T) => {
        callback(data);
        this.off(eventName, wrapper);
    };
    this.on(eventName, wrapper);
}

onBind<T>(eventName: string, callback: (data: T) => void, target: Component): void {
    this.on(eventName, callback);
    // Component 銷毀時自動解除監聽
    target.node.once(Node.EventType.NODE_DESTROYED, () => {
        this.off(eventName, callback);
    });
}
```

**Unit Test 評估**：✅ 應建立 — `once` 的「只觸發一次」語義、`onBind` 的「銷毀後自動解除」邏輯可完整測試。

**驗收條件**：所有 View 層 Component 改用 `onBind` 註冊事件，不再需要 `onDestroy` 中手動 `off`。

### S-6 Cyberpunk 風格中性素材遷移

**問題**：Cyberpunk Demo 中有 18 個風格中性的美術素材（粒子貼圖、光效形狀等）可直接用於三國 Demo，但尚未搬入專案。

## 三、中期優化（1~2 個月）

> **目標**：系統性重構，提升架構品質，為後續功能奠定基礎。

### M-1 UIManager 升級為分層架構

**問題**：目前的 `UIManager` 是單層管理，所有面板共享同一個 Z-order 空間，導致彈窗與主頁面可能互相干擾，且沒有佇列管理機制。

**參考來源**：dgflash/oops-framework 的 `LayerManager` 設計

> **Unity 對照**：類似 Unity 中不同 Canvas 的 Sort Order 分層，但 Oops 進一步為每層定義了行為語義（替換式/堆疊式/佇列式）。

**目標架構**：
```
UIManager（重構）
├── LayerGame     — 遊戲世界層（棋盤 UI）
├── LayerUI       — 主頁面層（替換式，同時只顯示一個）
├── LayerPopUp    — 彈窗層（堆疊式，可多個共存）
├── LayerDialog   — 對話框層（佇列式，一次一個，含遮罩）
├── LayerSystem   — 系統層（最高優先，系統通知）
└── LayerNotify   — 通知層（Toast / Loading）
```

**UIConfig 設定表**（資料驅動，參考 Oops 模式）：
```typescript
const UIConfig: Record<UIID, UIConfigEntry> = {
    [UIID.BattleHUD]:    { layer: LayerType.Game,   prefab: 'battle/BattleHUD',    bundle: 'resources' },
    [UIID.ResultPopup]:  { layer: LayerType.Dialog,  prefab: 'ui/ResultPopup',      bundle: 'resources', mask: true },
    [UIID.Toast]:        { layer: LayerType.Notify,  prefab: 'ui/ToastItem',        bundle: 'resources' },
    [UIID.SystemAlert]:  { layer: LayerType.System,  prefab: 'ui/SystemAlert',      bundle: 'resources', mask: true },
};
```

**Unit Test 評估**：✅ 應建立 — `LayerManager.open/remove/replace` 的狀態機邏輯可完整單元測試。

**驗收條件**：`ResultPopup`、`Toast`、`BattleHUD` 各自對應正確層級，不再需要手動 `setSiblingIndex`。

### M-2 UI 快取機制（NodePool for UI）

**問題**：頻繁開關的 UI（Toast、傷害標籤、Buff 圖示）每次重新 instantiate，造成 GC 壓力。

**參考來源**：oops-framework 的 `LayerUI.ui_cache` + `LayerGame` NodePool

**解法**：
- `LayerPopUp` / `LayerDialog` 的面板 remove 時不 destroy，快取到 `Map<UIID, Node>`
- 再次 open 時直接從快取取出並 reset 狀態（避免殘留舊資料）
- Toast、FloatText 類短壽命節點使用現有 `PoolSystem` 管理

**驗收條件**：反覆開關 ResultPopup 10 次後，記憶體使用量不增長。

---

### M-3 EffectSystem 整合 ParticleUtils + VFX Block 流程

**問題**：VfxComposerTool 目前是獨立的開發預覽工具，尚未與 `EffectSystem` 的執行流程整合。正式戰鬥中播放特效仍使用舊的 prefab 路徑直接載入，缺乏參數覆寫能力。

**目標**：
```
EffectSystem.playBlock(blockId: string, override?: ParticleOverride)
  → 查詢 VfxBlockRegistry 取得 prefabPath + renderMode
  → ResourceManager 載入並快取
  → PoolSystem 取得節點
  → applyParticleOverride(ps, override)
  → playGroup(node)
  → 依 VfxBlock 的 lifetime 計時後 recycleEffect
```

**Unit Test 評估**：⚠️ 部分可測 — `playBlock` 中的查詢邏輯、參數覆寫管道可 mock 測試；實際渲染不測。

**驗收條件**：戰鬥中所有特效呼叫改為 `effectSystem.playBlock()`，移除散落的 `playEffect(hardcodedPath)` 呼叫。

### M-5 ActionSystem — JSON 驅動技能演出時間軸

**問題**：目前技能演出邏輯（動畫、特效、音效、傷害時機）散落在 `BattleController`、`UnitRenderer`、`EffectSystem` 中，新增/修改技能需改程式碼，對設計師不友好且不符合「資料驅動」原則。

**參考來源**：Cyberpunk 可優化說明書 2 §4（ActionSystem — JSON-driven 技能配置）

> **Unity 對照**：類似 Unity Timeline / Playable API 的簡化版。Cyberpunk 的 ActionSystem 將技能流程定義在 JSON 中，runtime 按時間軸依序執行 action。本質上是策劃可編輯的 Sequencer。

**目標架構**：
```typescript
// assets/resources/data/skills.json
{
    "fireball": {
        "timeline": [
            { "t": 0.0, "action": "playAnim", "target": "caster", "clip": "cast_fire" },
            { "t": 0.3, "action": "playVfx", "key": "fireball_launch", "at": "caster" },
            { "t": 0.5, "action": "playVfx", "key": "fireball_hit", "at": "target" },
            { "t": 0.5, "action": "playSfx", "clip": "explode_fire" },
            { "t": 0.5, "action": "applyDamage", "formula": "fire_skill_dmg" },
            { "t": 1.0, "action": "applyBuff", "buffId": "burn", "duration": 3 }
        ]
    }
}
```

```typescript
// ActionSystem.ts（新服務，掛入 ServiceLoader）
class ActionSystem {
    async executeSkill(skillId: string, caster: UnitRenderer, target: UnitRenderer): Promise<void> {
        const def = this.getSkillDef(skillId);
        for (const step of def.timeline) {
            await this.waitUntil(step.t);
            this.dispatchAction(step, caster, target);
        }
    }
}
```

**Unit Test 評估**：✅ 應建立 — `ActionSystem.executeSkill` 的時間軸解析、action dispatch 邏輯是核心測試點。

**驗收條件**：至少 3 個技能透過 `skills.json` 配置演出流程，修改 JSON 後不需改程式碼即可調整演出時機。

### M-6 AudioSystem 動態音軌混合

**問題**：目前 `AudioSystem` 只有單軌 BGM（一個 AudioSource），切換 BGM 時只能硬切（stop → play），缺乏 crossfade 和多軌混合能力。戰鬥從瀏覽到高潮缺乏音景連續性。

**參考來源**：Cyberpunk 可優化說明書 2 §3（MusicSystem 的 multi-track lerp）

> **Unity 對照**：Unity 中通常用兩個 AudioSource + `CrossFade` 或 `Mathf.Lerp(volume)` 實現音軌混合。Cyberpunk 的 MusicSystem 用同樣的原理但封裝了 layers（exploration/combat/boss）的權重控制。

**解法**：
```typescript
// AudioSystem 擴充
private bgmSourceA: AudioSource;  // 現有
private bgmSourceB: AudioSource;  // 新增（用於 crossfade）
private activeBgm: 'A' | 'B' = 'A';

crossfadeTo(clip: AudioClip, duration: number = 1.0): void {
    const from = this.activeBgm === 'A' ? this.bgmSourceA : this.bgmSourceB;
    const to = this.activeBgm === 'A' ? this.bgmSourceB : this.bgmSourceA;
    to.clip = clip;
    to.volume = 0;
    to.play();
    // tween 漸變音量
    tween(from).to(duration, { volume: 0 }).call(() => from.stop()).start();
    tween(to).to(duration, { volume: this.bgmVolume }).start();
    this.activeBgm = this.activeBgm === 'A' ? 'B' : 'A';
}
```

**Unit Test 評估**：⚠️ 部分可測 — `activeBgm` 切換邏輯與 crossfade 狀態機可 mock 測試。

**驗收條件**：戰鬥階段切換（部署→戰鬥→結算）時 BGM 使用 crossfade 過渡，無硬切斷感。

### M-7 VFX_EFFECT_TABLE JSON 化 + 特效-音效-通知三位一體

**問題**：
1. `vfx-usage-table.ts` 目前是 TypeScript 常量表，修改需重編譯，設計師無法直接維護。
2. 特效與對應音效/UI 通知分散在不同系統，新增特效時容易遺漏音效或浮字通知，造成「視/聽/邏輯不一致」。

**參考來源**：Cyberpunk 可優化說明書 2 §12（VFX 管理三位一體化）

**解法**：

**Part A — JSON 化**：
```json
// assets/resources/data/vfx-effects.json
{
    "buff_gain_atk": {
        "prefab": "vfx_skills/buff_gain/prefabs/BuffGainEffect",
        "audio": "sfx_buff_positive",
        "notify": { "type": "floatText", "text": "攻擊力提升！", "color": "#FF6B35" },
        "lifetime": 3.0,
        "pool": true
    }
}
```

**Part B — 三位一體播放 API**：
```typescript
// EffectSystem 新增
playFullEffect(key: string, position: Vec3): void {
    const def = this.getEffectDef(key);  // 從 JSON 查表
    // 1. 播放特效
    this.playBlock(key, position);
    // 2. 播放音效
    if (def.audio) services().audio.playSfx(def.audio);
    // 3. 顯示通知
    if (def.notify) services().floatText.show(def.notify.text, position, def.notify.color);
}
```

**Unit Test 評估**：✅ 應建立 — JSON 載入+解析、三位一體 dispatch 邏輯可完整 mock 測試。

**驗收條件**：新增特效時只需在 `vfx-effects.json` 加一筆設定，呼叫 `playFullEffect(key)` 自動觸發特效+音效+浮字，不再需要在三個系統分別寫程式碼。

### L-1 Bloom 後處理效果

**問題**：戰場視覺缺乏高光溢出感，技能演出的衝擊力不足。

**參考來源**：新版 `CocosCyberpunk/extensions/pipeline/` 的 `pipeline/components/bloom.ts`、`pipeline/passes/bloom-pass.ts`、`pipeline/res/effects/bloom.effect`

> **Unity 對照**：Unity Post Processing Stack 的 Bloom。Cyberpunk Demo 的實作流程：Prefilter → DownSample × N → UpSample × N → Combine，與 Unity URP Bloom 流程幾乎相同。

**目標參數**（Cyberpunk 設定參考）：
- `threshold`：發光閾值（建議 0.8）
- `iterations`：模糊迭代次數（建議 4）
- `intensity`：發光強度（依特效類型動態調整）
- `blurRadius`：模糊半徑

**Unit Test 評估**：❌ 不需要 — 後處理為純渲染，沒有邏輯測試點。

**驗收條件**：武將技能演出時，技能特效節點的 Emissive 材質觸發 Bloom 光暈，視覺衝擊力顯著提升。

### L-6 Ghost 殘影效果（Sprite 適配版）

**問題**：衝鋒/突進類技能缺乏速度感，需要殘影效果來視覺化高速移動。

**參考來源**：Cyberpunk 可優化說明書 2 §5（FxGhostVertex）

> **Unity 對照**：Unity 中殘影常見兩種做法：(1) 每幀複製 MeshRenderer + 降 alpha 淡出；(2) Vertex Shader 沿移動方向偏移頂點。Cyberpunk 的 `FxGhostVertex` 用的是 (2)，但那是針對 3D SkinnedMesh。

**本專案適配方向**：
由於本專案使用 2.5D Sprite 角色而非 3D Mesh，需改用 **Sprite 殘影** 方案：
```
移動開始 → 每隔 N 幀複製當前 Sprite 至殘影容器
  → 殘影 Sprite 設定 tint color（半透明藍/白）
  → tween 淡出 alpha → 0 後回收到 PoolSystem
```
也可考慮 **Trail Renderer** 方案：在角色腳底掛 TrailModule 粒子，移動時自動產生拖尾。

**Unit Test 評估**：❌ 不需要 — 純視覺效果。

**驗收條件**：武將突進時身後產生 3~5 個殘影，逐漸淡出，不影響幀率。

### L-8 Core / Logic 分層純化

**問題**：`scripts/core/` 下的系統（如可能的 BattleController）可能包含遊戲特化邏輯，降低核心框架的可移植性。理想狀態下 `core/` 應只包含引擎抽象與通用服務，遊戲特化邏輯歸入 `battle/` 或 `logic/`。

**參考來源**：Cyberpunk 可優化說明書 2 §12.3（Core/Logic 分層建議）

> **Unity 對照**：類似 Unity 的 Packages/com.company.core（可跨專案共用）vs 項目的 Assets/Scripts/Game（遊戲特化）的分層概念。

**目標分層**：
```
scripts/
├── core/           ← 引擎抽象層（可跨專案共用）
│   ├── systems/    ← EventSystem, PoolSystem, ResourceManager, AudioSystem, UIManager
│   ├── utils/      ← MaterialUtils, ParticleUtils
│   └── base/       ← Component 基類、ViewModel 基類
├── battle/         ← 戰鬥特化邏輯（三國 Demo 專用）

│   ├── FormulaSystem.ts

│   └── DuelSystem.ts
```

**驗收條件**：`core/` 下所有 import 不引用 `battle/` 或 `ui/`，用 lint 規則自動檢查。

## 五、評估後不納入或降級的項目

> 以下項目來自 Cyberpunk 可優化說明書 2，但經評估與本專案（2.5D 策略棋盤）的特性不相容或投入產出比過低，因此不納入或僅作為備用方案。

### ❌ Steering Behaviors（不納入）

**來源**：可優化說明書 2 §2（原始標記 P0）

**降級理由**：
- 本專案為 **格子制策略遊戲**，單位移動是 **離散的格子到格子**（tween + easing），非連續空間的自由移動
- Steering Behaviors（seek / flee / arrive / wander）設計用於 **連續 2D/3D 空間**的自然運動模擬，在棋盤上完全多餘
- 現有 `UnitRenderer` 使用 `tween(worldNode).to()` 搭配 `easing('smooth')` 已能產生流暢的視覺移動
- `arrive()` 的漸進減速概念可在中期借鑑——直接在 tween easing 函式中實現類似效果即可，不需要完整的 Steering 框架

**替代方案**：在 M-tier 的 tween easing 優化中，借用 `arrive()` 減速曲線的概念，用自訂 easing 函式實現格子移動的「接近時自然減速」效果。

### ⚠️ IK 瞄準系統（不適用）

**來源**：Cyberpunk Demo 的 IK 射擊瞄準

**降級理由**：
- IK 瞄準用於 3D 角色即時指向敵人，本專案為 **回合制自動戰鬥 + 2D Sprite 角色**
- Sprite 角色無骨骼導致 IK 無法實施，且回合制不需要即時瞄準
- 完全不在計畫範圍內
