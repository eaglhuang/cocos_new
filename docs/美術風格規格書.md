# 美術風格規格書

更新日期: 2026-04-06
適用範圍: `3KLife` 全專案 UI，尤其是人物頁、血脈頁、商店、信件、戰令、九宮格框體與 skin family 設計
主要依據:
- `docs/UI參考圖品質分析.md (doc_ui_0051)` (doc_ui_0051)
- `docs/UI品質檢核表.md (doc_ui_0050)` (doc_ui_0050)
- `assets/resources/ui-spec/ui-design-tokens.json`

## 1. 文件目的

本文件把 `UI參考圖品質分析.md` (doc_ui_0051) 的高階品質判斷，轉成專案可以直接落地的視覺規格。

之後所有 Agent 在做：
- `layout / skin / screen` 設計
- sprite family 規劃
- 九宮格框體切圖
- design tokens 調整
- 顏色與質感 QA

都應優先遵守本文件。

## 2. 視覺北極星

本專案目前最應靠近的不是「厚重黑金傳奇裝備介面」，而是：

`Warm Parchment Base + Jade Header Chrome + Controlled Dark Contrast + Readable Atlas Hierarchy`

翻成更直白的標準：

- 大面積閱讀區要像高級圖鑑、古卷、羊皮紙承載面。
- 可互動元件才用深色金屬、黑金、深底高對比。
- jade 是點綴與 header 語言，不是全頁主底色。
- 金色是視覺節點與 CTA 強調，不是萬用主色。
- 高品質感來自結構、留白、材質、對比秩序，不來自堆滿重色、重 glow、重裝飾。

## 3. 與參考圖分析的差異結論

### 3.1 參考圖標準

根據 `UI參考圖品質分析.md` (doc_ui_0051)，參考圖的核心品質來源是：

- 背景層有水墨與紙感，不是死黑或死白。
- 框體永遠在背景之上「浮起」，不直接硬貼平面。
- 深框與淺框有明確分工。
- 文字顏色、描邊、底色之間有穩定規則。
- 大面積內容承載使用 `Parchment Frame` 或其變體。
- `Dark Metal Frame` 主要用在互動、數值、可選中物件，不是閱讀主面板。

### 3.2 我們專案目前最常犯的偏差

- 把 `dark metal / deep green relic / black-gold` 當成萬用高級感。
- 把 jade 當成整塊主底，而不是局部匾額、命紋、accent。
- 把 proof、source、flat sample 或切片樣張當成正式 UI 資產。
- 用顏色重量補結構不足，導致畫面變俗、變髒、變不透氣。

### 3.3 品質判斷

- 參考圖標準的品質更高，也更成熟。
- 我們目前專案中，偏 `r24-card-family-rebalance` 那條路線比較接近參考圖。
- 偏 `r25 ~ r27` 那種深綠重金、crest 素材語義錯置的方向，品質較差，且明顯偏離參考圖。

## 4. 框體家族與用途規格

本節是給 Agent 最重要的落地對照表。

### 4.1 `Deep Metal Frame`

用途:
- CTA 按鈕
- 小型互動按鈕
- 可選中 tab
- 裝備格、物品格、徽章槽
- 深底數值摘要卡

不適合:
- 長段閱讀文字
- 人物頁右側大面積總覽卡
- 血脈故事正文

專案對應 token:
- 底色: `surface`, `surfaceContainer`, `surfaceHigh`, `surfaceSepia`, `surfaceSepiaWarm`
- 文字: `textPrimary`, `textSecondary`, `textAccent`
- 邊框: `secondary`, `outline`, `outlineHeavy`

視覺規則:
- 只能局部使用，不能占滿畫面主體。
- 若整頁深框比例超過 35%，通常代表畫面太重。
- 深框中的文字要有明確描邊或暗陰影。

九宮格建議:
- border: `20~24px`
- bleed: `2~4px`
- 角區須保留完整金屬裝飾，不得 trim 掉角光或鉚釘

### 4.2 `Parchment Frame`

用途:
- 長文閱讀面
- 信件內文
- 獎勵展示大面板
- 人物頁 summary / bloodline / lore 類資訊卡
- 條目式資訊承載區

不適合:
- 主 CTA
- 高互動 toggle
- 危險操作

專案對應 token:
- 底色: `surfaceParchment`, `surfaceParchmentFill`, `stdPanelSummary`, `stdPanelModule`, `etcPanelMain`, `etcPanelModule`
- 文字: `textOnParchment`, `textOnParchmentMuted`
- 分隔: `dividerOnParchment`

視覺規則:
- 大面積閱讀資訊優先用淺色 parchment。
- 羊皮紙不應配黑描邊大字。
- 材質要輕，noise 只做紙感，不做髒污。

九宮格建議:
- border: `16~20px`
- bleed: 通常不用，改用極輕投影
- 若有紙張角落做舊效果，角區不得小於 `16px`

### 4.3 `Jade-Parchment Header`

用途:
- 頁面主標題匾額
- 區段主題條
- 人物頁 header chrome
- 稀有度 band / 模式 badge 的上層主結構

不適合:
- 整張卡背景
- 可大量重複的列表卡底
- 正文閱讀區

專案對應 token:
- jade: `stdButtonPrimary`, `stdButtonSecondary`, `stdArtCrest`, `mirrorSigilField`
- parchment: `surfaceParchment`, `surfaceParchmentFill`
- 文字: `textOnParchment`, `textOnParchmentMuted`

視覺規則:
- 必須維持 `bleed + band + frame` 三層。
- jade 只佔 header 視覺總量的小比例，建議低於 header 面積的 30%。
- 主要任務是建立匾額秩序，不是建立大面積深色存在感。

#### 4.3.1 共享牌匾 + tier-mark 規則（2026-04-08）

- `header-rarity-plaque` 是目前已驗證可跨 family 共用的 header recipe，已落地於 `GeneralDetailOverview`、`SpiritTallyDetail`、`Gacha`。
- header 本體必須拆成 `base + fill + band`；`base` 先建立匾額輪廓，`fill / band` 才提供 jade-parchment 的材質與節奏。
- 稀有度正式方向改為 `R / SR / SSR / UR / LR` 的 tier-mark dock，不再使用整張稀有度 badge 圖硬縮進固定窄框。
- tier 色只允許進入 `badge underlay / badge plate / badge label`，不可把整塊 header fill、整張 card 背景或整頁 parchment 直接染成稀有度色。
- tier dock 的視覺重量應高於副標，但不得蓋過主標；它屬於商業導視層，不是頁面主標題。
- `limited / event` 類畫面可提高 tier dock 的彩度與對比，但仍必須維持「小面積高彩、主體低彩」原則，避免把整個 header 做成促銷橫幅。

#### 4.3.2 已驗證的 runtime 用例

- `GeneralDetailOverview`: 驗證 tier-mark dock 可以取代被壓扁的稀有度 badge 圖。
- `SpiritTallyDetail`: 驗證第二個 family 可重用同一套 plaque header 與 rarity dock。
- `Gacha`: 驗證第三個 family 可把 pool 名稱、副標與 tier dock 收進共用 top plaque，並可切換 `hero / support / limited` 三池 preview state。

九宮格建議:
- band border: 左右 `20px` 起
- frame border: `16px`
- bleed opacity: `0.08 ~ 0.18`

### 4.4 `Gold CTA Frame`

用途:
- 主 CTA
- 確認、購買、升級、領取、進階入口

不適合:
- 次級資訊卡
- 文字標籤底
- 大面積區塊標題

專案對應 token:
- 邊框: `secondary`, `secondaryLight`, `resourceGold`
- 內填: `surfaceSepia`, `surfaceSepiaWarm`
- 文字: `textAccent`, `textWarmGold`, `textOffWhite`

視覺規則:
- 金 CTA 要稀少，整屏不應超過 1~2 個主 CTA。
- 只在互動層級最高的地方使用粗金框。
- 若畫面中到處都是金框，整體質感會立刻降級。

九宮格建議:
- border: `24~32px`
- 外 glow: 可有，但 blur 與 opacity 要收斂
- pressed: 內縮 `1px`

### 4.5 `Destructive Red Frame`

用途:
- 刪除
- 卸除
- 放棄
- 破壞性操作

規則:
- 只在危險操作使用，不可用來增加一般畫面戲劇性。
- 同畫面通常只容許 1 組紅色破壞元件。

### 4.6 `Item Cell Frame`

用途:
- 九宮格物品槽
- 裝備格
- 戰利品格
- icon 容器

規則:
- 固定尺寸時優先用 `simple sprite`。
- 響應式格網才使用 9-slice。
- 稀有度色只影響邊框與 badge，不可污染整塊 fill。

建議尺寸:
- `80~100px` 為主
- 9-slice border: `8px`

### 4.7 `Overlay / Bleed / Shadow`

用途:
- 背景壓暗
- 框體浮起
- 角色與背景分離

規則:
- `overlay` 負責壓亮度，不負責提供質感。
- `bleed` 負責收邊，不負責搶焦點。
- `shadow` 負責拉層級，不可做成黑塊。

### 4.8 `Battle Glyph / Icon 尺寸契約`

用途:
- `battle-badge-icon-v1` 的 troop type glyph
- `TigerTally` type badge 內的兵種符號
- `UnitInfoPanel` 的 type icon

規則:
- 若任務卡未另外指定，BattleScene 的兵種 glyph / badge icon 一律以 `256x256` 方形 master 為預設輸出尺寸。
- `stat micro glyph` (如: atk / hp / cost) 定位為內容層的微型符號，與數值搭配使用，必須在 `16~24px` 保持清晰辨識度。
- 微型符號禁止烘焙任何文字、數字 (禁含 ATK 等)、字母。
- `cost` 圖示語意一律視為「軍糧、補給」，禁止畫成金幣、元寶或商城貨幣。
- `hp` 圖示允許使用直觀的紅愛心 (Red Heart) 或戰場生存意象，唯需保持風格統一。
- `glyph` 與 `underlay` 必須分件；不得因為圖示小就把框體、底板或整塊 badge 烘進 glyph PNG。
- 構圖必須是單一主體置中、留出安全邊界；不可生成橫幅式、場景式或插畫式構圖。
- `glyph` 的可視主體建議控制在畫布寬高的 `60%~72%`，保留 runtime 疊到 underlay 時的呼吸空間。
- 線條語言：採用日式動漫古風平塗式 (Flat Anime Style)，粗黑描邊結合俐落 Cell-shading，強調在極小尺寸下的輪廓辨識度。
- 若某張畫面真的需要大於 `256x256` 的 master，必須在任務卡明寫理由與目標 runtime 尺寸，不能由生圖 prompt 臨時放大。

不建議:
- 為了追求細節直接要求 `512+` 大圖，再回頭硬縮成戰場小 icon。
- 生成非方形長寬比，例如 `1408x768` 這類偏插圖輸出。

## 5. 深淺色搭配規則

### 5.1 大原則

- 亮背景上，主資訊卡可用深框或淺框，但需看內容密度。
- 暗背景上，主閱讀面優先用淺框。
- 同一畫面的大閱讀區與小互動區，應優先形成「淺面 + 深物件」或「深場 + 淺紙」對比。

### 5.2 推薦配法

#### 配法 A: `Dark Scene + Parchment Reading Card`

適用:
- 信件
- 人物總覽
- 故事頁
- lore 頁

效果:
- 最接近參考圖成熟度
- 可讀性最好
- 最容易做出透氣感

#### 配法 B: `Parchment Main Panel + Dark Item Cells`

適用:
- 戰令
- 商店
- 獎勵展示
- 道具總覽

效果:
- 淺底承載資訊
- 深色格子突出可領取物件

#### 配法 C: `Dark Module + Gold CTA`

適用:
- 強互動功能面
- 精鍊、升級、確認行動

效果:
- 焦點清楚
- 互動引導強

### 5.3 不推薦配法

- `Deep Green Full Card + Gold Text`
- `Black Metal Full Page + Multiple Gold Frames`
- `Jade Full Panel + Jade Crest + Jade Header` 全部同色系堆滿
- `Parchment Card + Heavy Black Outline Everywhere`
- 把整張 `SSR / UR` 美術 badge 硬塞進窄框後再縮放，導致字形被壓扁、商業感變廉價

### 5.4 `BattleScene Tactical HUD` 補充（2026-04-08）

BattleScene 不是一套和大廳 / 人物頁 / 商店完全切開的第二套 UI 美術系統。

Battle HUD 仍然屬於同一套全域 UI 美術語言，只是因為功能不同，落地時要改成「戰場主畫面 + 指揮 HUD」的戰術化變體。

核心統一原則：

- 全專案共享同一套材質觀、框體秩序、token 與 CTA 階層。
- Battle HUD 與大廳 / 人物頁 / 商店的差異，主要來自資訊密度、互動優先級、對比分配與可讀性要求，不來自另一套獨立審美宇宙。
- 戰場全景地圖的晨昏、天氣、季節、冷暖色調，屬於 `Scene Background` 的敘事與氣氛問題，不應直接改寫 HUD 的正式 family 語言。
- 當底圖從夜景切到白天、陰雨切到晴天時，HUD 可以做的是透明度、亮度、局部描邊、陰影與對比微調；不能因此把常態 HUD 改成人物頁 plaque 語言，也不能把整套 HUD 換成跟著背景跑的主題皮膚。

適用語言:
- `deep-ink battlefield + cold tactical HUD + gold CTA`
- 戰場背景與單位模型是第一主體，HUD 只能做框定、導視、操作，不可變成壓在場景上的第二個主舞台。
- 這裡的 `deep-ink / cold tactical` 指的是 HUD 的對比與資訊層級策略，不是要求背景地圖永遠是冷夜色。
- Top HUD、TigerTally、Action Command 可用冷色深底與局部金色高光，但亮度必須服從主 CTA 階層。
- `BattleLog`、`ControlBar`、utility button 要保持低存在感，不能和奧義、關鍵戰況提示搶主焦點。

不適用語言:
- 不把 `Jade-Parchment Header` 當作主戰場常態 header。
- 不把人物頁 / 轉蛋頁的 collectible-card、crest medallion、商業 plaque 語言直接搬進戰場主 HUD。
- 不把 reward / reveal 的高彩框體放進常態戰場資訊層。

落地原則:
- Battle HUD 與其他系統 UI 共享同一套全域美術基底，不可被誤解為完全不同 style bible。
- 主戰場的深色面板只負責可讀性，不負責高級感堆料。
- 金色只用於最高優先操作、勝利/關鍵提示，不作為整排工具列的常駐邊框色。
- 若某個 HUD 元件亮度已經接近主 CTA，就必須下調，避免整屏同時搶焦。
- 若背景地圖非常亮或非常暗，優先調整 HUD 的 alpha、stroke、shadow、separator 與 panel depth，不是改掉戰場常態 HUD family。

## 6. 全域配色原則

### 6.1 色彩角色分層

- 背景色: 建立氣氛，不能搶資訊
- 承載色: 承接閱讀內容，是畫面的主要平衡面
- 強調色: 導視、互動、稀有度、進度、選中
- 文字色: 永遠服從底色對比，不服從情緒
- 狀態色: 成功、警告、危險、鎖定

### 6.2 全域顏色分工

| 角色 | 推薦 token | 說明 |
|---|---|---|
| 背景暗端 | `background`, `backgroundDeep`, `backgroundMid` | 水墨暗部、場景壓暗 |
| 背景亮端 | `surfaceParchment`, `surfaceParchmentFill` | 宣紙、留白、承載面 |
| 主金色 | `secondary`, `secondaryLight`, `resourceGold` | CTA、框邊高光、主強調 |
| jade 強調 | `stdButtonPrimary`, `stdButtonSecondary`, `stdArtCrest`, `mirrorSigilField` | header、命紋、少量 accent |
| 正文深色 | `textOnParchment`, `textInk` | 羊皮紙上文字 |
| 深底白字 | `textPrimary`, `textSecondary` | 深底資訊與互動元件 |
| 正向色 | `success`, `tertiaryLight`, `textPositive` | 成功、增益、進度 |
| 風險色 | `error`, `textNegative`, `rarityRed` | 危險與破壞 |

### 6.3 色彩數量控制

- 單一頁面主色群建議不超過 3 組:
  - 背景群
  - 承載群
  - 強調群
- 稀有度色屬於局部狀態色，不可算成主色群全面擴散。
- 同一區塊內若同時出現金、jade、紅、綠、藍、紫，通常表示色彩治理失敗。

## 7. 色彩管理規範

### 7.1 Token 優先

- 正式 skin JSON 優先使用 token key，不優先直寫 hex。
- 若必須直寫 hex，只允許 proof、compare 或短期過渡，結案前要收回 token。
- 同用途色不能在不同檔案各自長出新 hex。

### 7.2 禁止事項

- 禁止把「看起來差不多」的顏色在每個 screen 裡微調成不同版本。
- 禁止用 opacity 疊很多層去硬湊出一個沒有正式定義的顏色。
- 禁止用深綠或黑金去掩蓋 layout 層次不足。

### 7.3 Token 分層建議

之後所有新 token 應優先分類為：

- `background.*`
- `surface.*`
- `surface.parchment.*`
- `accent.gold.*`
- `accent.jade.*`
- `text.*`
- `status.*`
- `rarity.*`

若新 token 無法歸入上述分類，代表命名或用途需要再思考。

### 7.4 目前正式 alias

`ui-design-tokens.json` 應以 alias 方式逐步收斂，不直接大量改名舊 key，避免既有 skin family 一次斷裂。

目前新畫面優先使用：

- `accent.gold.base`
- `accent.gold.light`
- `accent.gold.cta`
- `accent.jade.base`
- `accent.jade.light`
- `accent.jade.crest`
- `accent.jade.field`
- `surface.parchment.base`
- `surface.parchment.fill`
- `surface.parchment.summary`
- `surface.parchment.module`
- `surface.parchment.main`
- `text.parchment.primary`
- `text.parchment.secondary`
- `divider.parchment`

舊 token 例如 `secondary`、`surfaceParchmentFill`、`stdButtonPrimary` 暫時保留作為 backward-compatible alias；新 spec、skin 與新 family 不再優先擴張舊式平鋪命名。

### 7.5 同步原則

- 改 `ui-design-tokens.json` 時，要同步檢查受影響的 skin family。
- 改 family 基準色時，要先看是否會影響：
  - `general_detail`
  - `equipment`
  - `commerce`
  - `mail`
  - `battle_pass`

## 8. `GeneralDetailBloodlineV3` 專用規格

### 8.1 目前可用結論

- `r24-card-family-rebalance` 是最近穩定基線。
- 右側大資訊卡應延續 `Parchment Frame` 邏輯。
- `header` 可用 jade-parchment。
- `crest card` 背景不能整塊壓成深綠。
- `crest` 本體目前可暫用 proof medallion，但前提是它看起來像完整徽章，不是切片樣張。

### 8.2 推薦搭配

- 背景: parchment + 極輕 noise
- header: jade-parchment 三層 chrome
- summary cards: light parchment variations
- bloodline summary: 偏 parchment / pale jade
- crest: pale jade glow + 完整 medallion
- CTA / badge: 稀少金色點綴

### 8.3 明確禁止

- `dark relic` 式深綠大面板
- `gold title on dark green block` 當成主閱讀區
- crest 使用 `source / flat / sample corner`
- 讓稀有度 badge 的視覺重量大於頁面主標題

## 9. 九宮格與切圖規格

### 9.1 角區保留

| 家族 | 建議 border |
|---|---|
| Deep Metal | `20~24px` |
| Parchment | `16~20px` |
| Jade Header Frame | `16~20px` |
| CTA | `24~32px` |
| Item Cell | `8px` |

### 9.2 Padding 與 trim

- atlas padding 至少 `4px`
- 含 glow、金屬角、紙張破損角的 sprite，不可過度 trim
- 若角區圖樣在 preview 中被拉扁、缺角、斷邊，直接判 fail

### 9.2b 連續紋樣整框禁則

- 凡是四邊存在連續捲草紋、回字紋、龍紋、雲紋，且四角本身是完整裝飾構圖的「整框」資產，**不得直接當成一般 9-slice 框體**。
- 原因不是 border 數值不夠，而是中段一旦被拉伸，邊紋節奏、線寬、間距與角區構圖會一起變形。
- 這類框體只允許三種做法：
  1. 固定尺寸 / 等比縮放的單張框
  2. 先拆成 `corner + edge + center/fill` 的分段框體，再只拉伸 stretch-safe 的中段
  3. 改畫成「角區重 ornament、邊中段低紋理」的 9-slice 專用版本
- 若資產第一眼像你提供的這種完整花角外框，就預設歸類為 `non-9-slice ornate frame`，除非美術明確提供可拉伸安全區。

### 9.3 Noise 與 bleed

- `noise` opacity 建議在 `0.008 ~ 0.03`
- `bleed` opacity 建議在 `0.08 ~ 0.18`
- 超過這個範圍通常會開始髒或開始俗

## 10. QA 驗收句型

3. 這版 crest 是完整徽章，還是只是素材樣張？

## 11. 最終標準句

`3KLife` 的高品質 UI，不是做得更重、更亮、更金，而是更有秩序、更有材質、更有呼吸感。

在本專案裡：
- 閱讀用淺
- 互動用深
- jade 只做匾額與靈性點綴
- gold 只做焦點與權重提升
- 任何違反這五句的方案，都應先被質疑，而不是先被實作
