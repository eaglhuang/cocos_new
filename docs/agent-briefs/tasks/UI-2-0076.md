# [UI-2-0076] Component Binder Migration — 11 個元件批量遷移

## 開單原因

`UITemplateBinder` 已提供完整的 `bind(rootNode, spec)` API，`DuelChallengePanel.ts` 是正確使用範例，但以下 11 個元件仍使用脆弱的舊式手動節點操作：

| 元件 | 問題模式 |
|------|---------|
| `BattleHUD.ts` | 15+ 個私有節點引用，`onBuildComplete` 手動填充 |
| `GeneralDetailPanel.ts` | `_bindClick('RightTabBar/BtnClose', ...)` 路徑字串、手動 Color |
| `GeneralListPanel.ts` | `getChildByPath` + `.string` 直接賦值 |
| `GeneralPortraitPanel.ts` | 手動節點查找 |
| `GeneralQuickViewPanel.ts` | 手動節點查找 |
| `ActionCommandPanel.ts` | 手動節點查找 |
| `ToastMessage.ts` | `getChildByName` |
| `BattleLogPanel.ts` | 手動 Label 設置 |
| `ResultPopup.ts` | 部份手動 |
| `UltimateSelectPopup.ts` | `onReady()` 未完整 |
| `StyleCheckPanel.ts` | 手動節點查找 |

路徑字串（如 `'RightTabBar/BtnClose'`）在 JSON 結構調整後不會報編譯錯誤，只會在 runtime 靜默失敗——這是目前最大的維護性脆弱點。

對照 Unity：等同 GameObject.Find/Transform.Find 硬編碼路徑，而不是 `[SerializeField]` Inspector 綁定。

## 完整描述

以 `DuelChallengePanel.ts` 為模板，逐一遷移：

```typescript
// 目標模式 (DuelChallengePanel 示範)
protected async onReady(binder: UITemplateBinder): Promise<void> {
    const btnPrimary = binder.getButton('btnPrimary');
    const titleLabel = binder.getLabel('titleText');
    btnPrimary?.node.on(Button.EventType.CLICK, this._onConfirm, this);
}
```

**遷移優先順序（建議）：**
1. `BattleHUD.ts` — 最多私有 ref，影響最重
2. `GeneralDetailPanel.ts` — 路徑字串最多
3. `GeneralListPanel.ts`、`GeneralPortraitPanel.ts`、`GeneralQuickViewPanel.ts`
4. 其餘元件

## 每個元件遷移步驟

1. 確認 layout JSON 已有 `id` 欄位（若缺少，補入 `"id": "btnClose"` 等）
2. 移除所有 `private _xxxNode: Node` 類私有節點引用
3. 移除 `getChildByName`、`getChildByPath`、`_bindClick` 路徑字串呼叫
4. 改為在 `protected async onReady(binder)` 中用 `binder.getLabel/getButton/getNode()`
5. 確認 runtime 行為等同於遷移前
