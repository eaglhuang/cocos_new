# [UI-2-0075] UISpecLoader 升級為 ServiceLoader 共享單例

## 開單原因

`assets/scripts/ui/core/UISpecLoader.ts` 在 15 個元件中各自被 `new UISpecLoader()` 建立獨立實例：

```
GeneralDetailPanel, BattleLogPanel, DuelChallengePanel, ActionCommandPanel,
BattleHUD, ResultPopup, UnitInfoPanel, GeneralQuickViewPanel, GeneralListPanel,
GeneralPortraitPanel, TigerTallyPanel, ToastMessage, StyleCheckPanel,
UltimateSelectPopup, UIScreenPreviewHost
```

每個實例都持有各自的 `_cache: Map<string, T>`，代表：
- 同一份 JSON 可能被讀入記憶體多次
- 快取清除 / 失效無法跨元件協調
- 無法由 ServiceLoader 統一管理生命週期

對照 Unity：等同每個 MonoBehaviour 各自維護 `Resources.Load()` 快取而不走 Addressables 共享 cache。

## 完整描述

1. 在 `ServiceLoader.ts` 加入 `specLoader: UISpecLoader` 共享入口：
   ```typescript
   // ServiceLoader.ts
   this._specLoader = new UISpecLoader(this._resource);
   ```
2. `UISpecLoader` 建構子改接受 `ResourceManager`（或直接從 `services()` 取），確保快取可共享：
   ```typescript
   export class UISpecLoader {
       constructor(private _rm: ResourceManager) {}
   }
   ```
3. 各元件移除 `private _specLoader = new UISpecLoader()`，改為：
   ```typescript
   const loader = services().specLoader;
   ```
4. 若部分元件在取得 `services()` 前就用到 loader（如建構期），改為 lazy init 或 `onLoad()` 延後取得。

## 如何驗證

1. `ServiceLoader.ts` 有 `get specLoader(): UISpecLoader` accessor
2. grep `new UISpecLoader()` → 0 個命中（或只剩 ServiceLoader 自身）
3. 所有 15 個元件改為取用 `services().specLoader`
4. 多個元件同時開啟時，同一 JSON 只被載入一次（可搭配 `console.log` 驗證快取命中）
5. `run-acceptance.js` 全綠
