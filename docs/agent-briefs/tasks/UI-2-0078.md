# [UI-2-0078] MemoryManager 基礎實作（LRU + releaseByScope）

## 開單原因

`MemoryManager.ts` 程式碼有明確的自我說明：

```typescript
// 追蹤層空殼，預留完整擴充點
// onThresholdExceeded = null;
// onAssetFullyReleased = null;
// releaseByScope() — 預留接口
```

目前所有資產都靠 `ResourceManager` 的 `Map<string, T>` 快取，**沒有** LRU 淘汰、**沒有** `releaseByScope()`、**沒有**記憶體壓力偵測。在 Demo 規模下可接受，但一旦 22 個畫面輪流開過，SpriteFrame / JSON 會持續佔用記憶體，且無任何釋放機制。

對照 Unity：等同所有 `Resources.Load()` 資產從不呼叫 `Resources.UnloadAsset()` 或 `Resources.UnloadUnusedAssets()`。

## 完整描述

### 最小可行實作

1. **LRU Cache 替換 `ResourceManager` 的 Map**：
   - 設定最大容量（SpriteFrame: 100、JSON: 200，可由 config 調整）
   - 超過容量時淘汰最少使用的 entry，呼叫 `asset.decRef()` / Cocos release

2. **`releaseByScope(scopeId: string)` 接口**：
   - `ResourceManager.loadSpriteFrame/loadPrefab` 接受可選 `scope` 參數並記錄
   - `MemoryManager.releaseByScope('BattleScene')` 釋放所有標記該 scope 的資產

3. **接通 MemoryManager 鉤子**：
   ```typescript
   memManager.onThresholdExceeded = (usageBytes) => {
       console.warn(`[Memory] 超過閾值: ${usageBytes} bytes`);
       services().resource.releaseOldestEntries(10);
   };
   ```

4. **不實作**：複雜的 GC root 追蹤、Bundle 動態解除安裝（這是 Phase 3 的工作）

## 如何驗證

1. `MemoryManager` 有 `releaseByScope(scopeId)` 可呼叫且不 throw
2. 連續載入超過 LRU 容量的資產後，最舊的 entry 被淘汰（可用 Map.size 驗證）
3. `BattleScene` → `LobbyScene` 切換時可呼叫 `releaseByScope('BattleScene')` 釋放戰鬥資產
4. `project.log` 無 decRef / release 相關錯誤
5. `run-acceptance.js` 全綠
