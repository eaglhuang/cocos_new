# UI-2-0080 — Content Contract Framework 實作

## 背景

AI Agent 目前可生成三層 JSON 骨架（layout/skin/screen），但 screen spec 中的內容欄位（`titleKey`、`bodyKey`、`tabs`、`railItems` 等）是鬆散的，沒有正式契約。這導致 Agent 產出的 screen config 是不完整的半成品，仍需人工填充。

Unity 對照：相當於只寫了 `Instantiate(prefab)` 但沒有 `ScriptableObject` 定義欄位，大幅降低自動化程度。

## 目標

為每個 template family 定義可驗證的 `ContentContractSpec`，讓 Agent 生成的 screen config 能被機器驗證，而非只靠人工審閱。

## 實作清單

### 1. UISpecTypes.ts 擴充（已完成，2026-04-05）
- [x] 新增 `ContentContractRef` 介面
- [x] `UIScreenSpec` 新增 `contentRequirements?: ContentContractRef`

### 2. UIContentBinder.ts（新建）
位置：`assets/scripts/ui/core/UIContentBinder.ts`

職責：
- 接收 `ContentContractRef` + runtime data object
- 型別安全地把欄位映射到 `UITemplateBinder` path
- 缺必填欄位時 `console.warn`（不 throw，不 silent）

### 3. Content Schema JSON（新建）
位置：`assets/resources/ui-spec/contracts/`

| 檔案 | Family |
|------|--------|
| `detail-split-content.schema.json` | `detail-split` |
| `dialog-card-content.schema.json` | `dialog-card` |
| `rail-list-content.schema.json` | `rail-list` |
| `fullscreen-result-content.schema.json` | `fullscreen-result` |

Schema 欄位格式：
```json
{
  "schemaId": "detail-split-content",
  "familyId": "detail-split",
  "version": 1,
  "fields": {
    "titleKey": { "type": "i18n-key", "required": true },
    "bodyKey":  { "type": "i18n-key", "required": true },
    "tabs":     { "type": "array",    "required": true, "minItems": 1 }
  }
}
```

### 4. validate-ui-specs.js 擴充
- 加入 `--check-content-contract` 旗標
- 讀取 screen spec 的 `contentRequirements`
- 載入對應 schema，驗證 `requiredFields` 是否都有值

### 5. 架構文件
位置：`docs/ui/content-contract-framework.md`（已建立）
