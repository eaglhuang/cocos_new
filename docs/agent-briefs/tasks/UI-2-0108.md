# [UI-2-0108] 虎符卡 Battle / 人物頁 共用契約整合

## 目標

建立統一的 `TigerTallyUnifiedCard` 雙態元件，戰場與武將頁共用同一個 prefab，僅透過 skin 切換視覺語言。從資料、layout、戰場操作到人物查詢都讀同一套 `rarity / badge / card-art / source crest / ATK / HP` 語法。

這張卡同時涵蓋三件事：

1. battle 上的虎符派遣卡（State A 摘要態為主，長按可快覽 State B）
2. 武將頁中的虎符裝備查詢卡（State A 預設，點擊展開 State B 詳情）
3. 提供美術可交付的最基本完整虎符卡資產包

### 雙態架構（v4 統一）

- **State A（摘要態）**：美術圖填滿 + ATK（底部左側大字）+ HP（底部右側大字）+ 右側兵種/戰法 icon 垂直列 + 稀有度外框 + 糧草消耗 badge + 兵種名稱。戰場與武將頁完全共用此 layout。
- **State B（展開態）**：右側 slide-out 面板（0.3s ease-out），顯示完整描述、特殊戰法列表、進階數值、來源 crest + 名將小頭像 + 家族徽記。
- **Skin 切換**：`battle-tally-skin`（cold tactical HUD）/ `parchment-tally-skin`（parchment-first 閱讀面）。Layout JSON 共用一份，skin JSON 各一份。

## 視覺 Guardrails

1. Battle 不可長成 jade-parchment 匾額或收藏卡面；它是戰術派遣卡，不是抽卡卡面。
2. 人物頁不可反向套用 battle 的厚重戰術框；它應該像同語法的 parchment summary module。
3. 稀有度與 badge 必須 runtime 驅動，不得把等級或兵種字樣烘進單張 PNG。
4. 卡片層次必須拆開：`card body / card art / rarity frame / badge / cost plate / source crest / state tag`，不可合烘成單張不可維護的大圖。
5. Source crest 必須能讓玩家一眼看出該虎符來自死亡、招降還是封賞，不得只在 tooltip 才看到。
6. 若虎符屬於特定名將遺留版本，卡上可額外出現小頭像與家族小徽記，但它們只能是 provenance 級別的次層識別件，不可壓過兵種 badge、rarity 與糧草資訊。
7. 家族小徽記中的單字印記（例如關氏家族的小 `關` 字）必須控制在 micro-seal 尺度，讀起來像家族標誌，而不是卡片主標題。

## Reject If

1. battle 與人物頁各做一張毫無關聯的卡，只有名稱一樣（必須共用同一個 `TigerTallyUnifiedCard` prefab）。
2. 稀有度、兵種或來源資訊被燒進同一張不可替換的 PNG。
3. battle 卡仍然無法顯示 `Food Short / Cap Full / Downgrade / Set Active` 至少四種基本戰場狀態。
4. 人物頁的虎符卡 State B 展開態不能作為 `SpiritTallyDetail` 的正式查詢入口。
5. 資產只能在單一武將或單一兵種成立，換資料就破版。
6. 名將小頭像或家族小徽記一旦上卡就搶走主視覺，導致虎符讀成角色收藏卡而非兵種卡，視為不通過。
7. ATK/HP 數值不在 State A 底部左右顯示，或兵種 icon 不在右側垂直列。

### Agent2 美術總監退件意見（兵種 Badge 首批 5 張）

這一批 5 個 badge 全數退件，不准沿用，也不准只做局部修補。核心問題不是單張畫得醜，而是整批的設計方向從一開始就偏了：它們現在讀起來是「小型角色插畫貼在牌子上」，不是「可縮放、可系統化、可在 32px~48px 上穩定辨識的兵種 badge」。

#### 主要問題

1. **語言錯置**：badge 應該是 emblem / insignia / tactical marker，但現在是完整角色或多角色插畫。這會直接破壞 battle card 的資訊層級。
2. **縮小後不可讀**：騎兵、長槍兵是多人物或大動作敘事構圖，縮到實際使用尺寸時只會變成一團噪訊，無法一眼辨識兵種。
3. **family 不一致**：弓、騎、步、盾、槍五張的背景、材質、鏡頭距離與敘事重量完全不同。長槍兵是紅底群像，盾兵是單體冷灰構圖，騎兵又偏紫藍奇幻，這不是同一套 badge family。
4. **中心符號不明**：兵種 badge 需要穩定的中心 glyph 或 silhouette anchor，現在每張都在講自己的小場景，沒有共同的骨架。
5. **材質與價值層級錯亂**：現有金框與底板太像裝飾相框，不像 tactical token。它把注意力拉去看插畫細節，而不是兵種識別。

#### 重畫方向

1. **改成 emblem-first**：每個 badge 必須以單一中心兵種語彙為主，例如騎兵是槍騎 / 馬首 / 衝鋒弧線，盾兵是盾面 / 壁陣，弓兵是弓弦 / 箭束，步兵是刀兵 / 陣列，長槍兵是槍尖 / 槍列。
2. **先做同一個 badge family 骨架**：固定容器、固定外框厚度、固定 underlay 材質、固定光影方向，再替換內部 glyph。不要五張各自構圖。
3. **縮尺寸導向**：以實際使用尺寸 `32×32` 與 `48×48` 可讀為第一原則，1024 只是母圖，不是拿來講故事的畫布。
4. **減少人物敘事**：最多只允許高度概括的單體 silhouette 或兵種輪廓，不可再出現完整人物站姿、群像衝鋒或像卡圖的半身構圖。
5. **色彩節制**：這組 badge 應該是 faction-neutral 的 tiger-tally 系統件，只允許用少量 accent 區分兵種，不准每張變成不同世界觀。
6. **普通兵種不得發亮**：標準 cavalry / infantry / archer / shield / pike badge 不得預設使用明顯發光、能量環或高階 aura；這些強視覺權重必須保留給未來特殊部隊、精銳卡或高 rarity 版本。
7. **先看整卡再定 badge**：badge 不應只在獨立圓章上判定；至少要補一張「完整 tiger tally battle card mockup」來檢查 badge 放進卡面後，是否與 card body、rarity frame、cost plate 與資訊層級相容。
8. **名將來源只能做 provenance 強化**：特定名將遺留虎符允許補小頭像與家族小徽記，但這兩者只能做次層 provenance 模組；玩家第一眼仍要先讀到兵種、稀有度與戰場功能，而不是角色收藏性。

#### 交付要求（重畫版）

1. 先提一張 `badge family board`：同框體下的 5 個黑白 silhouette 版。
2. Agent2 確認 silhouette / hierarchy 過關後，再上材質與色彩。
3. 最終輸出仍提供 1024 母圖，但必須附 `32px` 與 `48px` 縮小檢視圖，證明可讀性成立。
4. 舊版 5 個 badge 視為無效資產，已要求刪除，不可混在正式交付目錄內。
5. 另補一版 `named-general legacy tally` mockup，展示名將小頭像與家族小徽記（例如關氏的小 `關` 字印記）如何掛進卡片而不搶主視覺。
