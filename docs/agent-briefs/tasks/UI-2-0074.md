# [UI-2-0074] UIConfig UIID 補全 + UIManager 動態載入架構設計

## 開單原因

`UIConfig.ts` 的 `UIID` enum 只有 6 個入口（全部是戰鬥 UI），而專案已有完整 JSON 契約的畫面超過 22 個。這代表 LobbyMain、ShopMain、Gacha、GeneralDetail、GeneralList 等畫面**完全繞過** UIManager 的 6-Layer 系統，各自決定顯示/隱藏行為，沒有層級管理、遮罩控制、堆疊語義或統一 lifecycle。

此外，`UIManager.open()` 完全依賴 `register()` 預先掛載場景節點，缺少動態 Prefab 載入路徑，`UIConfigEntry.prefab?` 和 `bundle?` 欄位已預留但實作為零（M-2 TODO 標記）。

## 完整描述

### 第一階段：補全 UIID（可立即執行）

1. 在 `UIConfig.ts` 的 `UIID` enum 補入以下 16+ 個入口：
   - LobbyMain, ShopMain, Gacha
   - GeneralDetail, GeneralList, GeneralPortrait, GeneralQuickView
   - DuelChallenge, UnitInfoPanel, ActionCommandPanel
   - TigerTallyPanel, BattleHUD（已有但不在 UI 層），
   - UltimateSelectPopup, ResultPopup（已有，確認層別）
2. 為每個 UIID 補上對應的 `UIConfig` entry（指定 LayerType，加上 prefab 路徑佔位符）。

### 第二階段：UIManager 動態載入（設計 + PoC）

3. 在 `UIManager.open()` 加入動態載入分支：
   ```typescript
   if (entry.prefab) {
       const prefab = await services().resource.loadPrefab(entry.prefab, entry.bundle);
       node = instantiate(prefab);
   } else {
       node = this._registry.get(uiId); // 舊路徑向後相容
   }
   ```
4. 在 LoadingScene 或 BattleScene 保留 `register()` 路徑，不破壞現有流程。

## 如何驗證

1. `UIConfig.ts` 的 `UIID` enum 包含全部 22+ 個畫面入口
2. `UIConfig` record 每個 UIID 都有對應的 `UIConfigEntry`
3. `UIManager.open(UIID.LobbyMain)` 可在有 prefab 路徑時動態實例化
4. 現有的 `BattleHUD`、`DeployPanel`、`BattleLogPanel`、`ResultPopup`、`Toast`、`SystemAlert` 行為不受影響
5. `run-acceptance.js` 全綠
