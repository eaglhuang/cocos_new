# Demo Playbook
> 第一階段 Demo 的玩法、畫面構圖、操作方式與驗收標準，作為後續 AI / vibe coding 的共同規格依據。

**最後更新**: 2026-03-15

---

## 1. Demo 核心目標
- 驗證 5 路 x 8 格對推戰鬥是否成立
- 驗證大按鈕 UI + 點擊 / 觸控選目標是否順手
- 驗證 2.5D 俯視角下的棋盤、前線與目標可讀性
- 驗證目前架構是否能支撐後續正式版本擴充

---

## 2. 主玩法總覽

### A. 遭遇戰
- 玩家可直接進入完整戰鬥
- 用於快速驗證對推規則、操作手感與勝敗循環
- 遭遇敵將時，敵將固定出現在棋盤右上方並面向我方

### B. 推進模式
- 玩家先沿著道路持續推進，再觸發遭遇事件
- 可能遭遇：敵將、神秘商人、大寶箱、其他 NPC
- 若遭遇敵將，切入遭遇戰版型
- 若遭遇非戰鬥事件，完成互動後再繼續推進

---

## 3. 共通畫面構圖與視覺規則
- 視角採 2.5D 俯視角
- 我方武將固定在畫面左下角
- 左下角需預留額外空間，供未來玩家角色站在武將旁擔任輔助角色
- 左下角不可被主要按鈕 UI 遮擋
- 遭遇敵將時，敵將固定出現在棋盤右上方並面向我方
- 核心戰場為 5 路 x 8 格深度的小棋盤
- 推進模式需預留「棋盤轉化為道路向前移動」的視覺感受
- Demo 雛型優先使用 Cocos Creator 內建 placeholder 資源
- 第一版可接受 `cube`、`sprite`、顏色區分與簡單受擊反饋

---

## 4. 共通戰鬥規則
- 雙方各有一名將軍作為核心目標
- 小兵以格子為單位存在與移動
- 正前方有敵方單位時，優先攻擊正前方目標
- 正前方無敵方單位時，向前移動一格
- 不做複雜斜向移動與高複雜度尋路
- 當前線推進至敵將前方時，可攻擊敵方將軍
- 敵方將軍被擊倒則獲勝，我方將軍被擊倒則失敗

---

## 5. 玩法 A：遭遇戰規格
- 作為最直接的核心戰鬥驗證模式
- 由主入口直接進入
- 棋盤、雙方將軍與基礎單位配置完成後開戰
- 核心驗證：前線推進壓力、阻擋規則直覺性、點選操作順手度、單場重玩意願

---

## 6. 玩法 B：推進模式規格
- 作為中期玩法骨架的前置驗證
- 用於驗證「行軍推進 → 遭遇事件 → 繼續推進」節奏
- 我方武將帶領隊伍沿道路向前推進
- 畫面需表現道路往前移動或場景前進的感覺
- 每次推進後可能觸發一個遭遇事件
- 敵將：切入遭遇戰版型
- 神秘商人：未來商店或補給事件原型
- 大寶箱：未來獎勵事件原型
- 其他 NPC：未來對話、任務、增益或劇情事件原型
- 核心驗證：推進感、遭遇切換自然度、非戰鬥事件與主戰鬥共存性

---

## 7. 玩家操作與 UI 規格
- 以大按鈕 UI 搭配點擊 / 觸控選目標為主
- Web 使用滑鼠點擊，Android / iOS 使用手指觸控
- 主要按鈕區建議放在下方中間或右下方
- 左下方保留給我方武將與未來玩家輔助角色
- 玩家點擊後需有清楚的目標、格子或結果反饋
- 第一版建議操作集合：2 個兵種部署按鈕 + 1 個技能 / 戰術按鈕 + 1 個推進或確認按鈕

### 7.1 兵種按鈕與拖曳部署機制
- **按鈕生成規則**：左側（兵種池）的四顆按鈕會隨機從五個可用兵種中挑選產生，每次刷新時**最多只能有一個兵種重複**。
- **拖曳部署操作**：玩家可以直接拖曳這四顆兵種按鈕到戰場（棋盤）格子上進行部署。
- **部署位置限制**：
  1. **必須是空位**：若目標格子上方已有單位，則顯示警告提示（如 Toast）且無法部署。
  2. **武將前排限制**：只能部署在我方武將前方的**第一排格子**上。

## 12. 遭遇戰補充玩法規格

### 12.1 棋盤與勝利目標
- 核心棋盤維持 5 路 x 8 格，不引入斜向移動與複雜尋路
- 敵我雙方各有一名將軍與一個陣地防禦值
- 玩家勝利條件：敵將生命歸零，或敵方陣地防禦歸零
- 玩家失敗條件：我方將軍生命歸零，或超過回合上限仍未取勝

### 12.13 相機與 UI 構圖細節
- 相機變動固定值以避免後期調整造成畫面不一致；使用以下座標為基準：
  ```
  Position: (-3.80528, 7.598808, -7.911689)
  Rotation (Euler): (-44.111815, -143.244049, 5.5)
  FOV: 30度
  ```
- 此視角使棋盤最低角剛好貼近畫面底部中間，且左半部預留給 UI 承載兵種按鈕與玩家頭像。
- 左側兵種按鈕排列為兩列五行（騎、步、盾、弓、槍），與示意圖一致；兵種按鈕緊靠左邊緣，距離邊界約 60px。
- 左上角空間留作玩家 Profile 使用，同時避免任何 UI 元件遮擋。
- 發動技能按鈕固定放在右下角（距離邊界約 80px），結束回合按鈕置於其左上方。
- 棋盤縮放機制會自動計算 cellSize 和 cellGap，確保格數增多時不超出畫面。
- 這些規則已在 View 層程式碼中實作，場景檔只需保留基本 Canvas/Camera 結構即可。

### 12.2 兵種互剋鏈
- 核心互剋鏈：騎兵 > 步兵 > 盾兵 > 弓兵 > 長槍兵 > 騎兵
- 互剋時傷害倍率建議為 1.3，被剋時建議為 0.7，未互剋時為 1.0
- Demo MVP 第一階段先落地 3 種核心兵種：騎兵、步兵、盾兵
- 弓兵、長槍兵保留在資料與公式層，待第二階段接入

### 12.3 特殊兵種定位
- 工兵：可穿越友軍，進入敵方陣地後持續削減城池防禦
- 水軍：在河流格取得顯著攻防加成，其餘地形僅維持基準值
- 醫護兵：不作為主要輸出單位，戰鬥後對鄰近友軍或武將進行治療

### 12.11 單挑 AI 與對稱處理
- 判定發起：任一方武將推進到敵方前排即可發動單挑邀請
- 防守方拒絕判定：算分公式 `0.45×血量比+0.35×兵力比+0.2×戰力比`，分數 ≥0.58 則接受，可微抖動讓決策不死板
- 拒絕懲罰：拒絕方全軍攻防與武將HP減半，後續部署小兵同樣打折

- 地形類型：河流、山地、平原、城池、沙漠、森林

- 特殊兵種地形優勢需小於兵種互剋，避免地形壓過核心決策

### 12.5 回合流程
1. 玩家部署階段：消耗糧草，將單位放入我方部署區
2. 單位部署限制：敵我雙方每回合最多部署 1 隻小兵；部署合法性由糧草、起始格占用與兵種冷卻共同決定
2. 自動換回合：玩家成功部署後會先等待 2 秒（讓小兵部署動畫完成），再進入回合推進（敵方部署 → 自動移動 → 戰鬥 → 特殊）。
   - **敵軍回合會顯示「敵軍思考中...」面板 2 秒**，之後才真正開始敵方行動。
3. 手動結束回合：若玩家本回合選擇不部署，可按「結束回合」直接推進（同樣會經過 2 秒敵軍思考）。
5. 回合結算階段：檢查勝敗、更新糧草與兵種冷卻、切入下一回合

### 12.6 操作回饋與提示
- 兵種部署失敗會即時顯示 Toast（糧草不足 / 起始格被佔用 / 本回合已部署 / 兵種冷卻中）
- 玩家成功部署後會顯示 Toast 並自動切入下一回合
- 手動按下「結束回合」也會顯示 Toast，避免操作是否生效的疑慮
- 部署面板與 HUD 的糧草顯示保持同步更新

### 12.7 戰鬥紀錄小面板
- 右側新增「戰鬥紀錄小面板」，以滾動文字顯示近期戰況
- 顯示項目：回合更新、敵我部署、單位移動、單位陣亡、戰鬥結束
- 面板僅保留最近 N 筆（預設 10 筆），避免訊息淹沒主畫面
- 用途定位為可讀性與測試輔助，不取代正式戰鬥演出

### 12.8 傷害與治療公式
- 傷害建議公式：`max(1, floor((攻擊 × 互剋係數 × 地形攻擊係數 × 武將加成) - (防禦 × 地形防禦係數)))`
- 治療建議公式：`max(固定值, 目標最大生命 × 治療比例)`
- Demo 版本應優先保留公式集中管理，避免公式散落在單位腳本與 UI 腳本中

### 12.9 兵種進階特性 (Demo)
- **騎兵**：擁有高機動力，一回合內最多可移動 2 格（若前方被阻擋則停下）。
- **弓兵**：具備遠程打擊能力，可攻擊正前方距離 2 格內的敵人。
- **盾兵**：專精防禦，當正前方遭遇敵人阻擋時，會觸發「原地盾牆」狀態，當前防禦力加倍。

### 12.10 武將技能與能量 (SP) 系統
- 雙方武將皆擁有獨立的能量槽（SP）。
- 能量獲取：我方或敵方單位被消滅時，對應的武將可收集到能量（每擊殺 +20 SP）。
- 技能發動：當武將能量滿管時，可發動強大的武將技能，為戰局帶來逆轉機會。

### 12.12 狀態效果（Buff / Debuff）系統
- **暈眩（Stun）**：受暈眩的單位在持續回合內跳過移動與攻擊階段，且盾牆效果失效。
- 狀態效果由 `BuffSystem` 統一管理，以 `unitId` 為 key，支援多個效果同時存在。
- 每回合結束（`advanceTurn` → `tickBuff()`）時遞減剩餘回合數，歸零後自動移除。
- 單位陣亡時，所有附加在該單位上的狀態效果一併清除（避免 ID 復用污染）。

### 12.14 戰場場景攝影機與 UI 規格（左下 → 右上 2.5D 構圖）

本節以參考圖為基準，並套用本專案的 `5x8` 棋盤；重點是讓棋盤在畫面中呈現「左下往右上延伸」的透視方向。

#### 12.14.1 視角方向定義
- 螢幕方向：玩家前線位於偏左下，敵方前線位於偏右上。
- 相機偏航（Yaw）必須為負值，建議區間：`-25° ~ -40°`。
- 相機俯角（Pitch）建議區間：`-45° ~ -60°`。
- 基準參數（第一版）：`Pitch = -52°`、`Yaw = -32°`、`FOV = 42°`。

#### 12.14.2 棋盤尺度與場景佔比
- 可調棋盤維度：`lanes x depth`（目前預設 `5 x 8`）。
- 格子尺寸：`cellSize`（預設 `1.0` World Unit）。
- 格子間距：`cellGap`（預設 `0.1` World Unit）。
- 棋盤實際寬度：`boardWidth = lanes * cellSize + (lanes - 1) * cellGap`。
- 棋盤實際深度：`boardDepth = depth * cellSize + (depth - 1) * cellGap`。
- 視覺佔比目標：
  - 棋盤可玩區約佔畫面寬度 `45% ~ 58%`（中間偏右）。
  - 棋盤可玩區約佔畫面高度 `48% ~ 62%`（中下到中上）。
  - 街景/背景保留約 `20% ~ 30%` 作為環境層次。

#### 12.14.3 相機位置公式（不可寫死）
以棋盤中心 `center` 與棋盤對角線自動推算相機位置：

- 對角線：`diagonal = sqrt(boardWidth^2 + boardDepth^2)`
- 距離：`distance = diagonal * cameraDistanceFactor`（建議 `1.10 ~ 1.30`，預設 `1.18`）
- 設 `yawRad = yaw * PI / 180`、`pitchRad = pitch * PI / 180`
- 相機座標：
  - `camX = center.x - sin(yawRad) * cos(pitchRad) * distance`
  - `camY = center.y - sin(pitchRad) * distance`
  - `camZ = center.z - cos(yawRad) * cos(pitchRad) * distance`

此公式可在棋盤尺寸變更時自動維持同等構圖密度。

#### 12.14.4 單位與頭頂數值 UI 規格
- 敵我單位的地面占格比例統一：`0.78 ~ 0.85` 格寬。
- 頭頂浮動數值錨點高度：`headY + 0.9 ~ 1.2`（依角色模型高度微調）。

### 12.15 2026-03-11 玩法更新（已實作）

#### 12.15.1 部署操作簡化
- 已移除底部 `路1 ~ 路5` 五顆路線按鈕，避免重複操作入口。
- 現在統一為：選兵種後，直接點擊棋盤我方部署列（depth=0）完成部署。

#### 12.15.2 戰鬥演出節奏
- 小兵與主將受傷皆使用頭頂飄字。
- 傷害飄字規格：上飄 + 淡出 + 漸增至 3 倍，總時長 4 秒。
- 戰鬥演出改為逐筆播放：每筆受擊事件間隔 0.5 秒，避免同幀全部爆發。
- 小兵移動動畫統一定為 2 秒，避免節奏過快造成視覺衝擊。
- 玩家的部署行為會先等待 2 秒（讓部署動畫完成），再切換到敵方回合；敵方回合會顯示「敵軍思考中...」面板 2 秒作為節奏緩衝。

#### 12.15.3 SP 吸收演出
- 擊殺後 `+SP` 數字改為「先快後慢」飛向主將 HUD 的動線。
- 飛行期間數字會逐步放大，最大 3 倍，再消失。

#### 12.15.4 隨機地板 Buff 系統
- 每回合在棋盤隨機生成 `1~3` 個地板 Buff（開場回合也會生成）。
- Buff 因子範圍：`2~5`。
- Buff 生成限制：不能生成在雙方主將前方的第一排格子（即我方部署的 depth=0，以及敵方部署的 depth=7），且不能生成在目前有單位佔據的格子上。
- Buff 類型：
  - 攻擊 `x a`
  - 生命 `x b`
  - 攻擊 `/ a`
  - 生命 `/ b`
- Buff 會顯示在格子中央文字；單位踩到後該格 Buff 消失，獲得該 Buff 的增減益。
- 如果本回合沒有任何 Buff 被吃掉，下一回合不會再新增新的 Buff（避免場上同時堆疊太多）。
- 吃 Buff 後會彈出綠色數值飄字（上飄 + 漸增 + 淡出），敵我皆可觸發。

#### 12.15.5 Buff 疊加與可讀性
- Buff 效果會疊加在單位身上，不會因回合結束而清空。
- 小兵頭頂數值改為顯示基礎值差異：例如 `35(+10)`。
- 括號內數值代表相對原始能力的增減量。

#### 12.15.6 數值可維護性
- 地板 Buff 規則已改為資料驅動，設定檔：`assets/resources/data/tile-buffs.json`。
- 可直接調整：每回合生成數量、因子範圍、文字模板、規則池。
- 後續可透過 vibe coding 快速調平衡，不需修改核心戰鬥程式。
- 頭頂數值區塊寬高比建議：`3.2 : 1`。
- 螢幕像素基準（1080p）：約 `96~120px x 28~36px`。
- 顯示優先順序：
  - 第 1 層：HP / 護盾
  - 第 2 層：攻擊 / 防禦圖示
  - 第 3 層：特殊狀態（暈眩、增益）

#### 12.14.5 介面布局與不遮擋原則
- 左側英雄卡列保留寬度：螢幕寬 `14% ~ 18%`。
- 右側戰鬥控制（自動、倍速）保留寬度：螢幕寬 `8% ~ 12%`。
- 右下主操作按鈕直徑：螢幕短邊 `11% ~ 14%`。
- 底部部署列高度：螢幕高 `10% ~ 14%`。
- 棋盤中心應避開左側卡列與右下主按鈕投影區，避免點擊衝突。

#### 12.14.6 Unity 對照理解
- Cocos `Camera + Node 旋轉` 對應 Unity 的 `Camera Transform`。
- `Yaw / Pitch` 調整邏輯與 Unity 相同，差異在於 Cocos 的 Component 綁定流程。
- 本專案的 `BoardRenderer` 僅負責顯示（相當於 Unity 的 View/Presenter），戰鬥規則仍留在 `BattleController`（相當於純 C# Domain Logic）。

### 12.15 武將單挑系統

#### 12.15.1 觸發條件
- 敵我雙方各自的武將（GeneralUnit）皆配置於棋盤外（場邊）。
- 當**玩家方武將正前方那一整排格子**（depth = 0）完全沒有己方小兵佔據時，「武將單挑」按鈕可用。
- 玩家點擊右下角「武將單挑」按鈕後，進入**單挑佈陣模式**：
  - 玩家須點擊棋盤上任意一個**空格**來放置武將。
  - 若點擊的格子已有單位佔據，顯示提示訊息：「請找空格」。
  - 若武將已陣亡（`currentHp <= 0`）或已在場上，則無法再次發動單挑。

#### 12.15.2 武將化身小兵（General-as-Troop）
- 武將上場後，以 `TroopUnit` 的形式存在於棋盤上，其基礎數值如下：
  - **HP** = 武將的 `currentHp`
  - **攻擊力** = 武將 `maxHp × 8%`（可調參數）
  - **防禦力** = 30（固定值，可調）
  - **移動範圍** = 2 格（一次可跑兩格）
  - **攻擊範圍** = 1 格（含正前方與**斜對角**，使用 Chebyshev 距離判定）
- 武將小兵的死亡會同步回寫至 `GeneralUnit.currentHp = 0`，觸發正常的勝負判定。

#### 12.15.3 全軍增益與敵軍仇恨
- **己方增益**：武將上場後，**所有己方小兵的攻擊力翻倍**（`attackBonus += 基礎攻擊力`）。
  - 此增益也適用於武將上場後新部署的小兵。
- **敵軍仇恨**：武將上場後，**所有敵方小兵優先攻擊武將**。
  - 判定方式：若敵方小兵的攻擊範圍能觸及武將（Chebyshev 距離 ≤ attackRange），則強制將武將設為攻擊目標。
  - 若武將不在攻擊範圍內，敵方小兵照常攻擊距離最近的敵人。

#### 12.15.4 武將推進與單挑挑戰
- 每回合戰鬥結算時，武將小兵會嘗試攻擊**前方、左前、右前**的相鄰敵方單位（Chebyshev 距離 = 1）。
- 武將推進到棋盤上**敵方武將正前方**（depth = `FRONT_DEPTH[敵方陣營]`）時，觸發**單挑挑戰事件**：
  - 系統向敵方武將發出挑戰。
  - 敵方可選擇**接受**或**拒絕**（目前 AI 為 50% 隨機，未來可接入策略系統）。

#### 12.15.5 單挑結果
- **接受單挑**：
  - 進入對決模式（目前版本：雙方武將直接對戰，未來可擴展為演出動畫）。
  - 勝者武將留在場上繼續作戰，敗者武將陣亡。
- **拒絕單挑**：
  - **拒絕方受到懲罰**：該陣營所有現存小兵 + 武將本身的**攻擊力與生命值均減半**。
  - 此懲罰也適用於拒絕後新部署的小兵（`duelRejectedFaction` 標記持續生效至戰鬥結束）。

#### 12.15.6 武將陣亡
- 武將小兵在棋盤上被擊殺時：
  - 同步 `GeneralUnit.currentHp = 0`。
  - 發送 `GeneralDamaged` 事件，觸發正常的戰鬥結束判定。
  - 清除 `generalUnitId` 標記，該方武將無法再次上場。

#### 12.15.7 UI 按鈕配置
- 右下角操作區共 4 顆按鈕，由上到下依序為：
  1. **技能釋放**（`btnSkill`）— 武將主動技能
  2. **武將單挑**（`btnDuel`）— 發動單挑佈陣
  3. **計謀策略**（`btnTactics`）— 軍師計謀（開發中）
  4. **回合結束**（`btnEndTurn`）— 結束當前回合
- 按鈕尺寸統一為 200×70，間距 14px，X 座標 = 810。

#### 12.15.8 技術實作摘要
- **核心控制**：`BattleController` 新增 4 個狀態欄位追蹤單挑流程（`playerGeneralUnitId`、`enemyGeneralUnitId`、`isWaitingDuelPlacement`、`duelRejectedFaction`）。
- **事件驅動**：透過 5 個新事件（`GeneralDuelStart`、`GeneralDuelChallenge`、`GeneralDuelAccepted`、`GeneralDuelRejected`、`DuelPenaltyApplied`）解耦 UI 與邏輯。
- **場景層**：`BattleScene` 負責接收 UI 事件、呼叫 Controller、處理觸控佈陣、執行 AI 決策。
- **鄰接判定**：使用 Chebyshev 距離 `Math.max(|Δlane|, |Δdepth|)` 判定斜對角攻擊範圍。
- **Unity 對照**：此系統類似 Unity 中的「英雄下場」機制 —— GeneralUnit 相當於 ScriptableObject 資料，TroopUnit 相當於場上 MonoBehaviour 實體，兩者透過 ID 同步狀態。
