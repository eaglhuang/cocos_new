# 熱更新與版本控制規格書

> **文件目的**: 定義 Cocos Creator 跨平台 (Web, Android, iOS) 的存檔版本遷移 (Data Migration) 以及資源與腳本熱更新 (Hot Update) 的統一工作流。

---

## 一、版本號定義準則 (Semantic Versioning)

專案嚴格遵守 `Major.Minor.Patch` 的版號規則，並分為「App 端主程式版號」與「Server 快取內容版號」。

1. **Major (主版號)** 
   - 觸發條件：引擎升級 (Cocos 3.8.8 -> 3.9)、底層原生 SDK (支付、分析套件) 擴增、需要變更底層權限。
   - **行為：無法熱更。玩家開啟遊戲會被強迫中斷，跳出導向提示：「請前往 Google Play / App Store 下載最新版本」。**
2. **Minor (次版號 / 賽季版號)** 
   - 觸發條件：新增大型系統、新 UI 介面模組、新增大規模新武將與特效資源。
   - 行為：可支援熱更新。玩家會在登入進入 `LoadingScene` 前看到「有新的資料包正在下載 (25MB/100MB)」的進度條。
3. **Patch (修補版號)** 
   - 觸發條件：數值平衡微調 (修改 Config JSON)、修復 TypeScript 邏輯 Bug、替換小型 UI 圖示。
   - 行為：無感微熱更。進入遊戲時幾乎在 1 秒內極速驗證完成。

---

## 二、Data Schema 存檔遷移 (Data Migration)

由於我們的 Data Schema 已經轉型為全 JSON 架構，所有使用者的存檔檔案（或伺服器的快照）內必定帶有 `"Save_Version": "1.0.0"` 欄位。這與熱更能產生強大的相輔相成效果。

### 1. 版本比對與本機遷移流程
1. 玩家開啟遊戲並登入成功。
2. Client 向 Server 請求目前最新的 `Server_Schema_Version`（例如 `1.2.0`）。
3. 本機讀取存檔時，發現 JSON 的 `Save_Version` 為 `1.0.0`，版本過舊。
4. **觸發 MigrationService（升級程式）**：
   - 依序向上執行 `Migration_1_1_0()`: 例如將所有裝備資料的巢狀結構拍平。
   - 依序向上執行 `Migration_1_2_0()`: 補足新增系統（如「寵物系統」）所需的空陣列或預設 `0` 數值。
5. 全面轉換完成後，將存檔的 `Save_Version` 標記為最新的 `1.2.0`。
6. 後續離線遊玩與向上同步 (Online Sync)，皆以此新結構傳輸。

> **防禦性設計準則 (Defensive Design)**：絕不允許 Server 向下相容處理髒資料。舊資料的整容清洗、修補，一律由 Client 端下載含 Migration 腳本的新程式碼後，由 Client 自行升級後再提交給 Server，卸除所有雲端額外算力負擔與條件判斷。

## 三、跨平台熱更新差異化策略 (Hot Update Pipelines)

Cocos Creator 針對不同終端發布平台，底層有著截然不同的資源讀取控制權。雖然我們使用同一份 TypeScript 專案，但必須在自動化部署 (DevOps) 流水分別對待：

### 1. Web 端 (H5 / 瀏覽器) —— 檔名雜湊 (File Hash) + CDN Cache
*   **熱更本質**：瀏覽器沙盒不允許引擎直接寫入設備系統原生磁碟，熱更新純粹是依賴**重新下載**與覆蓋**瀏覽器快取 (Network Cache)**。
*   **做法**：
    1. 在 Cocos 打包建置面板開啟 `MD5 Cache`。
    2. 輸出的所有腳本與圖片會獲得 Hash 後綴名稱（如 `main.1a2b3c.js`、`bg_lobby.9f8d7e.png`）。
    3. 將產出目錄發布至 CDN 空間，並將這些資源設定無限期快取 (Cache-Control: max-age=31536000)。
    4. **只有入口頁 `index.html` 強制設為 `no-cache`**。
*   **更新流程優勢**：當研發團隊發布更新，`index.html` 內引用的 JS/JSON Hash 自動改變了，玩家重新整理網頁時，瀏覽器會直接抓取新的 JS 檔，舊的沒改過的圖片依然從快取極速載入，實現所謂的「無感熱更」。

### 2. Native 端 (Android / iOS) —— Bundle Manifest 比對
*   **熱更本質**：iOS 與 Android 原生 App 可以賦予引擎寫入 App 沙盒底下的 `Writable Path` 能力。Cocos 的檔案讀取會永遠優先去尋找 `Writable Path` 是否有新檔，若無才去載入包體 (APK/IPA) 裡的舊檔。
*   **做法 (運用 AssetsManager)**：
    1. 將遊戲專案徹底規劃為獨立的 **Asset Bundle**（例如：`main`, `resources`, `lobby_ui`, `battle_assets`）。
    2. 開啟熱更外掛，發布前 Cocos 會產出記錄所有檔案 MD5 值的 `project.manifest` 與主版本 `version.manifest`。
*   **更新流程**：
    1. 遊戲 App 啟動時最先載入一個容量極小、乾淨不掛載大量貼圖的 `HotUpdateScene`。
    2. 透過 Cocos 提供的 C++ 模組 `AssetsManager` 向 Server 取得 `version.manifest` 比對。
    3. 若發現新版號，則下載遠端的 `project.manifest`，並與本機 manifest 計算差異檔案清單 (Diff List)。
    4. 平行下載差異檔案寫入 `Writable Path`。
    5. 下載成功後，執行 `cc.game.restart()` 強制重啟並清空 V8 / JSC 虛擬機堆疊，自動引入熱更後的新代碼與新圖庫。

### 開發者核心警語
為實踐上述完美熱更，程式與美術在開發時必須遵守最高守則：
**`ResourceLoader` 與 `SceneManager` 必須強制利用 `assetManager.loadBundle` 動態載入。絕不允許在啟動場景 (Start Scene) 內，靜態拖曳參照後期的核心戰鬥場景資源，否則該龐大資源將被迫綁死入 Main Bundle，破壞掉模組化熱更的設計初衷！**
