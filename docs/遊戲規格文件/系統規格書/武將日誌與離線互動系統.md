# 武將日誌與離線互動系統規格書

> **來源文件**：打造三國生活感武將系統.md (doc_spec_0102)、結合玩家日誌的動態敘事.md (doc_spec_0112)、離線互動與武將人性化設計.md (doc_spec_0128)
> **定位**：承接武將日誌、晨報摘要、派遣整備、離線互動事件與離線戰鬥快照的正式母規格。

---

## A. 系統描述

武將日誌與離線互動系統負責把玩家「不在場」時期的世界狀態，轉譯成可追溯、可閱讀、可驗證的敘事層。它不是單純的留存包裝，也不是另一套獨立大廳玩法，而是把大廳、武將、派遣與戰場之間的長線記錄正式收斂成同一個資料模組：

1. `武將日誌`：把玩家的重要行為、武將互動與系統事件整理成可回看的時間軸。
2. `晨報摘要`：每次登入時，把前一段離線期間最值得知道的 1~3 則事件整理成短摘要。
3. `派遣整備`：讓未上場武將可在玩家離線或忙於其他模式時執行低風險任務。
4. `離線互動事件`：用武將個性、關係與前日行為，生成輕量但有記憶點的角色互動。
5. `離線戰鬥快照`：在自動解決、代勞支援或離線戰果結算時，保留可驗證的摘要快照，而不是只回傳裸數字。

正式邊界如下：

1. 本系統是**敘事與狀態承接層**，不覆寫留存系統的商業節奏，也不覆寫大廳系統的場景配置權威。
2. 晨報、日誌、派遣的 UI 入口由大廳與人物頁承接，但本輪不強制指定為獨立 fullscreen screen。
3. 任何離線獎勵、離線戰果與代勞結果，都必須能對應到 `Action_Records` 或摘要快照，不能只在 UI 端硬生結果。

> **Unity 對照**：可視為 `Quest Log + Daily Digest + Idle Dispatch Manager + Event Bus Archive` 的整合層。`武將日誌` 類似 Timeline Log，`晨報` 類似登入後的 Summary Overlay，`離線戰鬥快照` 類似可重播但先以摘要形式保存的 Battle Report DTO。

## B. 系統目的

| 目的 | 說明 |
|---|---|
| 生活感承接 | 讓武將在玩家離線時仍有存在感，而不是回到死板的純選單狀態 |
| 長線記錄 | 將日常事件、派遣與互動整理成可查詢的武將日誌 |
| 登入摘要 | 用晨報快速告知玩家「昨晚發生了什麼」，降低資訊遺漏 |
| 低風險代勞 | 讓閒置武將能處理低風險任務，但不取代玩家主動決策 |
| 可驗證離線結果 | 任何離線收益或離線戰果都必須留下摘要快照與驗證鉤點 |

## F. 公式相關

### F-1. 晨報摘要生成

```typescript
function buildMorningReport(logs: JournalEvent[], loginState: LoginState): JournalEvent[] {
    const candidates = logs
        .filter(log => isWithinPreviousWindow(log.timestamp, loginState.lastLoginAt))
        .filter(log => log.visibility !== 'HIDDEN')
        .sort((left, right) => right.priority - left.priority);

    const slotCount = clamp(loginState.officerTier >= 4 ? 3 : 2, 1, 3);

    return weightedPick(candidates, slotCount, event => {
        return event.priority * event.freshnessWeight * event.characterAffinityWeight;
    });
}
```

1. 晨報只負責「摘要排序」，不重新計算事件真相。
2. 事件真相以武將日誌與對應 Action / Snapshot 為權威。
3. 同一時間窗內，重複性低價值事件應被折疊為同類摘要，不應洗版。

### F-2. 派遣品質評分

```typescript
function getDispatchScore(job: DispatchJob, general: GeneralSnapshot): number {
    const roleFit = getRoleFit(job.tags, general.roleTags) * 45;
    const statFit = getStatFit(job.requiredStats, general.stats) * 25;
    const moodFit = getMoodModifier(general.mood) * 15;
    const journalFit = getJournalAffinity(job.storyTags, general.journalTags) * 15;
    return Math.round(roleFit + statFit + moodFit + journalFit);
}
```

1. `DispatchScore` 只決定結果品質區間與日誌表現，不直接覆寫任務主獎勵權威。
2. 本系統不使用固定「派遣成功率 100% / 失敗 0%」二元邏輯；正式採品質區間 `POOR / NORMAL / GOOD / EXCELLENT`。
3. 高風險或會改變大局的任務不得被派遣系統自動代勞，仍需玩家親自決策。

### F-3. 離線互動事件觸發

```typescript
function shouldTriggerOfflineInteraction(context: OfflineContext): boolean {
    return context.offlineHours >= 4
        && context.unreadMorningReport < 3
        && context.activeConflictTags.length > 0
        && !context.cooldowns.includes('OFFLINE_INTERACTION_LOCK');
}
```

1. 離線互動事件以「角色存在感」為主，不應取代正式任務結算。
2. 同一對角色在同一登入窗內，預設只保留 1 則主互動，避免重複刷屏。

### F-4. 離線戰鬥快照

```typescript
interface OfflineBattleSnapshot {
    snapshotId: string;
    battleType: 'AUTO_RESOLVE' | 'ASSIST_SUPPORT' | 'OFFLINE_SKIRMISH';
    seed: string;
    lineupDigest: string;
    tileStateDigest: string;
    tacticDigest: string;
    resultDigest: string;
    syncHashLink: string;
}
```

1. 離線戰鬥快照只保存摘要與驗證鉤點，不保存完整逐格 replay。
2. `syncHashLink` 必須可回扣到 `Action_Records` / `Last_Sync_Hash` 鏈，避免離線結果無從驗證。
3. 主戰場的即時戰鬥規則仍以 `戰場部署系統.md` (doc_spec_0040) 與 `戰法場景規格書.md` (doc_spec_0039) 為權威，本系統只負責摘要與回看入口。

## K. 智慧 NPC 對話與 AI 架構（二階段升級路徑）

> **決策來源**：整併疑問書 Q75, Q76, Q77

為了在不大幅增加 MVP 開發壓力的前提下，實現具備「長期記憶與性格」的武將互動，系統採購以下二階段演進路徑：

### K-1. 第一階段 (MVP)：靜態模板佔位
- **實作方式**：沿用本規格書第 16 點定義的「靜態離線事件模板」。
- **對話生成**：基於 JSON 語法樹的插值與排列組合（Slot-filling），無須 GPU 推論。
- **記憶存儲**：僅保留有限的布林標籤（例如：`Has_Met_Before: true`）。

### K-2. 第二階段 (升級)：Utility AI + LLM 混合架構
- **核心架構**：**Utility AI 做骨架（行為決策）+ LLM 做皮膚（語言修飾）**。
- **模型選型**：採用小型語言模型（< 1.5B），如 **Qwen2.5-1.5B** 或 **Phi-3-mini**。
- **推論方式**：推薦使用 Ollama 封裝 REST API，供 Cocos 端透過 fetch 介接。
- **向量記憶 (RAG)**：
    - **存儲方案**：採用 **SQLite-vec**（Cocos Native 整合）。
    - **同步策略**：與現有 Data Schema 同一資料庫，確保 Web/Mobile/PC 存檔與記憶的一致性。
    - **邏輯**：Utility AI 決定「要做什麼（如：搶糧、送禮）」，LLM 根據向量庫檢索出的「歷史往事」生成具備人性的對話文本。

## 三國日報 / 每日實錄系統 (Roguelike 動態敘事)
- **核心機制**：透過 Log Parser 分析玩家前 24 小時的事件（慘敗、抽卡、缺糧）。
- **互動表現**：將玩家遭遇拼接入「武將情緒模板」，例如張飛見玩家缺糧，自發「趁夜去敵營搶糧」，在每日登入的《三國日報》中匯報，並發放實質獎勵（15%糧草）。
- **全服快照**：可插入伺服器中偶發的高端玩家動向（如十字軍戰報），增添世界脈動感。
