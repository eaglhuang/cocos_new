## A. 系統描述

關卡設計系統定義《3KLife》戰場挑戰的正式入口、戰前偵查、連環破結構、機制鎖定、戰法槽位需求，以及與主戰場視覺層和場景戰法的對接方式。

本檔處理的是「關卡作為玩法與規格的母系統」，不是單一戰法場景的美術細節，也不是單一 UI 畫面的 layout 草稿。

---

## B. 系統目的

1. **建立正式挑戰鏈**：把典韋 → 許褚 → 曹操這類連環破挑戰變成正式規格，而不是散落提案。
2. **反向驅動養成**：透過戰前偵查與機制鎖定，引導玩家定向培養子嗣與戰法組合。
3. **分離數值與機制**：高數值不能完全碾平機制需求，避免戰場變成純戰力檢定。
4. **讓美術可量產**：背景、Decal、VFX、立牌分層，支援多場景疊加與後續擴充。

---

## C. 關卡結構

### C-1. 連環破主鏈

1. 挑戰關卡可由多個子關卡串接而成。
2. 玩家需按正式順序完成前置關卡，方可解鎖後續 Boss 關卡。
3. 每個子關卡都有主題機制、推薦戰法、推薦兵種與戰前提示。

### C-2. 關卡節點欄位

| 欄位 | 說明 |
|---|---|
| Stage_ID | 關卡唯一識別碼 |
| Chain_ID | 所屬連環破主鏈 |
| Boss | 守關主將 |
| Theme | 場景主題，例如落石 / 水淹 / 夜襲 |
| Required_Loadout | 推薦或強制攜帶的戰法與兵種 |
| Preview_Intel | 戰前偵查文字與視覺提示 |

---

## D. 戰前偵查與機制鎖定

### D-1. 戰前偵查

1. 進入關卡前，必須顯示敵方主將、戰場主題、主要風險與推薦應對戰法。
2. 偵查資訊不必完全揭露數值，但至少要揭露核心機制。
3. 偵查畫面應支援「本次陣容是否具備應對要件」的檢核。
4. 正式偵查只揭露敵將名稱、主機制與風險標籤，不揭露精確站位與完整 AI 腳本。

| Preview_Intel 欄位 | 顯示內容 | 正式邊界 |
|---|---|---|
| Enemy_Name | 敵方主將姓名或稱號 | 可揭露 |
| Core_Mechanic | 本關必讀主機制，例如落石、水淹、夜襲 | 可揭露 |
| Risk_Tags | 高風險提示，例如混亂、落單、深水 | 可揭露 |
| Recommended_Loadout | 推薦戰法與兵種關鍵字 | 可揭露 |
| Formation_Preview | 敵方開局站位、精確埋伏點 | 正式隱藏 |

### D-2. 連環破提示與機制鎖定

1. 連環破的前置順序仍屬正式解鎖條件，但每個子關卡的失敗原因要能被摘要成「高風險提示」，而不是只用灰掉按鈕表現。
2. 關卡可指定必備或高建議戰法，例如避水陣、抗混亂、直取敵首。
3. 高數值角色可降低容錯壓力，但不能完全略過核心機制。
4. 連環破不採硬鎖式「沒有某技能就完全不能進場」的唯一答案設計；正式做法是允許進場，但用高風險提示明示缺口。
5. 機制鎖定的正式用意是建立多培養路線，而不是強迫玩家只養單一套路。

### D-3. 戰前偵查 UI 契約

1. 戰前面板至少要有 `Enemy_Name`、`Core_Mechanic`、`Risk_Tags`、`Recommended_Loadout_Check` 四塊資訊。
2. `Recommended_Loadout_Check` 應標示「已具備 / 缺少 / 高風險」三態，不得只用二元鎖定。
3. 連環破鏈中的下一關若存在明顯缺口，UI 應顯示「高風險，仍可挑戰」而非直接鎖死。
4. 偵查與提示資訊只屬預戰 intel，不得在此層直接暴露開局站位解答。

### D-4. 軍師 HUD 正式資訊層

1. 主戰場 HUD 需額外提供軍師可讀的 `Theme_Hazard`、`Interactive_Object_Status`、`Stage_Salvage_Summary`、`Recommended_Loadout_Gap` 四個摘要欄位。
2. `Theme_Hazard` 只顯示當前回合會發生的主題危害，例如落石預警、水位上升、夜襲突入，不預先透出全部觸發點。
3. `Interactive_Object_Status` 用於標示可擊破營帳、落石機關、水門、陷阱等互動物件的存活 / 已觸發 / 已回收狀態。
4. `Recommended_Loadout_Gap` 只提示「缺少抗混亂」、「缺少反隱」這類缺口，不顯示最佳答案腳本。
5. 軍師 HUD 的用途是把場景規則讀清楚，不是把埋伏點、AI 路徑或開局站位答案一次交給玩家。

## E. 戰法槽位與子嗣養成聯動

1. 子嗣在養成過程中可以學會多個戰法，但出征前只能裝備有限槽位。
2. 槽位容量與智力、培育成果與關卡等級掛鉤。
3. 關卡規格必須寫明「推薦至少幾個對策槽位」，避免 UI 與系統規格脫鉤。

### E-1. 正式原則

| 原則 | 內容 |
|---|---|
| 學習數 ≠ 裝備數 | 武將可學會多戰法，但出征時只裝備部分 loadout |
| 智力決定戰術廣度 | 智力越高，可攜帶的戰法槽越多 |
| 關卡看 loadout，不看收藏量 | 真正進場檢核的是當前攜帶組合，而不是帳號總持有量 |

## F. 特殊戰場視覺分層

### F-1. 四層架構

1. 底層：高品質靜態背景
2. 中層：棋盤與地表 Decal
3. 上層：立牌、物件與 VFX
4. 最上層：HUD、傷害數字、模式標記與操作 UI

### F-2. 正式規則

1. 落石、火焰、營帳與陷阱等可變物件，不直接畫死在背景圖。
2. 棋盤格線應與地表融合，不採純浮空 UI 格線。
3. 主將與小兵維持固定比例，避免不同場景失真。
4. 關鍵格可用低強度微光、邊緣熱區或材質亮度差提示，但不得整片覆蓋成發光棋盤。

### F-3. 棋盤提示語言

1. 一般格線優先融入地表裂紋、石縫、草壓痕或戰旗陰影，讓玩家能讀格但不出戲。
2. 危險格、落石預警格、深水格與營帳互動格，才允許使用更明確的微光或 decal 提示。
3. 關鍵格的亮度提升應低於 HUD 主提示層，避免搶走角色與戰法特效焦點。
4. 若戰場主題為夜襲或霧戰，格線仍需維持可讀性，但應透過材質差與局部反光完成，而非回退成傳統白線棋盤。

## G. 典型主題關卡

### G-1. 峽谷落石

1. 關鍵機制：落石格、位移封鎖、石材回收。
2. 推薦應對：高機動、破石或位移類戰法。
3. 可回收物：被擊碎的石障與落石殘材可形成 `Stage_Salvage`，戰後轉為石材 / 防線補修摘要。

### G-2. 水淹圍堵

1. 關鍵機制：水流推移、深水傷害、非水戰單位懲罰。
2. 推薦應對：避水陣、抗混亂、水域適性。
3. 可回收物：水門、堤防與漂流物的控制權可在戰後結算為水路情報與補給回收，不在戰中直接改寫帳號經濟。

### G-3. 夜襲掠奪

1. 關鍵機制：營帳目標、低命中高暴擊、掠奪資源。
2. 推薦應對：偵查、突擊、營帳破壞戰法。
3. 可回收物：營帳、糧箱與軍械點會累積為補給型 `Stage_Salvage`，供戰後摘要判定掠奪成果。

### G-4. 森林埋伏

1. 關鍵機制：視野壓縮、陷阱、隱身突襲。
2. 推薦應對：反隱、偵查、範圍控場。
3. 可回收物：拆除陷阱、繳獲木樁與斥候線索可在戰後轉為木材 / 伏擊情報摘要。

---

## H. 攝影機與畫面構圖

1. 主戰場採固定 2.5D 斜視角，不為單一關卡改成自由鏡頭。
2. 棋盤需維持可讀性與 UI 安全區，不可因美術氣氛壓縮操作資訊。
3. 關卡主題可以改變光色、地表與物件，但不得破壞統一的戰場 HUD 可讀性。


## I. 互動物件回收與戰後摘要

1. 關卡內可互動物件統一歸類為 `Interactive_Object`，至少包含 `Rockfall_Node`、`Floodgate`、`Camp_Target`、`Trap_Cluster` 四種正式大類。
2. 互動物件造成的石材、木材、補給與情報收益，統一先寫入關卡級 `Stage_Salvage`，在戰鬥結束時一次結算，避免戰中 UI 與帳號經濟頻繁抖動。
3. `Stage_Salvage` 只提供戰後摘要與經濟結算依據，不改變「關卡看 loadout / 主題機制、經濟看戰後總結」的職責分層。
4. 關卡規格書需對每種主題戰場至少標示一種主要互動物件與一種主要可回收收益，避免美術做了物件卻沒有正式玩法用途。

---

## J. 正式共識摘要

1. 連環破與戰前偵查正式升格為系統規格，不再只是草案討論。
2. 關卡是「戰法 loadout 檢核器」，不是單純戰力門檻。
3. 棋盤、物件與特效採分層量產，避免背景圖與動態機制互相打架。
4. 主戰場視覺要維持冷色戰術感與清楚 UI，可隨場景切換主題但不能失去可讀性。
5. 戰前偵查正式只揭露敵將名稱與核心機制，不揭露站位；連環破則以高風險提示取代硬鎖唯一解。
6. 軍師 HUD 與 `Stage_Salvage` 正式成為關卡資訊層的一部分，但只提供摘要，不直接暴露解答。

## Boss AI 與特殊挑戰機制
- **親衛隊召喚**：在特定將軍（如：呂布、曹操）作為敵方主將時，當血量下降至 50% 以下，會自動召喚出「親衛隊」以阻擋玩家直接斬首，玩家往往需保留奧義用以清場。
