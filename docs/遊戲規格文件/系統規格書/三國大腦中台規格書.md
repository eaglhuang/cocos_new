---

## A. 系統描述

三國大腦中台是一套負責「把三國文本變成可查、可控、可生成遊戲內容」的資料與推論服務。它的第一優先任務不是先寫 Cocos UI，而是先完成《三國演義》PDF 到 Markdown、資料清洗、章回切片、人物事件抽取、persona card 生成、向量化入庫與 RAG 查詢閉環。

本中台提供四類能力：

1. **文本資料管線**：將《三國演義》PDF 轉成乾淨 Markdown，拆成章回與語意 chunk，抽取人物、事件、地點、人際關係與原文引用。
2. **知識檢索服務**：以 Pinecone 作為研發 / 雲端向量庫，依 `generalId + contextKey + metadata filter` 取回可信背景。
3. **角色生成服務**：結合 persona card、歷史片段、玩家日誌與模型推論，回傳符合武將性格的一句話、日誌反應或模板補丁。
4. **關鍵字互動服務**：由 ETL 管線自動整理與武將高度相關的互動關鍵字，供玩家在人物頁點擊後選詞發話，武將再以其歷史視角與 persona 口吻回應聯想內容。

Unity 對照：本中台相當於 `Narrative Backend + RAG Service + Dialogue Authoring Pipeline`。Cocos 端只呼叫 API 與接收 DTO，不直接承擔 PDF 清洗、向量索引與大型模型推論。

## B. 系統目的

| 目的 | 說明 |
|---|---|
| 建立三國知識底座 | 把 PDF / Markdown / 既有武將資料轉為可檢索資料 |
| 支援武將性格台詞 | 讓張飛、關羽、趙雲等依歷史與關係說出不同內容 |
| 連接人物日誌 | 將玩家行為、晨報、派遣、離線互動納入可查記憶 |
| 支援關鍵字互動 | 玩家可從人物、事件、地點、物品、生物等分類挑選關鍵字與武將對話 |
| 降低模型亂編 | 以 metadata filter、persona card 與 source quote 限制生成 |
| 提供研發練習路線 | 覆蓋 PDF 轉 MD、資料清洗、RAG、FastAPI、Pinecone、Ollama/vLLM |

## C. 技術架構

### C-1. MVP 技術棧

| 層級 | 工具 | 用途 |
|---|---|---|
| PDF 轉換 | Docling；Marker 作備援 | PDF to Markdown，保留章回結構 |
| 清洗與切片 | Python + Pydantic + LangChain text splitters | 清理頁碼、章回切片、schema 驗證 |
| 抽取模型 | Gemini / Vertex AI / 本地 Qwen | 抽取人物、事件、關係與摘要 |
| 向量庫 | Pinecone | 雲端 / 研發向量檢索 |
| Embedding | bge-m3 / text2vec / OpenAI embeddings | 將 chunk 與事件轉向量 |
| API | FastAPI + Uvicorn | 提供 Cocos 呼叫入口 |
| Runtime LLM | Ollama 優先；vLLM 第二階段 | 台詞生成與模板補丁 |
| 本地降級 | SQLite-vec / 模板 JSON | 無網路或低階設備時使用 |

### C-2. 服務分層

```text
Cocos Creator
  -> NpcDialogueService
  -> FastAPI /v1/npc/dialogue
  -> Persona Card Store
  -> Pinecone Retrieval
  -> Ollama / vLLM Inference
  -> Safety / Drift Guard
  -> DialogueResponse DTO
```

正式遊戲不得把 Pinecone 視為唯一真相。Pinecone 是知識與記憶的檢索層；武將 canonical roster、人物頁資料與存檔真相仍以本地 JSON / Data Schema / Server Save 為權威。

source .venv/bin/activate

vLLM 屬第二階段效能挑戰。8GB 顯存條件下必須採 4-bit AWQ / GPTQ 量化，限制 `max_model_len` 與併發數，並優先驗證 KV cache 壓力。

## E. 資料處理流程

### E-1. Phase 0：來源確認

1. 確認《三國演義》PDF 來源可用於研發資料處理。
2. 建立 `source-manifest.json`，記錄書名、版次、來源、授權狀態、轉換工具與 hash。
3. 若有可靠公版純文字，可作為交叉驗證；但本練習流程以 PDF -> Markdown 為主。

### E-2. Phase 1：PDF 轉 Markdown

1. 使用 Docling 轉換 PDF，Marker 作備援。
2. 輸出 `source.md`。
3. 保留章回標題，不保留頁碼、頁首、頁尾與掃描雜訊。
4. 產出 `conversion-report.json`，列出頁數、章回數、可疑 OCR 區段。

### E-3. Phase 2：Markdown 清洗與章回拆分

1. 依「第 X 回」或章回標題切成 `chapters/ch_###.md`。
2. 移除空白重複、頁碼、廣告、註解殘片。
3. 保留 paragraph index 與 source offset，供未來 evidence reference 使用。
4. 產出 `chapters-manifest.json`。

### E-4. Phase 3：武將主名冊、文本稱呼表與正式對照表

1. Phase 3 的資料模型拆三層：
  - `武將主名冊`：來自 `assets/resources/data/generals.json`，只負責定義 canonical `generalId`、主名、陣營與目前已採納的稱呼。它是產品內部的 identity seed，不宣稱自己就是文本原始來源。
  - `文本稱呼表`：記錄文本裡實際觀察到的稱呼字串、`sourceRef`、章回、段落、場景參與者等。這一層才是一手 evidence。
  - `正式對照表`：只收錄經規則或人工審核後，可以安全映射到 `generalId` 的稱呼對照，供後續召回與消歧使用。
2. 目前 `server/npc-brain/pipelines/sanguo-rag/build_alias_dict.py` 先產出 `武將主名冊` 與 `正式對照表` 的 bootstrap 版本；`server/npc-brain/pipelines/sanguo-rag/collect_observed_mentions.py` 再蒐集 `文本稱呼表`，兩者不得混成同一步。
3. 初始正式對照來源分四層：
  - `武將主名冊` 既有欄位：`name`、`alias[]`、`title`
  - `title` 去除 `【】` 等裝飾後可直接採納的稱號詞
  - tracked manual overrides：人工補強字號、稱號與演義稱呼，例如張飛、張翼德、翼德、燕人張翼德
  - 後續拆文流程中觀察到的 `文本稱呼候選`，但不得在同一輪自動升格為正式對照
4. `將軍`、`主公`、`先生`、`縣令`、`太守` 等泛稱預設不是高信心正式對照；它們屬於情境稱呼，必須在後續對話解析階段依場景消歧。
5. `文本稱呼候選` 只能先進 review queue，不得直接覆寫正式對照表；每次版本升級最多做一輪「候選蒐集 -> 人工審核 -> 發佈 vNext」，避免稱呼對照自我擴張成無限循環。
6. 抽取結果不得直接創建 runtime 武將；必須先映射回 canonical `generalId`。

### E-5a. Phase 4a：對話解析與稱呼消歧

每個 chunk 在事件抽取前，先做 deterministic-first 的對話解析與稱呼消歧。

1. 先切分旁白與引號內容，辨識 `speaker`、`addressee`、`mentioned target`。
2. 先用正式對照表做候選召回，再結合前後文中的場景參與者、說話動詞、稱呼位置與文本稱呼表中的已觀察稱呼，解析 `將軍`、`主公`、`先生` 等泛稱的實際指向。
3. 若引號中未直接出現人名，但旁白已指明對話雙方，仍應把泛稱解析為對應 `generalId`；若證據不足，保留 `unresolved address title`，不得硬綁人物。
4. 對話解析階段可同時抽出 `item`、`location`、`creature` 等 mention，但這些 mention 與人物的關係，必須在事件抽取階段正式定義。

```json
{
  "sourceRef": "ch_042#p18",
  "sceneParticipants": ["zhang-fei", "liu-bei"],
  "utterances": [
    {
      "speakerLabel": "軍士",
      "speakerGeneralId": null,
      "addresseeLabel": "將軍",
      "addresseeGeneralId": "zhang-fei",
      "resolutionMode": "scene-address-title",
      "text": "將軍，我送您一把寶刀！"
    }
  ],
  "entityMentions": [
    {
      "label": "將軍",
      "entityType": "address-title",
      "resolvedGeneralId": "zhang-fei",
      "confidence": 0.86
    },
    {
      "label": "寶刀",
      "entityType": "item",
      "resolvedItemKey": null,
      "confidence": 0.73
    }
  ]
}
```

### E-5b. Phase 4b：事件抽取

每個 chunk 在完成正式對照召回與對話解析後，再用 LLM 或結構規則做事件抽取。

1. 事件抽取的輸入至少包含：原始 chunk、正式對照候選、scene participants、utterance resolution、entity mentions。
2. 若對話中出現泛稱與物品互動，例如「將軍，我送您一把寶刀」，事件抽取必須把 `寶刀 -> offered_to -> zhang-fei` 這類關係明確寫成 edge，而不是只記錄字面共現。
3. 若泛稱未能在 E-5a 消歧成功，事件抽取只能保留 unresolved participant，不得為了完整事件而強行綁定錯誤 `generalId`。

```json
{
  "eventId": "romance.ch042.zhang-fei.changban-bridge",
  "chapterNo": 42,
  "eventKey": "changban-bridge",
  "generalIds": ["zhang-fei", "liu-bei", "cao-cao"],
  "location": "長坂橋",
  "summary": "張飛據橋斷後，以大喝震懾曹軍。",
  "sourceQuote": "燕人張翼德在此...",
  "relationshipEdges": [
    { "from": "zhang-fei", "to": "liu-bei", "type": "protects" }
  ],
  "moodTags": ["bold", "loyal", "intimidating"],
  "confidence": 0.92,
  "sourceRef": "ch_042#p18"
}
```

### E-6. Phase 5：互動關鍵字抽取與分類

ETL 必須和事件抽取一起產出可供遊戲互動的關鍵字集合，供玩家點擊張飛等武將時挑選。

1. 關鍵字來源以 `generalId` 關聯的事件、關係、章回片段、persona card 錨點為主，不可只靠單一 chunk 臆測。
2. 每個關鍵字必須至少包含：`keywordKey`、`label`、`category`、`generalIds`、`sourceRefs`、`confidence`。
3. MVP 先支援以下分類：
   - `person`：明確人物名、職稱人物、集團人物稱呼，例如張飛、劉備、某縣令、黃巾賊。
   - `event`：重要事件、戰役、盟約、斷後、護送、出征等可成為對話焦點的事件名稱或事件鍵。
   - `location`：城池、橋、關口、郡縣、山川、營寨等與武將經歷明確相關的地點。
   - `item`：武器、裝備、旗幟、車船、器械、文書、酒肉等無生命可見物件。
   - `creature`：馬、牛、犬、鷹等有生命生物，不含泛用抽象比喻。
4. 未來可擴充更多分類，但 category key 必須保持可機器判斷，不得把 UI label 直接當 enum。
5. 去噪原則：只保留可支撐武將聯想或歷史敘事的詞；純背景風景、無敘事功能的路人甲乙、一次性掃描噪音詞不得進正式 keyword pack。
6. 關鍵字不得直接創建 runtime 實體；人物類仍必須先映射回 canonical `generalId` 或標記為 `non-roster person`。
7. 輸出位置建議放在 `artifacts/data-pipeline/sanguo-rag/extracted/keyword-options/`，每位武將可生成一份 `generalId.keywords.json`。

```json
{
  "generalId": "zhang-fei",
  "keywordVersion": "general_keywords_v1",
  "categories": {
    "person": [
      {
        "keywordKey": "liu-bei",
        "label": "劉備",
        "generalIds": ["zhang-fei", "liu-bei"],
        "sourceRefs": ["ch_042#p18"],
        "confidence": 0.96
      }
    ],
    "event": [
      {
        "keywordKey": "changban-bridge",
        "label": "長坂橋斷後",
        "generalIds": ["zhang-fei"],
        "sourceRefs": ["romance.ch042.zhang-fei.changban-bridge"],
        "confidence": 0.94
      }
    ],
    "location": [
      {
        "keywordKey": "changban-bridge-location",
        "label": "長坂橋",
        "generalIds": ["zhang-fei"],
        "sourceRefs": ["ch_042#p18"],
        "confidence": 0.93
      }
    ]
  }
}
```

### E-7. Phase 6：去重、歸納與 Persona Card

1. 合併跨 chunk 的同一事件。
2. 為每位武將生成 persona card。
3. 低信心資料進 `review/low-confidence-events.jsonl`，不入正式向量庫。
4. persona card 必須先人工抽查核心名將，特別是張飛、關羽、趙雲、曹操、劉備、諸葛亮。

### E-8. Phase 7：Embedding 與 Pinecone 入庫

1. `romance_facts_v1`：演義事件與原文片段。
2. `history_facts_v1`：正史或百科摘要，若資料源不同需獨立 namespace。
3. `general_persona_v1`：persona card 與核心錨點。
4. `general_keywords_v1`：互動關鍵字與分類錨點；僅收正式 keyword pack，不收低信心髒詞。
5. `player_memory_dev`：玩家互動摘要，MVP 可先只 dev namespace。

Metadata 必含：`generalIds`、`eventKey`、`chapterNo`、`sourceType`、`faction`、`confidence`、`sourceRef`。

## F. API 規格

### F-1. `GET /health`

回傳中台、Pinecone、LLM backend 狀態。

### F-2. `GET /v1/npc/context-options`

Query：

| 欄位 | 型別 | 說明 |
|---|---|---|
| `generalId` | string | 武將 canonical id |
| `limit` | number | 回傳情境數量 |

Response：

```json
{
  "generalId": "zhang-fei",
  "options": [
    { "contextKey": "changban-bridge", "label": "長坂橋", "sourceType": "romance", "confidence": 0.92 },
    { "contextKey": "taoyuan-oath", "label": "桃園結義", "sourceType": "romance", "confidence": 0.89 }
  ]
}
```

### F-3. `GET /v1/npc/keyword-options`

Query：

| 欄位 | 型別 | 說明 |
|---|---|---|
| `generalId` | string | 武將 canonical id |
| `categories` | string[] | 可選，限制回傳類別 |
| `limitPerCategory` | number | 每類最多回傳幾個關鍵字 |

Response：

```json
{
  "generalId": "zhang-fei",
  "keywordVersion": "general_keywords_v1",
  "categories": {
    "person": [
      { "keywordKey": "liu-bei", "label": "劉備", "confidence": 0.96 },
      { "keywordKey": "huangjin-bandits", "label": "黃巾賊", "confidence": 0.81 }
    ],
    "event": [
      { "keywordKey": "changban-bridge", "label": "長坂橋斷後", "confidence": 0.94 }
    ],
    "location": [
      { "keywordKey": "changban-bridge-location", "label": "長坂橋", "confidence": 0.93 }
    ],
    "item": [
      { "keywordKey": "serpent-spear", "label": "丈八蛇矛", "confidence": 0.95 }
    ],
    "creature": [
      { "keywordKey": "warhorse", "label": "戰馬", "confidence": 0.72 }
    ]
  }
}
```

### F-4. `POST /v1/npc/dialogue`

Request：

```json
{
  "generalId": "zhang-fei",
  "contextKey": "changban-bridge",
  "selectedKeywordKeys": ["liu-bei", "serpent-spear"],
  "playerId": "PLAYER_TEST_01",
  "playerContext": "玩家路過長坂橋",
  "toneMode": "in-character",
  "maxChars": 60
}
```

Response：

```json
{
  "line": "長坂橋頭那聲喝，俺不是為逞勇，是要替大哥斷開曹軍的膽。",
  "emotion": "bold",
  "evidenceRefs": ["romance.ch042.zhang-fei.changban-bridge"],
  "usedFacts": ["張飛據橋斷後", "曹軍追擊劉備"],
  "fallbackUsed": false,
  "model": "qwen2.5:7b-instruct-q4_K_M",
  "latencyMs": 1800
}
```

規則：

1. `contextKey` 與 `selectedKeywordKeys` 可同時存在；若兩者衝突，以 `generalId + selectedKeywordKeys` 的交集 evidence 為優先。
2. 若玩家只選關鍵字、不選事件，中台仍可回應，但必須回扣該關鍵字的 `sourceRefs`。
3. 若玩家選到低信心或已淘汰 keyword，中台須回退到 persona card 的安全聯想，不得硬編重大史實。

### F-5. `POST /v1/npc/dialogue/stream`

第二階段支援。Web Preview 可用 SSE / chunked response；Native 端若 Cocos JSB 不穩定，回退短輪詢或非串流。

## G. Prompt 與生成規則

### G-1. Prompt 組合順序

1. System：三國世界觀、安全規則、不得自稱 AI。
2. Persona：武將 persona card、口吻、關係、禁忌。
3. Retrieved Facts：Pinecone 回傳的 3~5 條高信心片段。
4. Keyword Intent：玩家已選的 keyword labels、category、keyword-specific evidence refs。
5. Player Context：玩家當前選擇的下拉情境與補充文字。
6. Output Contract：只輸出一句 20~60 字繁體中文台詞與 emotion tag。

### G-2. 禁止事項

1. 不得產生未檢索到的重大史實。
2. 不得讓武將知道玩家未記錄的隱私或跨存檔資訊。
3. 不得使用現代網路語、破第四面牆或自稱模型。
4. 不得把玩家輸入直接覆蓋進 persona card。

### G-3. Fallback

若 Pinecone 無命中、LLM timeout、內容審查失敗或 JSON parse 失敗，回傳 persona card 的 `Safe_Fallback_Line`，並標記 `fallbackUsed=true`。

若玩家有選 `selectedKeywordKeys`，fallback 句仍應優先保留對該關鍵字的弱聯想，但不得虛構不在 evidence 內的重大史實。

## H. Cocos 串接要求

1. Cocos 端新增 `NpcDialogueService` 包裝 API，不允許 UI component 直接散落 fetch。
2. UI 下拉選項由 `GET /v1/npc/context-options` 取得，不在前端硬寫事件清單。
3. 人物頁 `GeneralDetail` 應可額外呼叫 `GET /v1/npc/keyword-options`，將關鍵字以分類 chip / tag 形式呈現，不得把 category 寫死在前端常數裡。
4. 玩家點擊關鍵字後，再進一步點「互動」才觸發 `POST /v1/npc/dialogue`；避免每次選詞都直接打模型。
5. 人物頁 `general-journal-content` 可顯示最近日誌與一句生成反應，但不得取代既有歷史趣聞 / 血脈傳聞。
6. 戰鬥 HUD 頭像點擊可作為未來入口；MVP 先以人物頁練習，降低戰鬥場景風險。
7. Cocos Preview 需設定 FastAPI CORS，開發期允許 `localhost:7456`；正式環境不可使用萬用 `*`。

## I. Data Schema 需求

中台不直接寫入玩家存檔完整向量庫。客戶端只保留摘要、快取與同步指標。

```json
{
  "Data": {
    "NPC_Memory": {
      "Brain_Service_Profile": "dev-pinecone-fastapi",
      "Local_Vector_Mode": "sqlite-vec",
      "Persona_Card_Version": "general_persona_v1",
      "Last_Context_Sync_At": "2026-04-28T08:00:00",
      "Recent_Context": [
        {
          "General_ID": "zhang-fei",
          "Context_Key": "changban-bridge",
          "Selected_Keyword_Keys": ["liu-bei", "serpent-spear"],
          "Evidence_Refs": ["romance.ch042.zhang-fei.changban-bridge"],
          "Line_Digest": "sha256:..."
        }
      ],
      "Pending_Unknown_Events": []
    }
  }
}
```

Server 端可另設 `npc_brain_events`、`npc_brain_persona_cards`、`npc_brain_keyword_options`、`npc_brain_player_memory`，但不得要求 Cocos 本地存檔保存完整 prompt 或完整 source quote。

Keyword option 資料至少需包含：

- `keywordKey`
- `label`
- `category`
- `generalIds`
- `sourceRefs`
- `confidence`
- `faction`（若適用）
- `retired`（是否已淘汰，不再供 UI 顯示）

## J. 驗收標準

### J-1. 資料管線驗收

| Gate | 標準 |
|---|---|
| PDF 轉 MD | 可產生 `source.md` 與章回 manifest |
| 清洗 | 頁碼、頁首、頁尾與重複空白被移除 |
| 稱呼消歧 | 類似「將軍，我送您一把寶刀」的對話可在有足夠場景證據時把 `將軍` 解析回正確 `generalId`，並保留物品關係 |
| 抽取 | 張飛長坂橋事件可穩定抽出 `generalId=zhang-fei` |
| 關鍵字分類 | 張飛可穩定抽出人物 / 事件 / 地點 / 物品 / 生物關鍵字，且雜訊詞不進正式 keyword pack |
| 去重 | 同一事件跨 chunk 不重複入庫 |
| 入庫 | Pinecone metadata filter 可按 `generalId + contextKey` 命中 |

### J-2. 對話驗收

| Gate | 標準 |
|---|---|
| 史實吻合 | 台詞可回扣 evidenceRefs |
| 口吻吻合 | 張飛豪烈、關羽沉穩、趙雲忠勇不得混淆 |
| 關鍵字成功 | 玩家點選「劉備」「丈八蛇矛」等關鍵字時，張飛能以自身視角與口吻產生相關聯想 |
| 字數 | 預設 20~60 字，UI 不溢出 |
| 安全 | 不現代化、不亂編重大史實、不破第四面牆 |
| Fallback | 中台故障時 Cocos 可顯示模板台詞 |

## K. 名詞定義

| 名詞 | 代碼 / Key | 定義 |
|---|---|---|
| 三國大腦中台 | `SanguoBrainService` | 三國文本抽取、向量檢索、persona 與台詞生成服務 |

| Chunk | `TextChunk` | 可向量化的語意片段 |

| Persona Card | `Npc_Persona_Card` | 武將性格與口吻的結構化描述 |
| Brain Profile | `Brain_Service_Profile` | 當前使用的中台 / 本地 / 離線模式設定 |

## L. 實作路線

1. **Pilot**：只處理張飛長坂橋章回，跑通 PDF -> MD -> chunk -> event -> Pinecone -> dialogue。
2. **Roster Expand**：擴展張飛、關羽、趙雲、劉備、曹操、諸葛亮 persona card。
3. **Keyword Pack MVP**：為張飛先做一版 `general_keywords_v1`，支援人物 / 事件 / 地點 / 物品 / 生物五類關鍵字。
4. **GeneralDetail Demo**：Cocos 人物頁顯示 context 下拉、關鍵字分類與一句台詞。
5. **Journal Integration**：將 `General_Journal` entry 加入 playerContext。
6. **Streaming**：支援 Web preview 串流顯示。
7. **vLLM Challenge**：在 Linux GPU 環境測量併發與 KV cache。
8. **Local Degrade**：把高頻 persona / context option 下發到 SQLite-vec，支援離線模板模式。
