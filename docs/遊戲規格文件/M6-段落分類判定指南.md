# M6 段落分類判定指南

> 本文檔提供「如何判定段落應分類為 consolidated / doubt / discarded」的決策樹與實例。

---

## 核心判定邏輯

```
段落內容

├─ 與現行正式規格一致
│  └─ → **consolidated** ✅
│     標記 --status consolidated --target <對應doc_id>
│
├─ 新增內容 + 未在現行規格中
│  └─ → **doubt** ⚠️（轉為 MCQ）
│     標記 --status doubt --target <可能相關doc_id>
│     產出 MCQ 供人類決策
│
└─ 舊內容 / 已被新規格取代 / 無相關承接檔
   └─ → **discarded** ❌
      標記 --status discarded
```

## 判定矩陣（速查表）

| 情況 | 判定 | 說明 |
|------|------|------|
| 內容完全匹配現行規格 | consolidated | 無需修改 |
| 內容部分新增（如細節補充） | consolidated | 若新增微調較小 |
| 內容明確新增且重要 | doubt | 需人類拍板 |
| 內容與現行規格衝突 | doubt | 需決策取捨 |
| 背景說明 / preamble / 緒論 | discarded | 不屬於規格內容 |
| 舊提案 / 已被替代方案 | discarded | 附註原因 |

## 實作建議

### 分類時的檢查清單

對每個段落執行：

```
□ 讀完整個段落
□ 標記段落的主要話題（如「支援卡階級」「重修代價」）
□ 在 cross-reference-index 或現行規格書中搜尋相關內容
□ 判定是否有「新增」或「衝突」
□ 若有疑問，傾向標為 doubt（寧可人類決策）
□ 記錄理由於備註欄
```

### MCQ 產出的觸發條件

以下情況**必須**產出 MCQ：

1. **新增重要機制**：如「重修代價」
2. **邏輯衝突**：如舊規格 vs 新規格的定義不同
3. **商業決策**：如「是否應加課金點」
4. **設計哲學差異**：如「玩家體驗優先 vs 數值平衡優先」

以下情況**不需**產出 MCQ：

1. 純粹確認 / 細節補充（直接 consolidated + 更新規格）
2. 已被新規格完全取代（直接 discarded）
3. 背景說明 / 緒論（直接 discarded）

## 常見判定困境 & 解決方案

### 困境 1: 「到底是 consolidated 還是 doubt？」

判斷標準：
- **consolidated** = 「確實已在現行規格中，無新增」
- **doubt** = 「有任何新增、補充或未確認的細節」

**建議**：傾向於寧可標 doubt 讓人類確認，也不要標 consolidated 後發現遺漏。

### 困境 2: 「細節補充要不要產出 MCQ？」

判斷標準：
- **細節補充（如 UI 配色）** → consolidated + 直接更新規格
- **新增機制（如重修代價）** → doubt + 產出 MCQ

### 困境 3: 「一個段落太長，應不應該拆分？」

判斷標準：
- 本 consolidation flow 按 `## ` 為單位（不拆分）
- 若單個 `## ` 內含多個主題，標記時寫上備註

## 備註欄書寫範例

標記時附上 `--notes` 參數：

```bash
node tools_node/consolidation-extract.js mark <file> 2 \
  --status doubt \
  --target doc_spec_0026 \
  --notes "File 2 § 五提出重修代價機制，現行規格無此定義。需人類決策是否採納。"
```
